# Tushare 数据扩展第二批 — 后端设计

> **范围**：`stk_holdertrade`（股东增减持）、`pledge_stat`（股权质押统计）、`fina_audit`（财务审计意见）、`disclosure_date`（财报披露计划）
>
> **积分要求**：stk_holdertrade / pledge_stat / fina_audit 各 2000，disclosure_date 仅 500；用户当前 6000 积分全部可调
>
> **目标**：新增 4 张 Prisma 表、4 个 mapper、4 个同步计划项，注册到现有 plan-driven 同步架构

## 一、接口字段确认（来源：Tushare 官方文档）

### 1.1 stk_holdertrade（股东增减持，doc_id=175）

**限制**：2000 积分起，单次最大 3000 行，总量不限。支持 `ann_date` / `start_date` + `end_date` 查全市场。

| Tushare 字段 | 类型  | 必有 | 说明                               |
| ------------ | ----- | ---- | ---------------------------------- |
| ts_code      | str   | Y    | TS 股票代码                        |
| ann_date     | str   | Y    | 公告日期                           |
| holder_name  | str   | Y    | 股东名称                           |
| holder_type  | str   | Y    | 股东类型：G 高管 / P 个人 / C 公司 |
| in_de        | str   | Y    | 增减持类型：IN 增持 / DE 减持      |
| change_vol   | float | Y    | 变动数量（股）                     |
| change_ratio | float | Y    | 占流通比例（%）                    |
| after_share  | float | Y    | 变动后持股数                       |
| after_ratio  | float | Y    | 变动后占流通比例（%）              |
| avg_price    | float | N    | 平均价格                           |
| total_share  | float | Y    | 持股总数                           |
| begin_date   | str   | N    | 增减持开始日期                     |
| close_date   | str   | N    | 增减持结束日期                     |

### 1.2 pledge_stat（股权质押统计，doc_id=110）

**限制**：2000 积分起，单次最大 1000 行。输入参数 `ts_code` 和 `end_date` 均为可选。

| Tushare 字段  | 类型  | 必有 | 说明                   |
| ------------- | ----- | ---- | ---------------------- |
| ts_code       | str   | Y    | TS 股票代码            |
| end_date      | str   | Y    | 截止日期（每周五统计） |
| pledge_count  | int   | Y    | 质押次数               |
| unrest_pledge | float | Y    | 无限售股质押数量（万） |
| rest_pledge   | float | Y    | 限售股份质押数量（万） |
| total_share   | float | Y    | 总股本                 |
| pledge_ratio  | float | Y    | 质押比例（%）          |

### 1.3 fina_audit（财务审计意见，doc_id=80）

**限制**：2000 积分起。**`ts_code` 为必填**，必须按股票遍历拉取。支持 `start_date` / `end_date` / `period` 筛选。

| Tushare 字段 | 类型  | 必有 | 说明                                                              |
| ------------ | ----- | ---- | ----------------------------------------------------------------- |
| ts_code      | str   | Y    | TS 股票代码                                                       |
| ann_date     | str   | Y    | 公告日期                                                          |
| end_date     | str   | Y    | 报告期                                                            |
| audit_result | str   | Y    | 审计结果（标准无保留意见 / 保留意见 / 否定意见 / 无法表示意见等） |
| audit_fees   | float | N    | 审计总费用（元）                                                  |
| audit_agency | str   | N    | 会计事务所                                                        |
| audit_sign   | str   | N    | 签字会计师                                                        |

### 1.4 disclosure_date（财报披露计划，doc_id=162）

**限制**：500 积分起，单次最大 3000 行，总量不限。支持按 `end_date`（报告期）查全市场。

| Tushare 字段 | 类型 | 必有 | 说明                            |
| ------------ | ---- | ---- | ------------------------------- |
| ts_code      | str  | Y    | TS 股票代码                     |
| ann_date     | str  | Y    | 最新披露公告日                  |
| end_date     | str  | Y    | 报告期（季末日期，如 20231231） |
| pre_date     | str  | Y    | 预计披露日期                    |
| actual_date  | str  | N    | 实际披露日期（尚未披露则为空）  |
| modify_date  | str  | N    | 披露日期修正记录                |

## 二、Prisma Schema 设计

### 2.1 `prisma/tushare_stk_holdertrade.prisma`

```prisma
// 股东增减持（来自 Tushare stk_holdertrade 接口，doc_id=175）
model StkHolderTrade {
  // 由于同一公告日同一股东可能有多笔交易，使用自增 id 作为主键
  id           Int      @id @default(autoincrement())

  tsCode       String   @map("ts_code") @db.VarChar(16)
  annDate      DateTime @map("ann_date") @db.Date
  holderName   String   @map("holder_name") @db.VarChar(128)
  holderType   String   @map("holder_type") @db.VarChar(4)   // G / P / C
  inDe         String   @map("in_de") @db.VarChar(4)         // IN / DE
  changeVol    Float?   @map("change_vol")
  changeRatio  Float?   @map("change_ratio")
  afterShare   Float?   @map("after_share")
  afterRatio   Float?   @map("after_ratio")
  avgPrice     Float?   @map("avg_price")
  totalShare   Float?   @map("total_share")
  beginDate    DateTime? @map("begin_date") @db.Date
  closeDate    DateTime? @map("close_date") @db.Date

  syncedAt     DateTime @default(now()) @map("synced_at")

  @@unique([tsCode, annDate, holderName, inDe])
  @@index([tsCode])
  @@index([annDate])
  @@map("stock_holder_trades")
}
```

**设计决策**：同一 `ts_code + ann_date` 下同一股东可同时有增持和减持记录，因此唯一约束为 `tsCode + annDate + holderName + inDe`。使用自增 id 作为主键以简化 upsert 逻辑，唯一约束用于去重。

## 四、Mapper 函数设计

### 4.1 mapStkHolderTradeRecord

```typescript
export function mapStkHolderTradeRecord(
  r: TushareRecord,
  collector?: ValidationCollector,
): Omit<Prisma.StkHolderTradeCreateManyInput, 'id'> | null {
  const tsCode = readString(r, 'ts_code')
  const annDate = readDate(r, 'ann_date')
  const holderName = readString(r, 'holder_name')
  const inDe = readString(r, 'in_de')
  if (!tsCode || !annDate || !holderName || !inDe) {
    collector?.add({ field: 'PK', message: 'ts_code/ann_date/holder_name/in_de 缺失', record: r })
    return null
  }
  return {
    tsCode,
    annDate,
    holderName,
    inDe,
    holderType: readString(r, 'holder_type') ?? 'P',
    changeVol: readNumber(r, 'change_vol'),
    changeRatio: readNumber(r, 'change_ratio'),
    afterShare: readNumber(r, 'after_share'),
    afterRatio: readNumber(r, 'after_ratio'),
    avgPrice: readNumber(r, 'avg_price'),
    totalShare: readNumber(r, 'total_share'),
    beginDate: readDate(r, 'begin_date'),
    closeDate: readDate(r, 'close_date'),
  }
}
```

## 五、同步策略设计

| 接口              | 同步策略                      | 原因                                                                    | 归属服务               |
| ----------------- | ----------------------------- | ----------------------------------------------------------------------- | ---------------------- |
| `stk_holdertrade` | **按 ann_date 批量**          | 支持全市场按公告日查询；每日拉当天公告                                  | `FinancialSyncService` |
| `pledge_stat`     | **按 ts_code 遍历**           | 单次最大仅 1000 行；按 end_date 查一周全市场可能超限 → 按股票遍历更稳妥 | `FinancialSyncService` |
| `fina_audit`      | **按 ts_code 遍历**           | `ts_code` 为必填参数，只能逐只拉取                                      | `FinancialSyncService` |
| `disclosure_date` | **按 end_date（报告期）批量** | 支持按报告期查全市场；每年仅 4 个报告期                                 | `FinancialSyncService` |

### 5.1 stk_holdertrade 同步逻辑

```
日常同步：
  1. 取当天日期（或最近交易日）作为 ann_date
  2. 调用 stk_holdertrade(ann_date=yyyyMMdd)，单次最多 3000 行
  3. 若返回恰好 3000 行，使用 start_date=end_date=ann_date 分页（offset）
  4. 按 unique(tsCode, annDate, holderName, inDe) 执行 upsert

回补同步：
  1. 从 2010-01-01 开始，逐日（或按 start_date/end_date 按月）拉取
  2. 每次 start_date ~ end_date 跨度 ≤ 30 天，避免超 3000 行
  3. 300ms 节流
```

### 5.2 pledge_stat 同步逻辑

```
日常同步：
  1. 获取全部股票列表（从 StockBasic 表）
  2. 对每只股票调用 pledge_stat(ts_code=xxx)，返回近期质押统计（多行）
  3. upsert（PK = tsCode + endDate）
  4. 500ms 节流

回补同步：
  1. 同日常逻辑，遍历全部股票
  2. 单只股票历史行数不多（每周一行 × 数年 ≈ 数百行），一次可拉完
  3. 预计 ~5000 只 × 500ms ≈ 42 分钟，适合夜间调度

优化选项：
  若后续发现按 end_date 查全市场分页可行（单次 < 1000），
  可改为按 end_date 遍历每周五日期，效率更高
```

### 5.3 fina_audit 同步逻辑

```
日常同步：
  1. 获取有新公告的股票列表（或全量遍历）
  2. 对每只股票调用 fina_audit(ts_code=xxx, start_date=近30天)
  3. upsert（PK = tsCode + endDate）
  4. 300ms 节流

回补同步（采用 rebuildRecentYears 模式）：
  1. 遍历全部股票列表
  2. 对每只股票拉取全部历史审计意见（一般 ≤ 30 条）
  3. 按 tsCode + endDate 去重写入
  4. ~5000 只 × 300ms ≈ 25 分钟
```

### 5.4 disclosure_date 同步逻辑

```
日常同步：
  1. 取最近的报告期（如当前 2026Q1 → end_date=20260331）
  2. 调用 disclosure_date(end_date=20260331)，返回所有股票的披露计划
  3. 若超 3000 行则翻页（offset）
  4. deleteMany(endDate=period) + createMany 全量替换该报告期

回补同步：
  1. 从 2010 年 Q1 开始，遍历所有报告期（~64 个季度）
  2. 每个报告期返回 ~5000 行（全市场），需分页拉取
  3. 预计 64 × 2 页 ≈ 128 次请求，无频控压力
```

## 八、频控与风险评估

| 接口              | 每分钟频次（6000 积分） | 日常同步请求次数            | 回补请求次数               | 风控策略                        |
| ----------------- | ----------------------- | --------------------------- | -------------------------- | ------------------------------- |
| `stk_holdertrade` | ~200                    | 1-2 次（当日公告）          | ~6000 次（16 年 × 365 天） | 中等，300ms 节流                |
| `pledge_stat`     | ~100                    | ~5000 次（全量遍历）        | 同日常                     | **高**，500ms 节流 + 仅周六调度 |
| `fina_audit`      | ~100                    | ~100 次（近期有公告的股票） | ~5000 次                   | 中等，300ms 节流                |
| `disclosure_date` | ~200                    | 2-4 次（当季 + 翻页）       | ~128 次（64 季度 × 2 页）  | 无压力                          |

**总体评估**：

- 日常同步压力集中在 `pledge_stat`（遍历全部股票），安排在周六夜间错峰。
- 回补时 `stk_holdertrade` 量最大（~6000 次），需分多日完成或配合 300ms 节流。
- `fina_audit` 和 `disclosure_date` 全量回补均可在 1 小时内完成。

## 九、与第一批的依赖关系

- 本批 4 个接口全部归入 `FinancialSyncService`，与第一批的 `forecast` / `stk_holdernumber` 同域。
- `fina_audit` 使用与第一批 `forecast` 类似的 **ts_code 遍历** 模式（`rebuildRecentYears` 辅助方法可复用）。
- 第一批的 `disclosure_date` 数据可为后续构建"财报日历"功能提供基础。
- 建议先完成第一批实施，再启动第二批，以避免 `FinancialSyncService` 同时增加过多方法带来的 review 复杂度。
