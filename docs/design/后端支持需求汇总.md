## 一、统一后端约束

这些约束在多个设计文档中反复出现，建议作为后端统一验收门槛：

1. **所有接口使用 `POST + JSON Body`**，不使用 GET、query string、URL path id。
   - 本文后续凡写成 `/api/...` 的接口路径，均按 `POST /api/...` 理解；P0 开发明细会尽量显式写出 `POST`，避免联调歧义。
2. **资源 ID 通过 Body 传递**，例如 `{ "runId": "..." }`、`{ "portfolioId": "..." }`。
3. **交易日期统一 `YYYYMMDD` 字符串**，尤其是 `tradeDate / trade_date / startDate / endDate / latestTradeDate`。
4. **时间戳统一 ISO 8601 字符串**，如 `createdAt / updatedAt / startedAt / completedAt`。
5. **数值字段允许缺失时必须显式 nullable**，Swagger / DTO 与真实返回保持一致。
6. **新增字段保持向后兼容**：optional 或有默认值，旧前端请求不应报错。
7. **分页统一 1-based**：`page / pageSize / total / items[]`，排序统一 `sortBy / sortOrder` 或现有模块约定。
8. **危险写操作必须审计**：用户管理、数据同步、组合、研究笔记、因子、回测任务、异常扫描、调仓执行等。
9. **WebSocket 事件必须可和业务对象关联**：payload 中补 `jobId / runId / scanId / tradeDate / status / progress / errorSummary` 等。
10. **导出必须带当前筛选、排序和列配置**，避免“屏幕看到的”和“导出的”口径不一致。

### 3.2 后端开发前强制检查模板

每个从本文拆出的后端任务，建议在 issue / PR 描述中补齐以下 10 项；缺任一项则视为“需求不完整，不应直接开工”：

1. **业务场景**：哪个页面 / 哪个用户动作触发，解决什么阻塞。
2. **完整接口路径**：必须写 `POST /api/...`，不写相对路径；资源 ID 放 Body。
3. **请求 Body DTO**：字段名、类型、是否必填、默认值、枚举值、日期格式。
4. **响应 DTO**：顶层结构、分页结构、nullable 字段、数组 item 结构。
5. **错误结构**：业务错误码、用户可读 message、`traceId` 或日志关联 ID。
6. **权限**：最低角色、是否仅 SUPER_ADMIN、是否本人/同级限制。
7. **审计**：写操作和失败操作是否记录，`action/category/details.before/after` 口径。
8. **兼容性**：旧字段是否保留、旧请求不传新字段时的默认行为。
9. **性能 / 缓存**：分页上限、排序稳定性、是否需要短缓存或异步 job。
10. **验收样例**：正常、空数据、部分失败、无权限、参数错误、nullable 六类样例。

#### 一、统一约束

1. 所有接口必须为 **POST + JSON Body**；不得使用 GET、query string、路径参数 ID。

`tsvector_col` 通过触发器维护：

```sql

BEGIN

  RETURN NEW;
END $$ LANGUAGE plpgsql;

DROP TRIGGER IF EXISTS trg_notes_tsv ON research_notes;
CREATE TRIGGER trg_notes_tsv BEFORE INSERT OR UPDATE ON research_notes

返回: ResearchNote; // 最新版本
```

实现：复制选中版本的 `title/content/tags/ts_code/is_pinned` 到主表；同时写入新版本（`version_no = version_count + 1`，summary = "还原至 v{versionNo}"）。

###### 3.10 🟡 `POST /api/research-note/batch-delete`（新）

```ts
入参: { ids: number[] }   // 上限 200
返回: { succeeded: number[]; failed: { id, reason }[] }
```

事务内逐条软删（仍记录到审计日志）。

###### 3.11 🟡 `POST /api/research-note/batch-tag`（新）

```ts
入参: { ids: number[]; add?: string[]; remove?: string[] }
返回: { updated: number; failed: { id, reason }[] }
```

实现要点：在 service 层取每条 `tags`（`jsonb`），合并/去重后写回；标签去重 + trim + 长度上限 32 字。

###### 3.12 🟡 `POST /api/research-note/tags`（**扩展返回**）

将 `{ tags: string[] }` 改为：

```ts
{
  tags: Array<{ tag: string; count: number }>;
}
```

> 前端做了向后兼容（若仍是 `string[]` 则按旧格式渲染），但建议升级。

###### 3.13 🔴 `POST /api/file/upload`（**通用，新增**）

放在通用 `apps/file/` 模块下，全站复用：

```
multipart/form-data  field: file
```

返回：

```ts
{
  url: string; // 可被前端 <img src> 直接引用（CDN 或同域）
  mimeType: string;
  size: number;
  fileName: string;
}
```

校验：

- mime 白名单：`image/png` `image/jpeg` `image/gif` `image/webp`
- 大小 ≤ 5MB
- 文件名/扩展名白名单 + 重命名（防止路径穿越）
- 鉴权：必须登录；记录到 `research_note_attachments`（即使尚未关联 note，先存 user）

存储：优先对象存储（S3/MinIO）；本地开发可落到 `uploads/` 目录 + `/static/uploads/*` 静态服务。

###### 3.14 🟢 `POST /api/research-note/templates`（可选，本期可不做）
