## 1. 总览结论

## 2. 已识别问题清单

### 2.1 调度（定时任务频率）

| #   | 问题                                                                                                                                                                                                                                                                                                                                        | 影响                                                                                                      | 优先级                                               |
| --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| S1  | `checkAndCatchUpMissedSchedules` 在 18:30 之前直接 `return`，导致 **08:10/08:15/08:20** 的基础任务（`stock_basic`/`trade_cal`/`stock_company`）以及 **周一 06:00 的 `index_classify`** 等早间任务在 18:30 之前重启容器时永远不被 catch-up                                                                                                   | 启动晚于 cron 时间 → 该日基础数据可能缺失（除非 bootstrap 启用）                                          | 高                                                   |
| S2  | `hasCronPassedToday` 只解析 hour/minute，**忽略 day-of-month / day-of-week**。周一-only 任务（如 `0 0 6 * * 1`、`0 0 4 1 * *`）在错误的星期/日期被 catch-up 误判为"今天该跑"                                                                                                                                                                | 周二跑周一 cron 任务，造成多余请求、日志噪音；甚至会把月初 cron `0 0 4 1 * *` 在每天的 04:00 之后都误触发 | 高                                                   |
| S3  | 多个 cron 在**同一分钟**触发：`0 0 21`（INCOME+STK_HOLDER_TRADE）、`0 30 21`（FINA_INDICATOR+REPURCHASE+FUND_PORTFOLIO）、`0 0 20`（FORECAST+INDEX_GLOBAL+INDEX_DAILY）、`0 30 20`（STK_HOLDER_NUMBER+CCASS_HK_HOLD）、`0 50 18`（HSGT_TOP10+HSGT_HK_TOP10）、`0 5 19`（market 一项 + 资金流第一项）、`0 0 9 * * 1`（OPT_BASIC+FUND_BASIC） | 全局锁 `running` 串行执行，被锁住的任务被丢弃必须等下一个整点 catch-up 才能跑                             | 中                                                   |
| S4  | `onHourlyCatchupCheck` 一小时一次。若同步轮 22:55 才结束，落锁丢失的任务要 23:00 才会重试，时间损失最多 1 小时                                                                                                                                                                                                                              | 偶发的丢任务恢复延迟                                                                                      | 低（已在 `runPlans` finally 立即 catch-up 一次缓解） |
| S5  | `bootstrapOnStart=false`（dev 默认）时，启动后仅做一次 catch-up，但被 S1/S2 限制，**早间任务** 无法在启动时回补                                                                                                                                                                                                                             | 开发联调时容易看到 stock_basic 等数据陈旧                                                                 | 高                                                   |

| I5  | `factor-data-sync` 的 `syncIndexWeight` 串行迭代多个 indexCode，未利用全局并发；`alternative-data-sync` 的 `syncShareFloat` 同理                                                                                                                                                                 | 同步周期长                                                           | 中     |

## 3. 本次实现的修复（按优先级）

> 原则：小步、可回退、可验证；仅修正确性/可靠性 Bug 与最有性价比的优化。

### 3.1 修复 S1 / S2 / S5 — 完整化 catch-up 解析

- 重写 `hasCronPassedToday`：
  - 增加对 day-of-month、month、day-of-week 字段的解析（支持 `*`、`数字`、`a,b,c`、`a-b`、`*/n`）。
  - 仅当**今天确实在该 cron 允许的"日"集合内**时，才进一步判断 hour/minute 是否已过；不满足则跳过。
  - 不再以"`*`=已过"草草通过，改为按真实表达式判断。
- 移除 `checkAndCatchUpMissedSchedules` 开头的 `18:30` 早退守卫；当无任何 plan 命中时早退由 `plansToRun.length===0` 自然完成。
- 影响：早间 cron（08:10/08:15/08:20/06:00/...）在 dev 重启 / 容器升级时也能被立即补跑；周一-only 任务不再被周二误触发。

### 3.2 修复 S3 — Cron 错峰

- 把以下重叠 cron 错开 1-2 分钟，让全局锁释放后不必再依赖 catch-up：
  - `STK_HOLDER_TRADE` `0 0 21 * * *` → `0 15 21 * * *`
  - `REPURCHASE` `0 30 21 * * *` → `0 35 21 * * *`
  - `FUND_PORTFOLIO` 月度任务保留 `0 0 4 1 * *`
  - `STK_HOLDER_NUMBER` 在 `0 30 20` 与 market 类无冲突（已是错开），保留
  - `MONEYFLOW` 第一项 `0 5 19` 与 `market` `0 5 19`：moneyflow 改为 `0 8 19`
  - `OPT_BASIC` `0 0 9 * * 1` 与 `FUND_BASIC` `0 0 9 * * 1`：option 改为 `0 5 9 * * 1`
- 不动 financial 内部已经错峰的 21:00→21:50 序列，因为它们彼此错开。

### 3.3 ~~修复 I2 — `balanceSheet` 回退路径数据丢失~~（推后）

- I2 涉及"VIP 部分成功 → 回退 per-stock 全量"是否要保留 VIP 写入的分片；改动会改变现有"全表幂等替换"语义，存在风险（per-stock 仅做 `createMany skipDuplicates`，不会覆盖陈旧数据），需配合 `unique key` 与重建策略一起评估。
- 本次合入 **不做** I2 修复，记录在 §5 后续待办，等下次单独评审。

### 3.4 修复 I3 — moneyflow 配额耗尽时入队 + 保护 retentionCutoff

- 检测到 40203 配额错误时：对剩余未同步的 `tradeDate` 调用 `enqueueRetry`；
- 当配额耗尽（即 `quotaExceeded=true`）时，**跳过本轮 `retentionCutoff` 清理**，等待下一轮补完再清理；
- 写入失败计数到 `payload.failedCount` 便于巡检。

### 3.5 修复 P2 — 智能触发 heatmap / signal

- `triggerHeatmapSnapshotAsync`：仅当执行的任务中包含 `category==='market'` 的 plan 时才聚合；
- `triggerSignalGenerationAsync`：仅当执行的任务中包含 `market` / `moneyflow` / `factor` 的 plan 时才生成。
- 实现方式：在 `RunPlansResult` 已有 `executedTasks` 基础上，根据 `registry.getPlan(task).category` 过滤。

### 3.6 修复 I6 — 节流统一

- 把 `cyq_chips`、`ths_member` 加入 `DOCUMENTED_RATE_LIMIT_MS`（分别 200ms、200ms），移除两处内联 `setTimeout`。
- `TushareClient` 已经按 api channel 做 channel-queue 串行，加上文档化频率即可。

> 其他较大改动（I1 — financial 全量重建增加 checkpoint；V1/V2 — 并行化质量检查 / batch financial period count；P1 — 差异化缓存清理；P3 — `getAllStockCodes` 内存缓存）**列入后续待办**，不在本次范围内实施，以保持每次合入风险可控。已在文末 §5 留下下一阶段 TODO。
