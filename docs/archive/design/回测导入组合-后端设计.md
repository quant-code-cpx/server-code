# 回测结果导入组合 — 后端设计（OPT-1.1）

> **对应优化项**：OPT-1.1（回测 → 组合桥梁）
> **目标**：打通"回测完成 → 一键导入组合持仓"链路，解决最后一公里断裂问题。
> **日期**：2026-04-11

---

## 一、背景与动机

当前 `BacktestModule` 和 `PortfolioModule` 完全隔离：

```
BacktestRun (status=COMPLETED)
  ├── BacktestPositionSnapshot（末日持仓）
  └── ❌ 无路径 → Portfolio.holdings
```

用户需手动逐条录入回测末日持仓到组合，容易出错且低效。

---

## 二、用户操作流程

```
用户查看已完成回测 → 点击「导入到组合」
  ├─ 已有组合 → 选择目标组合 → 显示调仓预览 → 确认执行
  └─ 无组合   → 自动创建新组合 → 直接写入持仓
```

---

## 三、接口设计

### 3.1 `POST /portfolio/apply-backtest`

将回测末日持仓导入到目标组合。

#### 请求 DTO

```typescript
export class ApplyBacktestDto {
  @IsString()
  backtestRunId: string // 回测任务 ID

  @IsOptional()
  @IsString()
  portfolioId?: string // 目标组合 ID（不传则自动创建新组合）

  @IsOptional()
  @IsString()
  @MaxLength(100)
  portfolioName?: string // 当 portfolioId 为空时，新组合名称（默认"回测导入-{回测名}"）

  @IsOptional()
  @IsEnum(ApplyMode)
  mode?: ApplyMode // 导入模式，默认 REPLACE
}

enum ApplyMode {
  REPLACE = 'REPLACE', // 清空目标组合原持仓，完全替换为回测持仓
  MERGE = 'MERGE', // 合并：已有相同标的加仓（加权平均成本），新标的新增
}
```

#### 响应 DTO

```typescript
export class ApplyBacktestResponseDto {
  portfolioId: string // 目标组合 ID（新建或已有）
  portfolioName: string // 组合名称
  backtestRunId: string // 来源回测任务 ID
  mode: ApplyMode // 实际使用的导入模式
  snapshotDate: string // 回测末日持仓日期（YYYY-MM-DD）

  changes: RebalanceAction[] // 调仓动作明细
  summary: {
    added: number // 新增持仓数
    updated: number // 更新（加仓）持仓数
    removed: number // 清除的原持仓数（仅 REPLACE 模式）
    unchanged: number // 未变动持仓数（仅 MERGE 模式）
    totalHoldings: number // 组合最终持仓数
  }
}

export class RebalanceAction {
  tsCode: string
  stockName: string
  action: 'BUY' | 'SELL' | 'ADJUST' | 'HOLD'

  // 变动前
  previousQuantity: number // 0 = 新增
  previousAvgCost: number

  // 变动后
  targetQuantity: number // 0 = 清仓
  targetAvgCost: number

  // 差异
  deltaQuantity: number // 正数=加仓 负数=减仓
}
```

## 四、核心逻辑

### 4.1 主流程

```
applyBacktest(dto, userId)
  │
  ├─ 1. 校验回测任务
  │     ├─ 查询 BacktestRun（runId）
  │     ├─ 验证 run.userId === userId（仅本人回测可导入）
  │     └─ 验证 run.status === 'COMPLETED'
  │
  ├─ 2. 获取末日持仓快照
  │     ├─ findFirst(runId, orderBy tradeDate desc) → 获取最新日期
  │     └─ findMany(runId, tradeDate) → 全部持仓
  │
  ├─ 3. 确定目标组合
  │     ├─ portfolioId 有值 → assertOwner(portfolioId, userId)
  │     └─ portfolioId 无值 → 创建新组合
  │         ├─ name: portfolioName ?? `回测导入-${run.name ?? runId.slice(0,8)}`
  │         └─ initialCash: run.initialCapital
  │
  ├─ 4. 读取目标组合当前持仓
  │     └─ findMany(portfolioId)
  │
  ├─ 5. 生成调仓计划（diff）
  │     ├─ REPLACE 模式：
  │     │   ├─ 回测中有、组合中无 → BUY
  │     │   ├─ 回测中有、组合中有 → ADJUST（数量/成本不同）或 HOLD（完全相同）
  │     │   └─ 回测中无、组合中有 → SELL（清仓）
  │     └─ MERGE 模式：
  │         ├─ 回测中有、组合中无 → BUY
  │         ├─ 回测中有、组合中有 → ADJUST（加仓合并）
  │         └─ 回测中无、组合中有 → HOLD（保持不动）
  │
  ├─ 6. 在事务中执行调仓
  │     prisma.$transaction([
  │       REPLACE 模式：先 deleteMany 清空原持仓，再 createMany 批量写入
  │       MERGE 模式：逐条 upsert，已有合并加权平均成本
  │     ])
  │
  ├─ 7. 清除组合缓存
  │
  └─ 8. 返回 ApplyBacktestResponseDto
```

### 4.2 成本价映射规则

| 回测字段    | Portfolio 字段 | 映射说明                 |
| ----------- | -------------- | ------------------------ |
| `costPrice` | `avgCost`      | 直接使用回测持仓成本价   |
| `quantity`  | `quantity`     | 直接使用回测持仓数量     |
| `tsCode`    | `tsCode`       | 直接映射                 |
| —           | `stockName`    | 从 `StockBasic` 查询补全 |

### 4.3 合并模式（MERGE）加权平均成本

```
newQty     = existing.quantity + snapshot.quantity
newAvgCost = (existing.quantity × existing.avgCost + snapshot.quantity × snapshot.costPrice) / newQty
```

与现有 `PortfolioService.addHolding()` 的加仓逻辑一致。

## 五、数据模型

无新增 Prisma Model。完全复用现有：

- **读取端**：`BacktestRun` + `BacktestPositionSnapshot`
- **写入端**：`Portfolio` + `PortfolioHolding`
- **查询辅助**：`StockBasic`（补全 stockName）

## 六、模块依赖

```
PortfolioModule
  imports: [WebsocketModule, BacktestModule]  ← 新增 BacktestModule import
```

`BacktestModule` 已导出 `BacktestRunService`，可直接注入。

但考虑到桥接逻辑不应直接依赖 `BacktestRunService` 的完整功能，采用通过 `PrismaService` 直接查询回测数据的方式更清洁：

- `BacktestRun` 和 `BacktestPositionSnapshot` 是数据库表，`PrismaService` 已全局可用
- 桥接服务只需读取两张表，无需引入整个 BacktestModule
- 避免引入不必要的循环依赖风险

**最终方案**：桥接服务通过 `PrismaService` 直接查询，不 import BacktestModule。

## 九、测试计划

### 9.1 单元测试用例

| 场景                            | 预期                                           |
| ------------------------------- | ---------------------------------------------- |
| 回测不存在                      | NotFoundException                              |
| 回测非当前用户                  | ForbiddenException                             |
| 回测未完成                      | BadRequestException                            |
| 回测无持仓快照                  | BadRequestException                            |
| REPLACE 模式 — 目标组合有原持仓 | 清空 → 写入 → 返回 SELL + BUY                  |
| REPLACE 模式 — 新建组合         | 创建组合 → 写入 → 返回全部 BUY                 |
| MERGE 模式 — 部分标的重叠       | 重叠=ADJUST（加权平均），非重叠=BUY，原有=HOLD |
| 回测持仓中有已退市标的          | 正常导入，stockName 查不到时用 tsCode 兜底     |

### 9.2 关键 Mock

- `prisma.backtestRun.findUnique` — 控制回测状态和归属
- `prisma.backtestPositionSnapshot.findFirst` / `findMany` — 控制持仓数据
- `prisma.portfolioHolding.findMany` / `deleteMany` / `createMany` — 验证写入行为
- `prisma.stockBasic.findMany` — 补全 stockName
- `prisma.$transaction` — 验证事务包裹
