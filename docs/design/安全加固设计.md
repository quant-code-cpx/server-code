## 一、任务 1：JWT Secret 启动校验

### 1.1 问题描述

`src/config/token.config.ts` 原来使用硬编码默认值兜底：

```typescript
secret: REFRESH_TOKEN_SECRET || 'refresh_secret',
secret: ACCESS_TOKEN_SECRET || 'access_secret',
```

若部署时遗漏配置环境变量，所有实例使用相同弱密钥，JWT 可被任意伪造。

### 1.2 实现方案

在 `TokenConfig` 工厂函数内添加 `requireJwtSecret()` 校验函数：

- 若环境变量为空 → 抛出 `Error` 阻止启动
- 若长度 < 32 字符 → 抛出 `Error` 阻止启动

由于 `TokenConfig` 在 `ConfigModule.forRoot()` 注册时执行，校验在应用启动阶段即触发，无法绕过。

### 1.3 改动文件

| 文件 | 变更说明 |
|------|---------|
| `src/config/token.config.ts` | 添加 `requireJwtSecret()` 校验，移除硬编码默认值 |
| `.env.example` | 注释中注明必填及生成方式 |

### 1.4 验证说明

- 未配置 `ACCESS_TOKEN_SECRET` / `REFRESH_TOKEN_SECRET` → 启动即报错退出
- 密钥长度 < 32 → 启动报错
- 正常配置 → 无影响

## 二、任务 2：细粒度限流

### 2.1 问题描述

全局仅配置 `20 req / 10s`，对登录、验证码、密码重置等敏感接口过于宽松，存在暴力破解风险。

### 2.2 实现方案

将全局 Throttler 策略命名为 `default`，在敏感接口上用 `@Throttle()` 覆盖：

| 接口 | 原限制 | 新限制 |
|------|--------|--------|
| `POST /auth/captcha` | 20/10s | 10/60s |
| `POST /auth/login` | 20/10s | 5/60s |
| `POST /auth/refresh` | 20/10s | 10/60s |
| `POST /user/create` | 20/10s | 5/60s |
| `POST /user/reset-password` | 20/10s | 5/60s |
| 其他接口 | 20/10s | 不变 |

### 2.3 改动文件

| 文件 | 变更说明 |
|------|---------|
| `src/app.module.ts` | 为全局 throttler 添加 `name: 'default'` |
| `src/apps/auth/auth.controller.ts` | 添加 `@Throttle()` 装饰器至 `captcha` / `login` / `refresh` |
| `src/apps/user/user.controller.ts` | 添加 `@Throttle()` 装饰器至 `create` / `reset-password` |

## 四、任务 4：CORS 生产环境配置审查

### 4.1 问题描述

原代码：`process.env.CORS_ORIGIN || false`

当 `CORS_ORIGIN` 未配置时，`origin` 为 `false`（布尔值），Express 将禁用所有 CORS 头，导致前端跨域请求静默失败，且无任何日志提示。

另外，`.env.example` 注释说明支持多域名逗号分隔，但代码未实现。

### 4.2 实现方案

重写 `main.ts` 中的 CORS 配置逻辑：

```typescript
// 开发环境：允许所有来源
// 生产环境：
//   - CORS_ORIGIN 已配置 → 按逗号分隔解析为数组白名单
//   - CORS_ORIGIN 未配置 → 打印 WARN 日志，禁用 CORS（最严格默认值）
```

### 4.3 改动文件

| 文件 | 变更说明 |
|------|---------|
| `src/main.ts` | 重写 CORS 逻辑，支持多域名白名单，未配置时打印 WARN |
| `.env.example` | 补充多域名配置示例 |

## 五、任务 5：请求体大小限制

### 5.1 问题描述

未显式配置请求体大小限制，依赖 Express 隐式默认值（100KB），且无明确文档说明。

### 5.2 实现方案

在 `main.ts` 中通过 `express.json()` 和 `express.urlencoded()` 中间件显式配置 1MB 限制：

```typescript
import { json, urlencoded } from 'express'
app.use(json({ limit: '1mb' }))
app.use(urlencoded({ limit: '1mb', extended: true }))
```

超出限制时 Express 自动返回 HTTP 413 Payload Too Large。

### 5.3 改动文件

| 文件 | 变更说明 |
|------|---------|
| `src/main.ts` | 添加 body size 限制中间件 |

## 六、任务 6：管理员操作审计日志

### 6.1 问题描述

所有管理员操作（创建/删除用户、修改状态、重置密码、更新配额）无任何审计记录，无法追溯"谁在什么时间对哪个账号做了什么操作"。

### 6.2 Prisma Schema

**文件**：`prisma/audit_log.prisma`

```prisma
enum AuditAction {
  USER_CREATE
  USER_DELETE
  USER_UPDATE_STATUS
  USER_UPDATE_INFO
  USER_RESET_PASSWORD
}

model AuditLog {
  id              Int         @id @default(autoincrement())
  operatorId      Int                         // 操作者 ID（不设 FK，保证审计数据不受用户删除影响）
  operatorAccount String      @db.VarChar(64) // 操作者账号（冗余）
  action          AuditAction
  targetId        Int?
  targetAccount   String?     @db.VarChar(64)
  details         Json?                       // 变更详情（不含密码明文）
  ipAddress       String?     @db.VarChar(64)
  createdAt       DateTime    @default(now())

  @@index([operatorId])
  @@index([targetId])
  @@index([action])
  @@index([createdAt])
  @@map("audit_logs")
}
```

### 6.3 AuditLogService

**文件**：`src/apps/user/audit-log.service.ts`

| 方法 | 说明 |
|------|------|
| `record(params)` | 写入审计记录；fire-and-forget，失败不阻塞主流程 |
| `findAll(query)` | 分页查询，支持按操作者/目标/类型/时间范围筛选 |

### 6.4 UserService 中的审计调用点

| 方法 | AuditAction | details 字段 |
|------|------------|-------------|
| `create()` | `USER_CREATE` | `{ role, backtestQuota, watchlistLimit }` |
| `updateStatus()` | `USER_UPDATE_STATUS` | `{ from: oldStatus, to: newStatus }` |
| `adminUpdateUser()` | `USER_UPDATE_INFO` | `{ changes: dto }` |
| `resetPassword()` | `USER_RESET_PASSWORD` | `{}` （密码本身不记录） |
| `remove()` | `USER_DELETE` | `{ previousStatus }` |

### 6.5 新增接口

`POST /user/audit-log/list`（`@Roles(UserRole.ADMIN)`）— 管理员分页查询审计日志

请求体：

| 字段 | 类型 | 说明 |
|------|------|------|
| `page` | number | 页码（默认 1） |
| `pageSize` | number | 每页条数（默认 20，最大 100） |
| `operatorId` | number? | 操作者 ID 筛选 |
| `targetId` | number? | 目标用户 ID 筛选 |
| `action` | AuditAction? | 操作类型筛选 |
| `startDate` | string? | 开始时间（ISO 8601） |
| `endDate` | string? | 结束时间（ISO 8601） |

### 6.6 改动文件

| 文件 | 变更说明 |
|------|---------|
| `prisma/audit_log.prisma` | 新增 `AuditAction` 枚举和 `AuditLog` 模型 |
| `src/apps/user/audit-log.service.ts` | 新增服务：`record()` + `findAll()` |
| `src/apps/user/dto/audit-log-query.dto.ts` | 查询 DTO |
| `src/apps/user/dto/audit-log-response.dto.ts` | 响应 DTO |
| `src/apps/user/user.service.ts` | 注入 `AuditLogService`，5 处写入审计记录 |
| `src/apps/user/user.module.ts` | 注册 `AuditLogService` |
| `src/apps/user/user.controller.ts` | 新增 `POST /user/audit-log/list` 端点 |

### 6.7 设计原则

- **只追加，不删除**：`AuditLog` 表无删除/修改接口
- **不阻塞主流程**：`record()` 采用 fire-and-forget，写入异常仅 `logger.error`
- **不记录敏感信息**：`USER_RESET_PASSWORD` 的 `details` 字段为空对象
- **去中心化 FK**：`operatorId` / `targetId` 为普通整型，不建立外键约束，防止用户删除导致审计记录联级删除

### 6.8 数据库迁移

在有 PostgreSQL 可连接的环境中执行：

```bash
npm run prisma:migrate
# 或
npx prisma migrate dev --name add_audit_log
```

## 安全加固后整体态势对比

| 维度 | 加固前 | 加固后 |
|------|--------|--------|
| JWT 密钥管理 | 硬编码弱默认值，遗漏配置无感知 | 强制环境变量 + 最小长度校验，缺失即启动失败 |
| 限流策略 | 全局一刀切 20/10s | 敏感接口 5/60s，普通接口保持 20/10s |
| Redis 认证 | Docker 侧已保护，应用侧静默无密码连接 | 非开发环境必须配置密码，否则启动失败 |
| CORS | 生产环境未配置时静默禁用 CORS | 显式多域名白名单 + WARN 日志提示 |
| 请求体防护 | 依赖 Express 隐式 100KB 默认值 | 显式 1MB 限制，超限返回 HTTP 413 |
| 操作审计 | 零审计记录 | 全部管理员操作写入 `audit_logs` 表，可查可追溯 |
