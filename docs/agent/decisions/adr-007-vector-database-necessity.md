# ADR-007：向量数据库必要性

## 背景

核心数据是结构化行情/财务；会话和记忆尚无成熟语料规模。PostgreSQL 已是权威库，项目未安装 pgvector 或独立向量库。

## 问题

MVP 是否引入向量数据库/RAG？

## 可选方案

1. 独立 Milvus/Qdrant/Weaviate。
2. PostgreSQL pgvector。
3. 暂不使用向量；结构化查询 + PostgreSQL 全文/关键词。

## 最终决策

MVP 采用方案 3。新闻由搜索实时获取，金融事实走固定 Tool，记忆用结构化字段/全文检索。阶段三若报告、公告快照和研究笔记达到规模，先做 pgvector 小范围试点，不引入独立向量服务。

## 选择理由

当前问题不是召回不足，而是时点、来源、口径和审计；向量库不能解决这些问题，反而增加新鲜度与权限同步。

## 放弃其他方案原因

独立向量库运维成本高；立即装 pgvector 也缺评测集与清晰语料生命周期。

## 正面影响

少一套基础设施，权限和数据一致性更简单。

## 负面影响

长篇研究资料的语义召回能力延后。

## 风险

团队误把关键词检索不足归因于模型。必须先建立查询/召回评测，再决策。

## 后续复审条件

可检索文档超过 10 万、关键词 Recall@10 低于目标、语义检索有明确收益且租户过滤/删除/重嵌入方案完成。
