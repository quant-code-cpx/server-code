## 二、A — ValidationCollector 工具类

### 2.1 文件路径

`src/tushare/sync/quality/validation-collector.ts`

### 2.2 设计

```typescript
export interface ValidationEntry {
  tsCode?: string | null
  tradeDate?: string | null
  ruleName: string // 'missing_pk' | 'invalid_number' | 'ohlc_violation' | …
  severity: 'info' | 'warn' | 'error'
  message: string
  rawData?: Record<string, unknown>
}

export class ValidationCollector {
  private readonly entries: ValidationEntry[] = []
  private readonly taskName: string
  /** 单次同步最多收集的异常条数，防止全量回补时 OOM */
  private readonly maxEntries: number

  constructor(taskName: string, maxEntries = 5000) {
    this.taskName = taskName
    this.maxEntries = maxEntries
  }

  get task(): string {
    return this.taskName
  }
  get size(): number {
    return this.entries.length
  }

  add(entry: ValidationEntry): void {
    if (this.entries.length < this.maxEntries) {
      this.entries.push(entry)
    }
  }

  /** 返回本次收集的所有条目并清空内部缓冲 */
  drain(): ValidationEntry[] {
    return this.entries.splice(0)
  }
}
```

### 2.3 设计要点

- **纯值对象**，不依赖 NestJS DI，不引入 PrismaService — 在需要写库时由调用方（sync service）负责
- `maxEntries` 防止全量回补时一次性收集百万条异常导致内存爆涨；默认上限 5000 条足够分析首批问题
- `drain()` 可在一次同步的多个阶段分批归集

## 三、B — mapper helper 函数增强

### 3.1 改动目标

让 mapper 在**丢弃记录**或**检测到数值异常**时，能将原因写入 collector。

### 3.2 方案：给 mapper 函数增加可选 collector 参数

**改动模式**（以 `mapOhlcvRecord` 为例）：

```typescript
// 改动前
function mapOhlcvRecord<T extends ...>(record: TushareRecord): Omit<T, 'syncedAt'> | null {
  const tsCode = readString(record, 'ts_code')
  const tradeDate = readDate(record, 'trade_date')
  if (!tsCode || !tradeDate) {
    return null
  }
  return { tsCode, tradeDate, open, high, low, close, ... }
}

// 改动后
function mapOhlcvRecord<T extends ...>(
  record: TushareRecord,
  collector?: ValidationCollector,
): Omit<T, 'syncedAt'> | null {
  const tsCode = readString(record, 'ts_code')
  const tradeDate = readDate(record, 'trade_date')
  if (!tsCode || !tradeDate) {
    collector?.add({
      tsCode, tradeDate: readString(record, 'trade_date'),
      ruleName: 'missing_pk',
      severity: 'error',
      message: `关键字段缺失: ts_code=${tsCode}, trade_date=${readString(record, 'trade_date')}`,
      rawData: record,
    })
    return null
  }

  const result = { tsCode, tradeDate, open, high, low, close, preClose, change, pctChg, vol, amount }

  // OHLC 合理性校验（仅当 collector 存在时执行，零开销）
  if (collector) {
    validateOhlcv(result, record, collector)
  }

  return result as Omit<T, 'syncedAt'>
}
```

### 3.3 OHLC 合理性校验函数

新增一个内部函数 `validateOhlcv`，**仅在 collector 存在时**被调用：

```typescript
function validateOhlcv(
  mapped: {
    open: number | null
    high: number | null
    low: number | null
    close: number | null
    vol: number | null
    amount: number | null
    pctChg: number | null
    preClose: number | null
  },
  raw: TushareRecord,
  collector: ValidationCollector,
): void {
  const { open, high, low, close, vol, amount, pctChg, preClose } = mapped
  const tsCode = readString(raw, 'ts_code')
  const tradeDate = readString(raw, 'trade_date')

  // 规则 1: high >= max(open, close) 且 low <= min(open, close)
  if (high != null && low != null && open != null && close != null) {
    if (high < Math.max(open, close) || low > Math.min(open, close)) {
      collector.add({
        tsCode,
        tradeDate,
        ruleName: 'ohlc_range_violation',
        severity: 'warn',
        message: `OHLC 不满足 high≥max(O,C) 且 low≤min(O,C): O=${open} H=${high} L=${low} C=${close}`,
      })
    }
  }

  // 规则 2: vol/amount 非负
  if ((vol != null && vol < 0) || (amount != null && amount < 0)) {
    collector.add({
      tsCode,
      tradeDate,
      ruleName: 'negative_volume',
      severity: 'warn',
      message: `成交量/成交额为负: vol=${vol}, amount=${amount}`,
    })
  }

  // 规则 3: pctChg 与 (close - preClose) / preClose 偏差 >1pp 且 preClose>0
  if (pctChg != null && close != null && preClose != null && preClose > 0) {
    const computed = ((close - preClose) / preClose) * 100
    if (Math.abs(pctChg - computed) > 1) {
      collector.add({
        tsCode,
        tradeDate,
        ruleName: 'pct_chg_mismatch',
        severity: 'info',
        message: `pctChg 偏差: API=${pctChg.toFixed(2)}, 计算=${computed.toFixed(2)}`,
      })
    }
  }
}
```

### 3.4 各 mapper 函数改动清单

所有 29 个 mapper 函数统一增加可选参数 `collector?: ValidationCollector`。

- **关键字段缺失记录**：所有 mapper 的 `return null` 分支增加 `collector?.add(...)` 调用。这是统一模式，改动机械化。
- **OHLC 校验**：仅 `mapOhlcvRecord`（被 daily/weekly/monthly 共用）调用 `validateOhlcv`
- **其他 mapper**（财务/事件/资金流）：仅记录关键字段缺失，不做数值合理性校验（财务报表字段多且合理范围难定义）

**函数签名变更示例**：

```typescript
// 改动前
export function mapDailyRecord(record: TushareRecord): Prisma.DailyCreateManyInput | null {
  return mapOhlcvRecord<Prisma.DailyCreateManyInput>(record)
}

// 改动后
export function mapDailyRecord(
  record: TushareRecord,
  collector?: ValidationCollector,
): Prisma.DailyCreateManyInput | null {
  return mapOhlcvRecord<Prisma.DailyCreateManyInput>(record, collector)
}
```

### 3.5 向后兼容

- `collector` 为可选参数，**不传则所有分支零开销**，不影响现有逻辑
- 所有调用处不传 collector 时行为与改动前完全一致
- `validateOhlcv` 仅在 `collector != null` 时被调用

## 四、C — Sync service 调用处集成 collector

### 4.1 通用模式

在每个 sync service 的 `fetchAndMap` 回调中创建 collector，同步结束后批量写入 `DataValidationLog`。

**改动位置**（以 `MarketSyncService.syncDaily` 为例）：

```typescript
  const collector = new ValidationCollector(TushareSyncTaskName.DAILY)
  await this.syncByTradeDate({
    task: TushareSyncTaskName.DAILY,
    label: '日线',
    modelName: 'daily',
    targetTradeDate,
    fullSync: mode === 'full',
```

### 4.2 `flushValidationLogs` 公共方法

放在各 sync service 的基类或直接作为 helper 方法。建议放在 `SyncHelperService`：

```typescript
// sync-helper.service.ts 新增方法
async flushValidationLogs(collector: ValidationCollector): Promise<number> {
  const entries = collector.drain()
  if (entries.length === 0) return 0

  await this.prisma.dataValidationLog.createMany({
    data: entries.map((e) => ({
      task: collector.task,
      tradeDate: e.tradeDate ?? '',
      tsCode: e.tsCode ?? null,
      ruleName: e.ruleName,
      severity: e.severity,
      message: e.message,
      rawData: (e.rawData ?? null) as Prisma.InputJsonValue | null,
    })),
  })

  this.logger.log(`[校验日志] ${collector.task}: 写入 ${entries.length} 条异常记录`)
  return entries.length
}
```

```typescript
async checkCompleteness(dataSet: string, startDate: string, endDate: string): Promise<DataQualityReport> {
  // ... 获取 tradeDates
  // ... 查 existingRows → existingDates
  const missingDates = tradeDates.filter((d) => !existingDates.has(d))
  // ... 返回结果

// 改动后 — 增加 suspendAware 逻辑
async checkCompleteness(dataSet: string, startDate: string, endDate: string): Promise<DataQualityReport> {
  const config = this.DATA_SET_CONFIG[dataSet]
  if (!config) {
    return { dataSet, checkType: 'completeness', status: 'warn', message: `未知数据集: ${dataSet}` }
  }

  const tradeDates = await this.helper.getOpenTradeDatesBetween(startDate, endDate)
  if (!tradeDates.length) {
    return { dataSet, checkType: 'completeness', status: 'pass', message: `${dataSet} 检查范围内无交易日` }
  }

  // ── 1. 查出已有数据的交易日 ──
  const model = (this.prisma as any)[config.modelName]
  const existingRows = await model.findMany({
    select: { [config.tradeDateField]: true },
    where: { [config.tradeDateField]: { in: tradeDates } },
    distinct: [config.tradeDateField],
  })
  const existingDates = new Set<string>(existingRows.map((r: Record<string, string>) => r[config.tradeDateField]))

  // ── 2. 查出停牌日（仅行情类数据集需要排除） ──
  const suspendAwareDataSets = new Set(['daily', 'dailyBasic', 'adjFactor'])
  let suspendedDates: Set<string> | null = null

  if (suspendAwareDataSets.has(dataSet)) {
    suspendedDates = await this.getSuspendedTradeDates(startDate, endDate)
  }

  // ── 3. 计算真正的缺失日期 ──
  const rawMissing = tradeDates.filter((d) => !existingDates.has(d))
  let missingDates: string[]
  let suspendedCount = 0

  if (suspendedDates && suspendedDates.size > 0) {
    missingDates = []
    for (const d of rawMissing) {
      if (suspendedDates.has(d)) {
        suspendedCount++  // 停牌日的"缺失"是正常的
      } else {
        missingDates.push(d)
      }
    }
  } else {
    missingDates = rawMissing
  }

  // ── 4. 构建报告 ──
  if (missingDates.length === 0) {
    const suffix = suspendedCount > 0 ? `（另有 ${suspendedCount} 个停牌日正常缺失）` : ''
    return {
      dataSet,
      checkType: 'completeness',
      status: 'pass',
      message: `${dataSet} ${startDate}~${endDate} 数据完整（${tradeDates.length} 个交易日）${suffix}`,
      details: suspendedCount > 0 ? { suspendedCount } : undefined,
    }
  }

  const missingRatio = missingDates.length / tradeDates.length
  return {
    dataSet,
    checkType: 'completeness',
    status: missingRatio > 0.1 ? 'fail' : 'warn',
    message: `${dataSet} 缺失 ${missingDates.length}/${tradeDates.length} 个交易日数据` +
      (suspendedCount > 0 ? `（已排除 ${suspendedCount} 个停牌日）` : ''),
    details: {
      missingDates: missingDates.slice(0, 50),
      totalMissing: missingDates.length,
      ...(suspendedCount > 0 && { suspendedCount }),
    },
  }
}
```

### 5.3 新增辅助方法：`getSuspendedTradeDates`

```typescript
/**
 * 查询指定日期范围内所有停牌日期的集合。
 * 注意：SuspendD.tradeDate 是 String（YYYYMMDD 格式）。
 * 返回的是交易日维度的并集（任一股票停牌即该日有停牌记录），
 * 但对 completeness 检查而言，一个交易日只要有部分股票有数据就不算 gap。
 *
 * 此处做的是：若某个交易日 ALL 股票都停牌（极端情况，实际不会出现），
 * 才将该日视为"全市场停牌"。但更实际的场景是某些个股停牌导致该日数据条数偏少。
 * 对于 completeness（仅检查有无数据），只需排除"有SuspendD记录且daily确实无该日数据"的日期。
 *
 * 实际做法：查出范围内所有出现在 SuspendD 中的 tradeDate（去重），
 * 用于在 missingDates 中标注为"停牌相关缺失"。
 */
private async getSuspendedTradeDates(startDate: string, endDate: string): Promise<Set<string>> {
  const rows = await this.prisma.suspendD.findMany({
    select: { tradeDate: true },
    where: {
      tradeDate: { gte: startDate, lte: endDate },
    },
    distinct: ['tradeDate'],
  })
  return new Set(rows.map((r) => r.tradeDate))
}
```

### 5.4 停牌感知的精确说明

当前 `checkCompleteness` 的粒度是**交易日级别**：即某个交易日是否在目标表中存在至少一条记录。这个粒度下：

- **全日制停牌**（如 daily 表某个交易日确实没有任何数据）→ 这种情况**不会发生**，因为 daily 是全市场按日期拉取的，只要当天有任何股票交易就有数据
- **个股停牌**→ 某个交易日 daily 表仍有数据（其他股票在交易），所以日期不会被判为 missing

因此，**停牌排除的真正用途**是：对于 `SuspendD` 本身的 completeness 检查 — 有些交易日没有任何停牌事件发生，SuspendD 表中确实没有该日期的记录，这是正常的。将 SuspendD 从 `suspendAwareDataSets` 中移除即可避免误报。

但更重要的场景是 **per-stock completeness**（后续 Task 拓展），这里先为日期级检查提供基础设施，后续可直接复用 `getSuspendedTradeDates`。

**本次改动更核心的价值**是在报告中明确标注"停牌相关信息"，让管理员能区分数据缺失的原因。

### 5.5 新增：新股/退市股日期裁剪（嵌入 completeness 检查）

## 六、E — 盘后自动触发质量检查

### 6.1 改动文件

`src/tushare/sync/sync.service.ts`

### 6.2 方案

在同步完成的两个出口（有失败任务重试后 / 全部成功后），于 `broadcastSyncCompleted` 之后追加异步质量检查调用：

```typescript
// 现有代码（两处出口结构相同）：
await this.invalidateCachesAfterSync(result)
this.triggerHeatmapSnapshotAsync(result)
this.eventsGateway.broadcastSyncCompleted(result)

// 改为：
await this.invalidateCachesAfterSync(result)
this.triggerHeatmapSnapshotAsync(result)
this.triggerDataQualityCheckAsync(result) // ← 新增
this.eventsGateway.broadcastSyncCompleted(result)
```

### 6.3 新增方法

```typescript
constructor(
  // ... 现有依赖
  private readonly dataQualityService: DataQualityService,  // ← 新增注入
) {}

private triggerDataQualityCheckAsync(result: RunPlansResult): void {
  // 仅在有实际执行任务时才跑质量检查，避免空同步浪费
  if (result.executedTasks.length === 0) return

  void this.dataQualityService.runAllChecks().catch((error) => {
    this.logger.error(`盘后数据质量检查失败: ${(error as Error).message}`)
  })
}
```

### 6.4 设计要点

- **非阻塞**：使用 `void ... .catch()` 模式，与 `triggerHeatmapSnapshotAsync` 风格一致
- **不重复**：仅在 `executedTasks.length > 0` 时触发
- **触发时机**：在 `broadcastSyncCompleted` 之前调用，这样质量检查的日志和结果可以在前端通过后续查询获取
- **无需额外 cron**：质量检查跟随同步流程自然触发。bootstrap 同步和 scheduled 同步完成后都会执行
