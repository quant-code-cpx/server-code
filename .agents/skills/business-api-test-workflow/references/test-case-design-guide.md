# 测试用例设计指南

## 设计立场

你同时扮演资深测试工程师和资深后端开发，但**用例设计阶段只站在测试工程师视角**：把系统当黑盒，从业务契约推导期望结果。

当用户说“测试某个模块”时，必须按**全新测试任务**处理：重新理解业务、重新设计用例、重新判断风险。已有测试文件只能说明仓库里曾经有人写过测试，不能说明业务已经被正确覆盖。

不要问“代码现在会返回什么”，要问：

- 用户在什么场景下需要这个接口？
- 这个接口承诺解决什么业务问题？
- 如果这里错了，用户、资金、策略、数据或系统稳定性会受到什么影响？
- 哪些输入最容易让实现暴露错误？

## 信息来源优先级

从高到低使用：

1. 用户本轮明确要求、联调问题、前端期望类型。
2. 需求文档、设计文档、业务规则、金融公式、A 股交易规则。
3. Swagger / OpenAPI、Controller 路由、DTO 入参出参。
4. 外部 API 官方文档与真实返回字段（例如 Tushare）。

现有 `*.spec.ts` 不属于用例设计信息源。只有在测试用例文档已经完成后，才可以读取现有测试基础设施（如 helper、fixture、mock 工厂）来落地自动化；不得读取现有断言来决定业务期望。

禁止把 service/repository 当前实现作为期望来源。实现只能在修 Bug 阶段阅读。

## 全新测试模式

每个模块都按以下方式重新开始：

1. **先画业务地图**：接口服务哪个页面/流程、谁调用、为什么调用、成功后状态怎么变化。
2. **再列风险清单**：资金、安全、权限、数据精度、交易日、缓存、幂等、并发、性能。
3. **再写用例矩阵**：覆盖 `BIZ/EDGE/ERR/SEC/DATA/RACE/PERF/LOAD/STRESS/E2E/REG`。
4. **再映射自动化**：此时才看现有测试目录，判断哪些新用例需要新增、哪些旧弱用例需要替换。
5. **最后执行与修复**：发现 Bug 后再阅读实现定位根因。

禁止事项：

- 禁止先打开现有 spec 看它测了什么，再围绕它补漏。
- 禁止因为已有 spec 通过，就降低测试深度。
- 禁止把旧测试中的 mock 数据、断言值、错误行为当作正确业务规则。
- 禁止用“现有测试覆盖率”替代本轮业务风险覆盖。

## 业务理解检查清单

每个模块至少回答：

- **用户角色**：匿名、普通用户、管理员、系统任务、前端页面、队列任务。
- **业务目标**：查询、创建、交易模拟、风控检查、同步数据、触发通知、生成报告。
- **状态变化**：数据库写入、缓存刷新、队列入队、WebSocket 推送、审计日志。
- **核心不变量**：资金不能为负、权重和约等于 1、同一事件不可重复、用户不能访问他人资源。
- **时间规则**：YYYYMMDD、交易日、时区、T+1、节假日、盘前/盘后。
- **空值语义**：`null`、`0`、空数组、缺字段分别代表什么。
- **权限边界**：谁能看、谁能改、越权时 403 还是 404。
- **数据来源**：DB、Redis、Tushare、任务产物、前端传参。
- **失败策略**：抛业务错误、降级为空、重试、跳过、告警。
- **可观测性**：日志、指标、错误信息是否足够定位问题。

## 用例分类矩阵

| 标签     | 类型       | 必测问题                                             |
| -------- | ---------- | ---------------------------------------------------- |
| `BIZ`    | 常规业务   | 正常路径、核心公式、状态流转、字段语义是否正确       |
| `EDGE`   | 边界场景   | 空数据、单条数据、极值、临界阈值、分页边界、日期边界 |
| `ERR`    | 错误处理   | 参数非法、依赖失败、数据不存在、外部超时、异常映射   |
| `SEC`    | 安全       | 鉴权、越权、角色、注入、敏感信息、账户枚举、限流     |
| `DATA`   | 数据一致性 | DB 与缓存、同步字段映射、精度、幂等、重复数据、事务  |
| `RACE`   | 并发竞态   | 重复提交、并发更新、并发刷新 token、队列重复消费     |
| `PERF`   | 性能基准   | 单接口响应时间、慢查询、序列化成本、缓存命中收益     |
| `LOAD`   | 负载       | 目标并发与吞吐下错误率、p95/p99、资源占用            |
| `STRESS` | 压力       | 超出设计容量后的降级、限流、错误是否可控             |
| `SOAK`   | 稳定性     | 长时间运行、内存泄漏、连接泄漏、队列堆积             |
| `E2E`    | 端到端     | 跨模块业务闭环是否可用                               |
| `REG`    | 回归       | 已修复 Bug 是否不会复现                              |

## 用例设计层次

### 1. 契约层

验证接口是否符合约定：

- 路径、POST 方法、Body DTO、Param、鉴权、角色。
- 响应字段名、类型、空值、分页结构、错误结构。
- `TransformInterceptor` 包装后前端实际看到的结构。

### 2. 业务规则层

验证业务正确性：

- 金融计算必须用可手算小数据。
- 状态机必须覆盖合法/非法跳转。
- 聚合统计必须能追溯到原始样本。
- 排序、分页、筛选必须验证组合条件，不只验证返回非空。

### 3. 数据层

验证数据可靠性：

- 上游字段映射、Decimal/BigInt/Date 精度。
- 缓存穿透后写回格式与读取格式一致。
- 重复同步、重复提交、幂等键、唯一约束。
- 事务失败时不能半成功。

### 4. 稳定性层

验证系统在“不顺利”时仍可控：

- DB/Redis/Tushare 超时或异常。
- 队列任务失败重试。
- 大分页、大日期范围、大结果集。
- 并发请求导致的锁、重复写入、脏读。

### 5. 性能与压力层

至少设计以下指标：

- 基准数据量：如 1 用户、100 组合、1 年日线、10 万事件。
- 并发模型：恒定并发、阶梯升压、突发流量。
- 指标：吞吐、平均耗时、p50/p95/p99、错误率、CPU/内存、DB 慢查询、Redis 命中率。
- 阈值：先用合理目标或历史基线；没有基线就记录本轮观测值，不强行虚构达标线。
- 稳定性：持续运行时间、连接泄漏、队列积压、日志错误。

## 期望值推导规范

- 期望结果必须写出业务依据。
- 数值计算必须写手算过程或独立公式。
- 日期必须写明时区和交易日假设。
- 排序必须写清主排序和并列排序规则；不明确则标记待澄清。
- 空数据必须写清应返回空数组、`null`、`0` 还是业务错误；不明确则标记待澄清。

## 待澄清记录规范

待澄清不是失败，而是专业测试的一部分。记录格式：

| 编号  | 问题                      | 影响用例     | 可选方案                     | 建议默认       | 阻塞级别 |
| ----- | ------------------------- | ------------ | ---------------------------- | -------------- | -------- |
| Q-001 | 市值分桶 LARGE 阈值是多少 | CAL-PERF-001 | >100亿 / >300亿 / 按前端配置 | 与前端统一配置 | 中       |

如果待澄清会影响核心业务正确性，先不要改代码；可以写探测性测试或文档化预期。

## 优先级建议

- `P0`：安全、资金、核心计算、数据污染、生产不可用。
- `P1`：主流程不可用、前端关键页面异常、重要字段错误。
- `P2`：边界错误、低频流程、可降级功能。
- `P3`：体验问题、提示不准确、非阻塞性能优化。

## 反模式

- 只断言 `toBeDefined()` 或 `status 201`。
- Mock 什么就期望什么。
- 先跑一遍代码，把实际输出复制成期望值。
- 先看现有 spec，然后只做局部补丁式测试设计。
- 因为已有测试通过，就认为该模块业务风险低。
- 只测 happy path，不测权限/空值/边界/异常。
- 性能测试不记录环境和数据规模。
- 压测外部第三方 API，导致结果不可控或触发频控。
