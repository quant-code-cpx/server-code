# Tushare 数据扩展第三批 — 后端设计

> **范围**：`fina_mainbz`（主营业务构成）、`index_classify` + `index_member_all`（申万行业分类与成分）、`repurchase`（股票回购）、`cb_basic` + `cb_daily`（可转债基础信息 + 行情）
>
> **积分要求**：fina_mainbz / index_classify / index_member_all / cb_basic / cb_daily 各 2000；repurchase 仅 600
>
> **目标**：新增 6 张 Prisma 表、6 个 mapper、6 个同步计划项

---

## 一、接口字段确认（来源：Tushare 官方文档）

### 1.1 fina_mainbz（主营业务构成，doc_id=81）

**限制**：2000 积分起，`ts_code` 为必填，单次最大 100 行，总量不限。支持 `type`（P 按产品 / D 按地区 / I 按行业）和 `period` 筛选。`fina_mainbz_vip`（5000 积分）支持按 period 查全市场。

| Tushare 字段 | 类型  | 必有 | 说明                                     |
| ------------ | ----- | ---- | ---------------------------------------- |
| ts_code      | str   | Y    | TS 股票代码                              |
| end_date     | str   | Y    | 报告期                                   |
| bz_item      | str   | Y    | 主营业务来源（产品名 / 地区名 / 行业名） |
| bz_sales     | float | N    | 主营业务收入（元）                       |
| bz_profit    | float | N    | 主营业务利润（元）                       |
| bz_cost      | float | N    | 主营业务成本（元）                       |
| curr_type    | str   | N    | 货币代码（CNY 等）                       |
| update_flag  | str   | N    | 是否更新                                 |

### 1.2 index_classify（申万行业分类，doc_id=181）

**限制**：2000 积分起。返回行业分类目录（非成分股）。支持 `level`（L1/L2/L3）和 `src`（SW2014/SW2021）。

| Tushare 字段  | 类型 | 必有 | 说明                     |
| ------------- | ---- | ---- | ------------------------ |
| index_code    | str  | Y    | 指数代码（如 801010.SI） |
| industry_name | str  | Y    | 行业名称                 |
| parent_code   | str  | Y    | 父级代码（一级为 0）     |
| level         | str  | Y    | 行业层级（L1/L2/L3）     |
| industry_code | str  | Y    | 行业代码                 |
| is_pub        | str  | Y    | 是否发布了指数           |
| src           | str  | N    | 行业分类来源（SW）       |

### 1.3 index_member_all（申万行业成分-分级，doc_id=335）

**限制**：2000 积分起，单次最大 2000 行，总量不限。支持 `l1_code` / `l2_code` / `l3_code` / `ts_code` / `is_new` 筛选。

| Tushare 字段 | 类型 | 必有 | 说明            |
| ------------ | ---- | ---- | --------------- |
| l1_code      | str  | Y    | 一级行业代码    |
| l1_name      | str  | Y    | 一级行业名称    |
| l2_code      | str  | Y    | 二级行业代码    |
| l2_name      | str  | Y    | 二级行业名称    |
| l3_code      | str  | Y    | 三级行业代码    |
| l3_name      | str  | Y    | 三级行业名称    |
| ts_code      | str  | Y    | 成分股票代码    |

| in_date      | str  | Y    | 纳入日期        |

### 2.1 `prisma/tushare_fina_mainbz.prisma`

```prisma
// 主营业务构成（来自 Tushare fina_mainbz 接口，doc_id=81）
// 同一 ts_code + end_date 下可能有多条记录（按产品/地区/行业拆分）
model FinaMainbz {
  id         Int      @id @default(autoincrement())

  tsCode     String   @map("ts_code") @db.VarChar(16)
  endDate    DateTime @map("end_date") @db.Date
  bzItem     String   @map("bz_item") @db.VarChar(128)

  bzSales    Float?   @map("bz_sales")
  bzProfit   Float?   @map("bz_profit")
  bzCost     Float?   @map("bz_cost")
  currType   String?  @map("curr_type") @db.VarChar(8)
  updateFlag String?  @map("update_flag") @db.VarChar(4)

  syncedAt   DateTime @default(now()) @map("synced_at")

  @@unique([tsCode, endDate, bzItem])
  @@index([tsCode])
  @@index([endDate])
  @@map("financial_main_business")
}
```

## 四、Mapper 函数设计

### 4.1 mapFinaMainbzRecord

```typescript
export function mapFinaMainbzRecord(
  r: TushareRecord,
  collector?: ValidationCollector,
): Omit<Prisma.FinaMainbzCreateManyInput, 'id'> | null {
  const tsCode = readString(r, 'ts_code')
  const endDate = readDate(r, 'end_date')
  const bzItem = readString(r, 'bz_item')
  if (!tsCode || !endDate || !bzItem) {
    collector?.add({ field: 'PK', message: 'ts_code/end_date/bz_item 缺失', record: r })
    return null
  }
  return {
    tsCode,
    endDate,
    bzItem,
    bzSales: readNumber(r, 'bz_sales'),
    bzProfit: readNumber(r, 'bz_profit'),
    bzCost: readNumber(r, 'bz_cost'),
    currType: readString(r, 'curr_type'),
    updateFlag: readString(r, 'update_flag'),
  }
}
```

## 五、同步策略设计

| 接口               | 同步策略               | 原因                                              | 归属服务               |
| ------------------ | ---------------------- | ------------------------------------------------- | ---------------------- |
| `fina_mainbz`      | **按 ts_code 遍历**    | `ts_code` 为必填；单次限 100 行                   | `FinancialSyncService` |
| `index_classify`   | **全量替换**           | 行业目录总量固定（~500 条），一次拉完替换         | `BasicSyncService`     |
| `index_member_all` | **按 l1_code 遍历**    | 按一级行业拉取成分，31 个一级行业，每个 ≤ 2000 行 | `BasicSyncService`     |
| `repurchase`       | **按 ann_date 批量**   | 支持全市场按公告日查询                            | `FinancialSyncService` |
| `cb_basic`         | **全量替换**           | 可转债存量 ~500-600 只，一次拉完                  | `BasicSyncService`     |
| `cb_daily`         | **按 trade_date 批量** | 每日行情，按交易日拉取                            | `MarketSyncService`    |

### 5.1 fina_mainbz 同步逻辑

```
日常同步：
  1. 获取最近有财报公告的股票列表
  2. 对每只股票调用 fina_mainbz(ts_code=xxx, type='P')
  3. 每次最多 100 行（单只股票历史通常 < 100 行）
  4. 按 unique(tsCode, endDate, bzItem) 执行 upsert
  5. 300ms 节流

回补同步：
  1. 遍历全部股票列表
  2. 对每只股票分别拉取 type='P'（按产品）数据
  3. ~5000 只 × 300ms ≈ 25 分钟
```

### 5.2 index_classify 同步逻辑

```
同步策略（全量替换，无日常/回补区分）：
  1. 分别拉取 L1、L2、L3（各加 src='SW2021'）
  2. 合并后 deleteMany + createMany 全量替换
  3. 总计 ~500 行，3 次请求，无频控压力
  4. 每周同步一次即可（行业分类极少变动）
```

### 5.3 index_member_all 同步逻辑

```
同步策略（全量替换）：
  1. 先从 index_classify 表获取所有 31 个 L1 代码
  2. 对每个一级行业调用 index_member_all(l1_code=xxx, is_new='Y')
  3. 单次最多 2000 行（一个一级行业最新成分通常 < 2000）
  4. deleteMany(isNew='Y') + createMany 全量替换最新成分
  5. 如需保留历史调入调出记录，可追加拉取 is_new='N'

回补（含历史）：
  1. 拉取 is_new 不限（含历史剔除记录），按 unique 去重写入
```

### 5.4 repurchase 同步逻辑

```
日常同步：
  1. 取当天日期作为 ann_date
  2. 调用 repurchase(ann_date=yyyyMMdd)
  3. 按 unique(tsCode, annDate) 执行 upsert

回补同步：
  1. 从 2010-01-01 开始，按月使用 start_date + end_date 拉取
  2. 不填参数时默认返回 2000 条，故需设定日期范围约束
  3. 300ms 节流
```

### 5.5 cb_basic 同步逻辑

```
同步策略（全量替换）：
  1. 不带参数拉取全部可转债基础信息（存量 ~500-600 只）
  2. 若超 2000 行需翻页
  3. deleteMany + createMany 全量替换
  4. 每周同步一次即可
```

### 5.6 cb_daily 同步逻辑

```
日常同步：
  1. 取最近 5 个交易日
  2. 对每个交易日调用 cb_daily(trade_date=yyyyMMdd)
  3. 每次返回 ~300-500 行（当日有交易的可转债）
  4. deleteMany(tradeDate=xxx) + createMany

回补同步：
  1. 从 2018-01-01 开始（可转债市场扩容起点），遍历所有交易日
  2. 每次 ~300-500 行，无频控压力
  3. ~1500 个交易日 × 300ms ≈ 8 分钟
```

## 八、频控与风险评估

| 接口               | 每分钟频次 | 日常同步请求次数        | 回补请求次数             | 风控策略           |
| ------------------ | ---------- | ----------------------- | ------------------------ | ------------------ |
| `fina_mainbz`      | ~100       | ~100 次（近期有财报的） | ~5000 次（全部股票）     | **高**，300ms 节流 |
| `index_classify`   | ~200       | 3 次（L1+L2+L3）        | 同日常                   | 无压力             |
| `index_member_all` | ~200       | 31 次（31 个一级行业）  | ~62 次（含历史）         | 极低               |
| `repurchase`       | ~200       | 1-2 次（当日公告）      | ~200 次（16 年 × 12 月） | 低                 |
| `cb_basic`         | ~100       | 1 次                    | 同日常                   | 无压力             |
| `cb_daily`         | ~200       | 5 次（近 5 交易日）     | ~1500 次（2018 至今）    | 低，300ms 节流     |

**总体评估**：

- 日常同步：`fina_mainbz` 是该批压力最大的接口（需遍历股票），其余均极轻量
- 回补：`fina_mainbz` 需约 25 分钟，`cb_daily` 约 8 分钟，其余均可在数分钟内完成
- 该批包含 3 个全量替换类接口（index_classify / index_member_all / cb_basic），均为低频小数据量

## 九、与前两批的依赖关系

- `index_classify` + `index_member_all` 组合可构建完整的申万行业映射；`index_member_all` 的 `syncIndexMemberAll` 依赖 `index_classify` 表获取一级行业列表，因此 **`index_classify` 必须先于 `index_member_all` 执行**（order 50 < 55 已保证）
- `cb_basic` 为 `cb_daily` 提供基础信息参照；**`cb_basic` 必须先于 `cb_daily` 存在**（order 60 < 180 已保证）
- `fina_mainbz` 与第二批的 `fina_audit` / `disclosure_date` 同属财务域，共享 `FinancialSyncService`
- `repurchase` 可与第二批的 `stk_holdertrade`（股东增减持）配合，形成公司治理事件数据集
- 建议按"第一批 → 第二批 → 第三批"顺序实施，但三批之间无硬依赖，可并行开发不同批的独立接口
