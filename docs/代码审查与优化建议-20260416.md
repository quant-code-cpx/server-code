## 二、性能优化建议

### 2.1 🔴 数据库连接池过小（高优先级）

**位置**：`src/shared/prisma.service.ts:7`

```
DEFAULT_PRISMA_CONNECTION_LIMIT = 15
```

**问题**：
- PostgreSQL 允许 100 连接，但应用侧仅 15
- 并发回测任务 + HTTP 请求 + 定时同步可能同时竞争连接
- Queue Processor 中回测/Walk-Forward/Comparison 同时运行时易耗尽
- 30s 事务超时期间持续占用连接

**建议**：
- 生产环境建议调为 30-50（通过已有 `PRISMA_CONNECTION_LIMIT` 环境变量）
- 补充 Prometheus 指标监控活跃连接数
- 长事务操作考虑使用独立的只读连接（read replica）

### 2.2 🔴 Redis 缓存无驱逐策略（高优先级）

**位置**：`dockerfiles/redis/redis.conf`

**问题**：
- `maxmemory-policy noeviction`：Redis 满时拒绝写入，直接报错
- 缓存 key 通过 SHA1(请求参数) 生成，不同参数组合会产生无限 key
- Namespace Set（`cache:namespace:*`）用于追踪 key，但 Set 自身没有 TTL
- 当 Redis 达到 512MB 上限时，所有缓存写入失败

**建议**：
- 修改驱逐策略为 `volatile-lru` 或 `allkeys-lru`
- 为 Namespace Set 添加 TTL（如 7 天），避免永久积累
- 添加 Redis 内存使用率的 Prometheus 告警
- 考虑为高频 key（如 `stock:list:*`）设置更短的 TTL

### 2.3 🟠 $queryRaw 查询性能（中优先级）

**涉及文件**（39 处 `$queryRaw` 调用）：
- `stock-analysis.service.ts`（10 处）
- `factor-compute.service.ts`（20 处）
- `market.service.ts`（9 处）
- `industry-rotation.service.ts`（11 处）
- `portfolio-risk.service.ts`（5 处）
- `heatmap.service.ts`（4 处）

**问题**：
- 原生 SQL 不受 Prisma 查询优化器管理，无法自动 `select` 裁剪
- 部分查询聚合了多表 JOIN，但无 EXPLAIN 分析验证执行计划
- `stock-analysis.service.ts` 单次请求可能触发 5+ 条 SQL

**建议**：
- 对 TOP 10 高频原生查询做 `EXPLAIN ANALYZE` 分析
- 考虑为复杂聚合查询创建 PostgreSQL 物化视图（Materialized View），尤其是市场概览类数据
- 评估是否可将部分 `$queryRaw` 转为 Prisma `findMany` + `select`，获得类型安全

### 2.4 🟠 缺失的数据库索引（中优先级）

**已识别的缺失索引**：

| 表名 | 缺失索引 | 影响 |
|------|----------|------|
| `income_statements` | `@@index([tsCode])` 单列 | 按股票查全部利润表走全表扫描 |
| `balance_sheets` | `@@index([tsCode])` 单列 | 同上 |
| `cashflow_statements` | `@@index([tsCode])` 单列 | 同上 |
| `financial_indicator_snapshots` | `@@index([tsCode])` 单列 | 同上 |
| `business_revenue` (fina_mainbz) | `@@index([tsCode])` | 主营业务查询无索引 |
| `macro_cpi` | `@@index([month])` | 按月查询 CPI 走全表 |
| `macro_ppi` | `@@index([month])` | 同上 |
| `macro_gdp` | `@@index([quarter])` | 按季度查询 GDP 走全表 |
| `shareholder_trading` (stk_holdertrade) | `@@index([tsCode, endDate(sort: Desc)])` | 按股票查股东增减持效率低 |

**预估改善**：财务报表和宏观数据查询速度可提升 10-100 倍（取决于数据量）。

### 2.5 🟠 大文件 Service 拆分（中优先级）

| 文件 | 行数 | 建议 |

| `stock-analysis.service.ts` | 1108 | 按分析类型拆为 technical / timing / chip / margin / factor 五个子 service |
| `factor-compute.service.ts` | 1029 | 将矩阵运算、截面计算、时序计算分离 |
| `market.service.ts` | 948 | 按资金流/情绪/估值/板块拆分 |
| `factor-expression.service.ts` | 911 | 表达式解析器和求值器可分离 |
| `data-quality.service.ts` | 775 | 按检查策略类型拆分 |
| `industry-rotation.service.ts` | 769 | 按分析维度（收益/动量/资金/估值）拆分 |
| `market-sync.service.ts` | 700 | 行情和指标同步分离 |

### 2.6 🟡 缓存策略未覆盖高频计算（低优先级）

**未缓存但计算密集的场景**：

| 场景 | 调用频率 | 计算复杂度 |
|------|----------|------------|
| 因子分析结果（IC/分位数/相关矩阵） | 按需 | 高（全市场截面） |
| 行业轮动排名 | 频繁 | 中（多维聚合） |
| 股票技术指标 | 高频 | 低-中（纯计算） |
| 市场估值分位 | 频繁 | 中（历史分位数） |
| 热力图数据 | 频繁 | 高（全市场聚合） |

**建议**：
- 行业轮动、市场估值等数据按交易日缓存（TTL 24h），盘后同步完成后自动失效
- 因子分析结果缓存 key 可以包含参数 hash + 最新交易日
- 热力图快照机制已有，但实时热力图数据仍应缓存

### 2.7 🟡 PostgreSQL 语句超时偏短

**位置**：`dockerfiles/postgresql/postgresql.conf`

```
statement_timeout = 15000  # 15秒
```

**问题**：
- 因子预计算（全市场截面）、参数扫描（批量回测）等操作可能超过 15 秒
- 超时后直接抛异常，无优雅降级
- 报告生成涉及多表聚合，也可能超时

**建议**：
- 全局 `statement_timeout` 保持 30s
- 对已知的长查询（回测、预计算、报告）在事务中显式设置 `SET LOCAL statement_timeout = 120000`

## 三、代码质量与架构优化

### 3.1 硬编码值需提取为配置

| 位置 | 硬编码值 | 建议 |
|------|----------|------|
| `prisma.service.ts:13` | `SLOW_QUERY_THRESHOLD_MS = 500` | 已有 env 读取，但默认值建议可调 |
| `cache.service.ts:35-36` | `SCAN_COUNT = 200, DELETE_CHUNK_SIZE = 200` | 提取为可配置常量 |
| `logging.interceptor.ts:6` | `EXCLUDED_PATHS` 固定数组 | 允许通过配置扩展 |
| `events.gateway.ts:37` | `cors: { origin: '*' }` | 生产环境应读取 `CORS_ORIGIN` |
| `tushare.constant.ts:191` | `MONEYFLOW_RECENT_TRADE_DAYS = 60` | 提供环境变量覆盖 |

### 3.2 循环依赖需解耦

当前存在 4 处 `forwardRef`：

| 模块 A | 模块 B | 原因 |
|--------|--------|------|
| SignalModule | PortfolioModule | 信号服务需读取组合，组合也需信号 |
| SignalModule | BacktestModule | 信号生成依赖回测结果 |
| TushareModule | SyncService | 模块内部循环 |

**建议**：
- 引入 EventEmitter2 事件总线，Signal → Portfolio 改为事件驱动
- 或提取共享的 DTO/Interface 层，打破直接服务依赖

### 3.3 WebSocket CORS 安全隐患

**位置**：`src/websocket/events.gateway.ts:37`

```typescript
cors: { origin: '*' }
```

**问题**：
- HTTP 层已有 CORS 白名单（`main.ts` 读取 `CORS_ORIGIN` 环境变量）
- 但 WebSocket 层 CORS 完全开放，绕过了 HTTP 层安全策略
- 攻击者可从任意域名建立 WebSocket 连接

**建议**：
```typescript
cors: {
  origin: process.env.CORS_ORIGIN?.split(',') ?? ['http://localhost:5173'],
}
```

### 3.4 WebSocket 断开无清理

**位置**：`src/websocket/events.gateway.ts:62-63`

```typescript
handleDisconnect(client: Socket) {
  this.logger.log(`Client disconnected: ${client.id}`)
}
```

**问题**：
- 仅记录日志，未清理用户房间状态
- 长时间运行后，`backtest:${jobId}` 类房间会持续累积
- 无连接数监控指标

**建议**：
- 断开连接时显式从所有已加入的房间退出
- 添加定期清理已完成回测任务的房间
- 添加 Prometheus 指标：`ws_active_connections`、`ws_rooms_total`

### 3.5 Queue 缺少 DLQ（Dead Letter Queue）

**位置**：`src/queue/backtesting/backtesting.processor.ts`

**问题**：
- 回测任务失败后仅更新数据库状态和发送 WebSocket 通知
- 无 DLQ 机制，失败任务无法被自动重试或人工审查
- 数据库更新失败时仅记录日志，用户可能无感知

**建议**：
- BullMQ 配置 `attempts: 3` + `backoff: { type: 'exponential', delay: 5000 }`
- 超过重试次数的任务移入 DLQ
- DLQ 中的任务暴露管理接口（查看/重试/删除）

### 3.6 错误处理不一致

**具体表现**：
- `GlobalExceptionsFilter` 对开发/生产模式做了区分，但部分 Service 直接 throw 原生 Error 而非 BusinessException
- Queue Processor 中 DB 更新失败时 catch 仅 log，不做补偿
- Tushare API 40203 频控有重试，但其他错误码无分类处理

**建议**：
- 统一使用 `BusinessException` + `ErrorEnum`，避免裸 Error
- Queue 失败补偿改为：log + 通知 + 标记为需人工处理
- Tushare 错误码新增分类：认证失败(40101)、无权限(40301)、无数据(40001)

### 3.7 metrics 补充建议

**当前已有指标**：
- HTTP 请求计数/延迟（HttpMetricsInterceptor）
- Prisma 查询计数/延迟
- Queue 指标（QueueMetricsService）

**建议补充**：

| 指标 | 类型 | 用途 |
|------|------|------|
| `redis_memory_usage_bytes` | Gauge | 监控 Redis 内存使用 |
| `cache_hit_ratio` | Gauge (per namespace) | 缓存命中率趋势 |
| `ws_active_connections` | Gauge | WebSocket 连接数 |
| `prisma_pool_active_connections` | Gauge | 数据库连接池使用率 |
| `tushare_sync_duration_seconds` | Histogram | 各同步任务耗时 |
| `tushare_api_errors_total` | Counter (by error code) | API 错误分类统计 |
| `backtest_job_duration_seconds` | Histogram | 回测任务耗时分布 |

## 四、Tushare 数据挖掘建议（≤6000 积分）

### 4.1 当前已接入的 Tushare 接口

已覆盖 **57 个** Tushare API，涵盖 9 大分类：
- 基础数据（8）、行情数据（10）、财务数据（18）、资金流向（4）
- 因子数据（5）、基金数据（3）、宏观经济（4）、期权数据（2）
- 另类数据（3：龙虎榜/大宗/融资融券）

### 4.2 推荐新增接口（6000 积分以内）

以下接口在当前数据体系中有明确应用场景，且积分可控：

#### 方案 A：事件驱动增强（总计 4000 积分）

| 接口 | 积分 | 用途 | 应用场景 |
|------|------|------|----------|
| `limit_list_d`（每日涨跌停统计） | 2000 | 涨停/跌停个股明细，含封单量 | 打板策略、连板监控、市场情绪指标增强 |
| `stk_surv`（股票技术面因子-VIP） | 2000 | 量价异动、形态突破、技术面评分 | 增强现有 Pattern 模块的预筛选能力 |

**价值分析**：
- `limit_list_d` 可直接补全 Alert 模块的"连板监控"功能，当前仅靠 pct_chg 推算涨停不够精确
- `stk_surv` 提供 Tushare 官方的技术面因子计算结果，可与本地计算做交叉验证

#### 方案 B：ETF/LOF 量化增强（总计 4000 积分）

| 接口 | 积分 | 用途 | 应用场景 |
|------|------|------|----------|
| `fund_portfolio`（基金持仓） | 2000 | 公募基金季度持仓明细 | 机构持仓分析、重仓股策略 |
| `fund_share`（基金份额变动） | 2000 | 基金份额申购赎回数据 | ETF 资金流向分析、机构行为监控 |

**价值分析**：
- 公募基金持仓是市场关注度极高的数据，可构建"跟踪明星基金经理"策略
- 基金份额变动可用于判断机构资金进出趋势

#### 方案 C：量价深度数据（总计 4000 积分）

| 接口 | 积分 | 用途 | 应用场景 |
|------|------|------|----------|
| `stk_mins`（分钟线行情） | 2000 | 1/5/15/30/60 分钟 K 线 | 日内策略研究、开盘/尾盘模式分析 |
| `daily_info`（每日市场全景） | 2000 | 全市场涨跌停家数/成交额/换手率等 | 替代现有自行计算的市场情绪指标 |

**价值分析**：
- 分钟线数据是进阶量化研究的必备，当前仅有日/周/月三个级别
- `daily_info` 可直接替代 MarketService 中多条 SQL 聚合的市场概览计算

#### 推荐组合（6000 积分内）

| 推荐 | 组合 | 总积分 | 理由 |
|------|------|--------|------|
| ⭐ **首选** | `limit_list_d` + `fund_portfolio` + `daily_info` | 6000 | 覆盖事件+机构+市场三个维度 |
| 次选 | `limit_list_d` + `stk_surv` + `fund_share` | 6000 | 侧重交易信号增强 |
| 备选 | `stk_mins` + `fund_portfolio` | 4000 | 侧重深度研究 |

### 4.3 不建议接入的接口

| 接口 | 积分 | 原因 |
|------|------|------|
| `news`（新闻资讯） | 5000 | 积分高，且需 NLP 处理才有价值，投入产出比低 |

## 五、功能缺失分析（除 AI 外）

### 5.1 🔴 高优先级缺失

#### 5.1.1 数据导出功能

**现状**：全部 250+ 接口返回 JSON，无 CSV/Excel 导出能力。

**需求场景**：
- 回测交易明细导出为 Excel 供线下分析
- 因子值导出供 Python/R 二次研究
- 组合持仓导出为 CSV 对接券商系统

**建议实现**：
- 添加通用 `ExportService`，支持 CSV 和 XLSX 格式
- 在 Backtest、Factor、Portfolio 模块各增加 `export` 端点
- 大数据量导出采用流式写入，避免内存溢出

#### 5.1.2 定时报告与邮件推送

**现状**：Report 模块支持手动生成报告，但无定时生成和推送。

**需求场景**：
- 每日盘后自动生成组合日报
- 每周发送因子跟踪报告
- 月度策略绩效回顾

**建议实现**：
- 新增 ScheduledReportService，支持 cron 配置
- 报告生成后存储为 PDF（依赖 Report 模块现有的 Handlebars 模板）
- 通过 SMTP 或 WebSocket 推送

#### 5.1.3 多用户协作与分享

**现状**：所有资源（策略、组合、因子、笔记）为用户私有，无分享机制。

**需求场景**：
- 团队内分享策略模板
- 共享自定义因子给协作者
- 研究笔记评论与协作

**建议实现**：
- 添加资源级权限控制（私有/团队可见/公开只读）
- 策略和因子支持"发布到市场"功能
- 研究笔记支持 @提及 和评论

### 5.2 🟠 中优先级缺失

#### 5.2.1 交易日历与事件提醒

**现状**：Alert 模块有事件日历雏形（`AlertCalendarService`），但数据来源不完整。

**缺失内容**：
- 财报披露日历（已有 `disclosure_date` 数据，但未与日历关联）
- 限售股解禁日历（已有 `share_float` 数据，但未提供日历视图）
- 分红除权日历（已有 `dividend` 数据，但未构建前端可用的日历 API）
- IPO / 可转债发行日历

**建议**：将已有的 Tushare 数据整合为统一的事件日历 API，每日同步后自动刷新。

#### 5.2.2 策略模板市场

**现状**：仅有 5 种内置策略模板，用户自定义策略无法复用。

**建议**：
- 支持将策略标记为"模板"并分类展示
- 提供预设的经典策略（双均线、MACD 金叉、布林带突破等）
- 策略模板支持参数化，用户可 fork 后修改

#### 5.2.3 批量操作接口

**现状**：多数 CRUD 仅支持单条操作。

**缺失的批量接口**：
- 批量添加持仓（从 Excel 导入）
- 批量创建价格预警规则
- 批量删除回测记录
- 批量更新策略状态

#### 5.2.4 数据可视化后端支持

**现状**：前端需要自行聚合多个接口数据来渲染图表。

**建议增加的聚合 API**：
- 行业对比雷达图数据（一次性返回多维度对比）
- 组合历史持仓变化（时间轴上的持仓演变）
- 因子暴露变化趋势（时序因子暴露热力图）

### 5.3 🟡 低优先级缺失

#### 5.3.1 操作历史与撤销

**现状**：审计日志仅记录管理员操作，普通用户操作无历史。

**建议**：
- 为组合操作（加仓/减仓/删除）提供操作历史
- 支持"撤销上一步操作"（如误删持仓）

#### 5.3.2 用户自定义仪表盘

**现状**：首页仪表盘数据固定。

**建议**：
- 支持用户自选仪表盘组件
- 组件类型：市场概览、组合盈亏、自选股涨跌、信号列表等
- 配置持久化到用户偏好

#### 5.3.3 WebSocket 事件回放

**现状**：WebSocket 仅推送实时事件，断开重连后丢失。

**建议**：
- 维护最近 N 条事件的环形缓冲区
- 客户端重连时补发离线期间的事件
- 或结合 Notification 持久化机制做去重推送

#### 5.3.4 API 使用量统计

**现状**：有 HTTP 请求指标（Prometheus），但无用户级 API 调用统计。

**建议**：
- 按用户统计 API 调用量（用于未来的配额管理）
- 识别高频调用模式，优化热点接口
- 为 Tushare 数据查询统计缓存命中率

## 六、安全加固补充建议

### 6.1 WebSocket 层 CORS 开放

**已在 3.3 节详述**。WebSocket 的 `origin: '*'` 需与 HTTP 层保持一致。

### 6.2 Redis AOF 配置风险

**位置**：`dockerfiles/redis/redis.conf`

```
appendfsync everysec    # 每秒刷盘
save 3600 1             # 每小时 RDB
save 300 100            # 5分钟内100+变更时 RDB
```

**问题**：
- AOF + RDB 双持久化导致磁盘 IO 增加
- RDB 保存时 Redis 主线程 fork，可能造成毫秒级延迟抖动
- `appendfsync everysec` 最多丢失 1 秒数据（缓存场景可接受）

**建议**：
- 缓存用途的 Redis 可禁用 RDB（`save ""`），仅保留 AOF
- 添加 `auto-aof-rewrite-percentage 100` 和 `auto-aof-rewrite-min-size 256mb` 防止 AOF 无限增长

### 6.3 限流规则过于宽松

**位置**：`src/app.module.ts`

```typescript
ThrottlerModule.forRoot([{ name: 'default', ttl: 10000, limit: 10 }])
```

**问题**：
- 全局 10 秒 10 次 = 60 req/min，对数据密集查询接口偏宽松
- 关键操作（登录、注册、密码重置）已有独立限流
- 但数据导出、批量查询等未来接口可能被滥用

**建议**：
- 数据密集型接口（因子计算、全市场筛选）添加独立 `@Throttle()`
- 未认证用户的限流应更严格
