# 业务优先的 API 测试工作流

## 目标

把一次测试任务沉淀为可复用闭环：**业务理解 → 测试用例文档 → 自动化/手工执行 → 缺陷归纳 → 后端修复 → 回归验证 → 本轮总结**。

最高优先级是**准确发现业务问题**，其次是稳定可复现，最后才是执行速度。测试不是证明现有代码没问题，而是主动寻找它可能错在哪里。

## 何时使用

- 用户要求测试某个接口、模块、业务流程、联调问题清单或全量模块。
- 用户说“测试某个模块/接口”时，默认按**全新测试任务**启动，而不是基于现有 spec 做增量补丁。
- 需要从业务场景出发设计测试用例，而不是照着实现写断言。
- 需要生成测试用例文档、测试执行报告、缺陷清单、回归结论。
- 需要测试后以资深后端开发身份修 Bug，再以测试身份回归。
- 需要覆盖常规流程、边界、异常、安全、数据一致性、并发、性能、压力、稳定性。

## 硬性原则

1. **模块测试默认全新开始**：用户指定某个模块时，必须从 0 建立该模块的业务理解、风险模型和测试用例矩阵；不能因为仓库已有 spec、已有测试清单或已有覆盖率，就跳过重新设计。
2. **现有测试不得干扰用例设计**：设计测试用例前不要阅读现有 `*.spec.ts` 的用例、断言、mock 返回值或覆盖结论；现有测试不能作为“已经测过/无需再测”的依据。
3. **先业务，后实现**：设计测试用例前不要阅读 service/repository 等实现逻辑；只允许使用业务文档、接口契约、Swagger、前端类型/联调期望、Controller 路由和 DTO 入参出参来理解接口边界。
4. **不能把现有行为当正确行为**：期望值必须来自业务规则、金融公式、A 股交易规则、产品需求或可手算数据；也不能从现有测试断言反推出期望值。
5. **不清晰就记录待澄清**：业务语义、阈值、枚举、空值含义、权限边界不明确时，写入文档“待澄清”，不要擅自拍脑袋改业务。
6. **模块循环执行**：如果范围包含多个模块，按模块逐个跑完整闭环；不要把多个模块混成一份大报告。
7. **Bug 修复阶段才读实现**：测试设计和首轮执行完成后，进入后端修复模式，此时再阅读实现、定位根因、做最小改动。
8. **修完必须回归**：每个 Bug 修复后必须重跑失败用例、相关影响用例和必要的构建/集成验证。
9. **文档先行**：新增或重大改造测试前，先产出测试用例文档；新增 `docs/` 文档后同步更新 `docs/README.md`。
10. **遵守本仓库约定**：Controller 仅 POST、测试放模块 `test/` 子目录、代码改动同步维护测试、真实验证优先。

## 工作流总览

对每个模块独立执行以下循环：

1. **界定范围**
   - 明确模块、接口路径、用户角色、调用入口、依赖数据、外部系统。
   - 明确本轮是“全新测试设计”，不继承现有 spec 的覆盖结论；如果需要评估已有自动化，只能在测试方案完成后进行映射。
   - 多模块任务先建立模块队列，按业务风险排序：资金/安全/核心计算 > 数据正确性 > 高频页面 > 辅助功能。

2. **收集契约，不读实现**
   - 优先读取：需求/设计文档、当前联调问题、`swagger.json`、前端期望类型、Controller 路由、DTO、公开接口说明。
   - 仅把 Controller/DTO 用作“接口地图”，不要用 service 代码推导正确行为。
   - 不读取现有 `*.spec.ts` 来决定该测什么、期望是什么或哪些场景“已覆盖”。如需了解测试辅助工具，只能在用例文档定稿后读取 helper/fixture 或测试基础设施。
   - Tushare 或外部数据接口相关测试，按仓库规则同时核对官方文档和真实返回字段。

3. **建立业务模型**
   - 写清：用户为什么调用、成功路径、状态变化、核心业务不变量、金融公式、权限边界、时间/交易日规则、缓存/异步/幂等要求。
   - 识别歧义并记录“待澄清”，必要时用选择题向用户确认。

4. **设计测试用例文档**
   - 使用 [测试用例文档模板](./assets/测试用例文档模板.md)。
   - 用例必须覆盖：`BIZ` 常规业务、`EDGE` 边界、`ERR` 错误处理、`SEC` 安全、`DATA` 数据一致性、`RACE` 并发/竞态、`PERF` 性能、`LOAD` 负载、`STRESS` 压力、`E2E` 端到端、`REG` 回归。
   - 每个用例写明：业务依据、输入数据、步骤、独立推导的期望结果、自动化方式、优先级、是否阻塞。
   - 详细设计方法见 [测试用例设计指南](./references/test-case-design-guide.md)。

5. **实现或准备测试**
   - 单元测试：验证纯业务规则、公式、mapper、状态机；期望值用手算/独立公式。
   - Controller/集成测试：验证 DTO 校验、Guard、异常映射、响应包装、权限。
   - E2E 测试：验证跨模块业务链路和真实应用行为。
   - 性能/压力测试：优先设计可重复的基准数据、并发模型、指标阈值；避免把不稳定外部 API 当作压测对象。
   - 用例文档完成后，才允许查看现有 spec 以决定“新增文件、替换弱用例、保留可复用 helper、避免重复命名”；即便如此，业务期望仍以新文档为准。

6. **执行验证并记录结果**
   - 先跑目标模块最小测试，再逐步扩大到相关测试、E2E、构建。
   - 常用命令：`pnpm exec jest <spec> --runInBand`、`pnpm test`、`pnpm test:e2e`、`pnpm build`。
   - 性能结果必须记录环境、数据规模、并发、耗时分位数、吞吐、错误率；禁止伪造指标。
   - 执行和回归细节见 [执行与回归指南](./references/execution-and-regression-guide.md)。

7. **缺陷归纳**
   - 每个 Bug 记录：ID、模块、严重度、优先级、业务影响、复现步骤、期望/实际、证据、疑似根因、是否需求不清。
   - 需求不清的项目进入“待澄清”，不直接改业务行为。

8. **后端修复模式**
   - 现在才阅读实现代码定位根因。
   - 做最小、安全、可回退的修复；保持现有公共 API，除非测试文档明确说明契约需调整。
   - 若改 Prisma/Docker/Redis/外部同步流程，按仓库规则运行迁移/重建/日志检查。

9. **回归循环**
   - 重跑失败用例。
   - 补充防回归测试：让同类 Bug 以后自动暴露。
   - 跑影响面测试和 `pnpm build`。
   - 若仍失败，回到缺陷归纳与后端修复，直到修复或明确阻塞。

10. **本轮总结**
    - 使用 [测试执行报告模板](./assets/测试执行报告模板.md)。
    - 汇总：范围、业务理解、执行结果、Bug 状态、修复内容、回归结论、风险、待澄清、下轮建议。

## 多模块调度规则

- 先列模块队列和风险排序，再逐模块执行完整闭环。
- 每个模块都必须有独立测试用例表、缺陷表、回归结论。
- 跨模块链路只在单模块闭环后进入 E2E 阶段，避免根因混淆。
- 如果时间不足，优先完成高风险模块的完整闭环，而不是所有模块都只做半截。

## 输出物约定

- 测试用例文档：建议放在 `docs/design/<模块名>测试方案-YYYYMMDD.md`，文件名中文。
- 测试执行报告：建议放在 `docs/<模块名>测试执行报告-YYYYMMDD.md` 或合并到用户指定文档。
- 自动化测试：放在对应模块的 `test/` 子目录；全局辅助工具放 `test/helpers/`。
- 新增 `docs/` 文档后必须同步更新 `docs/README.md`。

## 完成标准

一次模块测试任务完成时，必须能回答：

- 该接口服务什么业务场景？核心业务规则是什么？
- 哪些用例覆盖了常规、边界、异常、安全、数据、并发、性能和压力？
- 发现了哪些 Bug？哪些是需求不清？证据是什么？
- 修了什么？为什么这样修？有没有破坏公共契约？
- 回归跑了哪些测试？结果如何？还有什么风险？
