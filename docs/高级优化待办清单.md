## 一、现状总评

### 1.1 已形成闭环的链路

| 链路 | 涉及模块 | 完整度 |
|------|----------|--------|
| 因子研究 → 因子选股 → 回测验证 | Factor → Backtest | ✅ 完整 |
| 策略模板 → 参数校验 → 回测执行 | Strategy → Backtest | ✅ 完整 |
| 策略草稿 → 提交回测 | StrategyDraft → Strategy → Backtest | ✅ 完整 |
| Walk-Forward 滚动验证 → 过拟合检测 | Backtest (WF) | ✅ 完整 |
| 蒙特卡洛模拟 → 鲁棒性评估 | Backtest (MC) | ✅ 完整 |
| 多策略对比 → 择优 | Backtest (Comparison) | ✅ 完整 |
| 选股器 → 条件订阅 → WebSocket 推送 | Stock Screener → ScreenerSubscription | ✅ 完整 |
| Tushare 同步 → 数据质量校验 → 自动补数 | Tushare Sync → Quality | ✅ 完整 |

### 1.2 存在断裂的链路

| 链路 | 断裂点 | 影响 |
|------|--------|------|
| 回测 → 组合持仓 | 回测结果无法自动映射为组合持仓 | 用户需手动逐条录入持仓 |
| 组合 → 风控告警 | 风控检测不触发告警推送 | 超限后用户无感知 |
| 因子 → 策略模板自动生成 | 因子分析结论无法自动构建策略 | 研究成果与策略脱节 |
| 报告 → 多模块联动 | 报告无法聚合因子+回测+组合数据 | 报告维度单一 |
| 告警 → 自选股/组合 | 价格预警与自选股/组合完全隔离 | 关注的标的需重复配置 |

### 1.3 从量化从业者角度看核心缺失

作为一名日常进行因子挖掘、策略开发、回测验证、组合管理、风险监控的量化从业者，以下能力是当前平台**最亟需补齐**的：

1. **「研究→实盘」最后一公里**：回测跑完后，目前只能看报告，无法一键将目标持仓导入组合、生成调仓清单。
2. **归因分析**：回测有收益曲线但无 Brinson 归因、因子归因，无法回答"为什么赚 / 亏"。
3. **组合级风险实时监控**：Portfolio 模块有风控规则，但规则触发后不推送告警，等于纸上谈兵。
4. **多周期回测**：当前仅支持日频，缺少周频/月频再平衡场景下的撮合精度。
5. **交易成本敏感性分析**：佣金率、滑点对策略表现的影响无法量化。

## 二、工具链断裂分析

### 2.1 理想工具链全景

```
Tushare 数据同步 ──→ 因子研究 ──→ 策略构建 ──→ 回测验证 ──→ 归因分析
                                                              │
                          信号生成 ──→ 组合调仓 ──→ 风控监控 ──→ 报告归档
                              ↑              │            │
                         （盘后定时）    自选股关联     告警推送
```

### 2.2 当前断裂点详解

#### 断裂 A：回测结果 → 组合持仓（最关键）

**现状**：`BacktestModule` 和 `PortfolioModule` 完全隔离，无任何 import 依赖。

```
BacktestRun (status=COMPLETED)
  ├── BacktestPositionSnapshot（末日持仓）
  └── ❌ 无路径 → Portfolio.holdings
```

**影响**：量化从业者最核心的操作是"回测通过 → 部署策略 → 管理持仓"。当前流程断裂意味着用户需手动将回测末日持仓逐条录入组合模块，容易出错且低效。

**修复方案**：新增 `POST /portfolio/apply-backtest` 端点，从 BacktestPositionSnapshot 读取末日持仓，映射为 Portfolio Holdings。

#### 断裂 B：风控规则触发 → 告警推送

**现状**：`RiskCheckService` 检测违规并写入 `RiskViolation` 表，但不触发 WebSocket 推送。

```
RiskCheckService.checkRisk()
  ├── 检测集中度/行业/回撤违规
  ├── 写入 RiskViolation 表
  └── ❌ 无 EventsGateway.emitToUser() 调用
```

**影响**：用户配置了止损线、仓位上限等风控规则，但违规时毫无通知，需要手动查询。

**修复方案**：`RiskCheckService` 注入 `EventsGateway`，违规时推送 `risk_violation` 事件。

#### 断裂 C：因子研究 → 策略模板自动构建

**现状**：因子筛选可以直接提交回测（`FactorBacktestService`），但无法自动保存为策略模板。

```
FactorScreening 结果
  ├── POST /factor/backtest/submit → 一次性回测
  └── ❌ 无路径 → Strategy 模板（可复用）
```

**影响**：研究员发现有效因子组合后，每次回测都要重新配置参数，无法复用。

**修复方案**：新增 `POST /factor/backtest/save-as-strategy` 端点，将因子筛选条件保存为 `FACTOR_SCREENING_ROTATION` 类型的 Strategy 模板。

#### 断裂 D：告警系统 → 自选股/组合

**现状**：`AlertModule` 仅导入 `WebsocketModule`，与 `WatchlistModule`、`PortfolioModule` 无关联。

**影响**：用户自选股中的标的无法自动应用预警规则；组合持仓股票的异动检测需要单独配置。

**修复方案**：新增"关联告警"能力——创建价格预警时可关联自选股组或组合 ID，系统自动对成员股应用规则。

#### 断裂 E：报告 → 多维聚合

**现状**：`ReportDataCollectorService` 按类型单独采集（backtest / stock / portfolio），无法生成跨模块综合报告。

**影响**：无法生成"某策略的因子分析 + 回测表现 + 归因 + 风险分析"的综合研究报告。

**修复方案**：新增 `STRATEGY_RESEARCH` 报告类型，串联因子分析 + 回测 + 风险评估数据。

## 三、功能缺口矩阵

从量化从业者日常工作流角度，按"研究 → 策略 → 回测 → 交易 → 监控 → 复盘"六阶段评估：

| 阶段 | 子能力 | 当前状态 | 缺口说明 |
|------|--------|----------|----------|
| **研究** | 因子 IC/分位/衰减分析 | ✅ | — |
| | 因子正交化 & Fama-MacBeth | ✅ | — |
| | 自定义表达式因子 | ✅ | — |
| | **因子收益归因（Brinson）** | ❌ 缺失 | 无法拆解策略收益来源 |
| | **多因子组合优化（MVO/风险平价）** | ❌ 缺失 | 无法基于因子暴露优化权重 |
| | 行业轮动分析 | ✅ | — |
| | 事件驱动研究 | ✅ | — |
| **策略** | 策略模板 CRUD & JSON Schema | ✅ | — |
| | **因子研究 → 策略自动构建** | ❌ 断裂 | 研究成果需手动转为策略 |
| | **策略版本对比** | ⚠️ 有版本号但无对比接口 | 策略参数迭代无法可视化 |
| **回测** | 日频撮合引擎 | ✅ | — |
| | Walk-Forward 验证 | ✅ | — |
| | 蒙特卡洛模拟 | ✅ | — |
| | 多策略对比 | ✅ | — |
| | **Brinson 归因分析** | ❌ 缺失 | 回测结果只有指标，无归因 |
| | **交易成本敏感性分析** | ❌ 缺失 | 无法评估费率对策略的影响 |
| | **参数敏感性分析（热力图）** | ❌ 缺失 | 无法批量扫描参数空间 |
| **交易** | 回测持仓 → 组合导入 | ❌ 断裂 | 最后一公里断裂 |
| | **调仓清单生成** | ❌ 缺失 | 无法生成"买入/卖出/持有"清单 |
| | **盘后信号生成** | ❌ 缺失 | 无法在收盘数据上自动生成次日信号 |
| **监控** | 组合风险分析 | ✅ | — |
| | 风控规则引擎 | ✅ | — |
| | **风控 → 告警推送** | ❌ 断裂 | 规则触发无通知 |
| | **组合绩效跟踪** | ⚠️ 有P&L但无基准对比 | 缺少相对收益跟踪 |
| | **策略漂移检测** | ❌ 缺失 | 无法检测策略实际表现偏离回测预期 |
| **复盘** | 回测报告 / 个股报告 | ✅ | — |
| | **综合研究报告** | ❌ 缺失 | 无法聚合多模块数据 |
| | **交易日志复盘** | ❌ 缺失 | 无法回顾历史操作决策 |

## 四、优化项清单

### Phase 1 — 串联核心工具链（最高优先级）

> 目标：打通"因子研究 → 策略 → 回测 → 组合 → 监控"全链路。

#### OPT-1.1 回测结果导入组合（回测 → 组合桥梁）

- **端点**：`POST /portfolio/apply-backtest`
- **逻辑**：
  1. 读取指定 `backtestRunId` 的 `BacktestPositionSnapshot`（末日持仓）
  2. 创建或更新目标 Portfolio 的 Holdings
  3. 计算初始成本、权重分配
  4. 返回调仓清单（新增/加仓/减仓/清仓）
- **依赖**：`PortfolioModule` 需 import `BacktestModule`（获取 `BacktestRunService`）
- **文件变更**：
  - `src/apps/portfolio/portfolio.module.ts` — 添加 BacktestModule import
  - 新增 `src/apps/portfolio/services/backtest-portfolio-bridge.service.ts`
  - `src/apps/portfolio/portfolio.controller.ts` — 新增端点
  - 新增 DTO + Response DTO + 测试

#### OPT-1.2 调仓清单生成

- **端点**：`POST /portfolio/rebalance-plan`
- **逻辑**：
  1. 接受目标权重配置（来自回测末日持仓 / 因子筛选 / 手动输入）
  2. 与当前组合持仓对比，生成差异：`BUY / SELL / HOLD / ADJUST`
  3. 考虑最小交易单位（100 股整手）、停牌股跳过
  4. 估算交易成本
- **文件变更**：
  - 新增 `src/apps/portfolio/services/rebalance-plan.service.ts`
  - 新增 DTO + 测试

#### OPT-1.3 风控规则触发 → 告警推送

- **逻辑**：`RiskCheckService` 检测到违规后，调用 `EventsGateway.emitToUser()` 推送 `risk_violation` 事件
- **文件变更**：
  - `src/apps/portfolio/portfolio.module.ts` — 添加 WebsocketModule import
  - `src/apps/portfolio/risk-check.service.ts` — 注入 EventsGateway，违规时推送
  - 更新测试

#### OPT-1.4 因子筛选 → 保存为策略模板

- **端点**：`POST /factor/backtest/save-as-strategy`
- **逻辑**：
  1. 接受因子筛选条件（与 `submitBacktest` 相同的 DTO）
  2. 自动构建 `FACTOR_SCREENING_ROTATION` 类型的 Strategy 记录
  3. 填入默认回测参数
  4. 返回 Strategy ID
- **依赖**：`FactorModule` 需额外 import `StrategyModule`（或通过 Prisma 直接写入）
- **文件变更**：
  - `src/apps/factor/services/factor-backtest.service.ts` — 新增方法
  - `src/apps/factor/factor.controller.ts` — 新增端点
  - 新增 DTO + 测试

### Phase 2 — 归因与分析增强

> 目标：让量化研究者能回答"为什么赚/亏"，进行参数空间探索。

#### OPT-2.1 Brinson 归因分析

- **端点**：`POST /backtests/runs/attribution`
- **逻辑**：
  1. 从 BacktestDailyNav + BacktestPositionSnapshot 获取每日持仓权重
  2. 获取行业分类（申万一级，来自已同步的 `IndexClassify`）
  3. 计算 Brinson 三因素分解：资产配置效应 / 个股选择效应 / 交互效应
  4. 按行业、按时间段输出归因明细
- **数据依赖**：`IndexClassify`（申万行业分类，已由 Tushare 同步）
- **文件变更**：
  - 新增 `src/apps/backtest/services/backtest-attribution.service.ts`
  - `src/apps/backtest/backtest.controller.ts` — 新增端点
  - 新增 DTO + 测试

#### OPT-2.2 交易成本敏感性分析

- **端点**：`POST /backtests/runs/cost-sensitivity`
- **逻辑**：
  1. 基于已完成的回测，批量模拟不同佣金率 / 滑点参数
  2. 不需要重跑引擎——基于已有 TradeRecord 重新计算费用
  3. 输出：参数 → 净收益 / 夏普比率映射表
- **文件变更**：
  - 新增 `src/apps/backtest/services/backtest-cost-sensitivity.service.ts`
  - 新增端点 + DTO + 测试

#### OPT-2.3 参数敏感性分析（参数扫描热力图）

- **端点**：`POST /backtests/runs/param-sensitivity`
- **逻辑**：
  1. 接受两个参数的扫描范围（如 shortWindow: 5-20, longWindow: 20-60）
  2. 批量创建回测任务（BullMQ 批量入队）
  3. 汇总结果为二维热力图数据（X=参数1，Y=参数2，Value=Sharpe/MaxDD/Return）
- **文件变更**：
  - 新增 `src/apps/backtest/services/backtest-param-sensitivity.service.ts`
  - 新增端点 + DTO + 测试

#### OPT-2.4 策略版本对比

- **端点**：`POST /strategies/compare-versions`
- **逻辑**：
  1. 查询同一策略名不同版本的配置差异（JSON diff）
  2. 若版本关联了 BacktestRun，则拉取回测指标对比
  3. 输出：版本变更明细 + 绩效对比表
- **文件变更**：
  - `src/apps/strategy/strategy.service.ts` — 新增方法
  - 新增端点 + DTO + 测试

### Phase 3 — 盘后信号与组合跟踪

> 目标：支持盘后信号生成，缩小回测与实战的差距。

#### OPT-3.1 盘后信号生成引擎

- **描述**：策略挂载 Schedule，在 Tushare 每日同步完成后（约 18:30+）自动运行，生成次日买卖信号
- **端点**：
  - `POST /signal/strategies/activate` — 激活策略的每日信号生成
  - `POST /signal/strategies/deactivate` — 停用
  - `POST /signal/strategies/list` — 已激活策略列表
  - `POST /signal/latest` — 查询最新信号
- **逻辑**：
  1. 新增 `SignalModule`，定时任务监听 Tushare 同步完成事件后触发
  2. 加载已激活策略的配置，在最新截面数据（日线 + 因子）上生成信号
  3. 信号入库 + WebSocket 推送给关联用户
  4. 可关联 Portfolio，生成调仓建议
- **数据模型**：
  - 新增 `TradingSignal` Prisma Model（strategyId, tradeDate, tsCode, action, confidence, targetWeight）
- **文件变更**：
  - 新增 `src/apps/signal/` 模块（signal.module.ts / signal.controller.ts / signal.service.ts / signal-activation.service.ts）
  - 新增 `prisma/trading_signal.prisma`

#### OPT-3.2 组合绩效跟踪（vs 基准）

- **端点**：`POST /portfolio/performance`
- **逻辑**：
  1. 计算组合自创建以来的累计收益曲线（基于每日持仓 × 日线收盘价）
  2. 对比基准指数（默认沪深300，来自已同步的 `IndexDaily`）
  3. 输出：超额收益、跟踪误差、信息比率
  4. 如组合关联了策略回测，则叠加显示回测预期 vs 实际表现
- **文件变更**：
  - 新增 `src/apps/portfolio/services/portfolio-performance.service.ts`
  - 新增端点 + DTO + 测试

#### OPT-3.3 策略漂移检测

- **端点**：`POST /portfolio/drift-detection`
- **逻辑**：
  1. 对比组合当前持仓与关联策略最新信号的差异
  2. 计算偏离度：权重偏离 / 持仓偏离 / 行业暴露偏离
  3. 超过阈值时标记告警并推送 WebSocket
- **文件变更**：
  - 新增 `src/apps/portfolio/services/drift-detection.service.ts`
  - 新增端点 + DTO + 测试

### Phase 4 — 高级研究工具

> 目标：提供专业量化研究者日常使用的高级分析能力。

#### OPT-4.1 多因子组合优化（MVO / 风险平价）

- **端点**：`POST /factor/optimization`
- **逻辑**：
  1. 基于因子暴露矩阵 + 协方差矩阵（由已同步的日线收益率序列估计）
  2. 实现均值方差优化（MVO）、最小方差、风险平价、最大多样化四种模式
  3. 输出最优权重向量
  4. 可直接生成 `CUSTOM_POOL_REBALANCE` 类型策略配置
- **文件变更**：
  - 新增 `src/apps/factor/services/factor-optimization.service.ts`
  - 新增端点 + DTO + 测试

#### OPT-4.2 综合研究报告

- **报告类型**：`STRATEGY_RESEARCH`
- **逻辑**：
  1. 聚合：因子分析（IC / 分位 / 衰减） + 回测结果 + Brinson 归因 + 风险评估
  2. 生成结构化 JSON → 渲染为 PDF/HTML（Handlebars 模板 + Puppeteer，已有基础设施）
  3. 支持模板自定义（段落开关控制输出章节）
- **文件变更**：
  - `src/apps/report/report-data-collector.service.ts` — 新增 `collectStrategyResearchData()` 方法
  - 新增报告模板文件

#### OPT-4.3 告警系统增强（关联自选股 / 组合）

- **逻辑**：
  1. 创建价格预警时可关联 `watchlistId` 或 `portfolioId`
  2. 系统自动为关联组的成员股应用规则，成员变动时实时同步
  3. 告警触发时消息体中注明来源（自选股组名 / 组合名）
- **Prisma 变更**：
  - `PriceAlert` Model 新增 `watchlistId?` / `portfolioId?` 可选字段
- **文件变更**：
  - `src/apps/alert/price-alert.service.ts` — 注入 WatchlistService + PortfolioService
  - 更新相关 DTO + 测试

#### OPT-4.4 交易日志复盘

- **端点**：`POST /portfolio/trade-log`
- **逻辑**：
  1. 记录组合每次调仓操作（添加/更新/移除持仓）的完整日志
  2. 关联调仓原因（手动 / 回测导入 / 策略信号 / 风控止损）
  3. 支持按时间范围、操作类型、标的筛选
- **Prisma 变更**：
