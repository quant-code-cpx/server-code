## 一、功能定位与目标

### 1.1 解决的问题

当前 `StockAnalysisService.getTechnicalIndicators()` 基于 OHLCV 原始数据 **实时计算** 25+ 技术指标。这种方式有两个瓶颈：

| 场景 | 问题 |
|------|------|
| **单股查询** | 每次请求需取 days+300 条原始 K 线 + CPU 密集型计算，响应延迟较高 |
| **全市场技术筛选** | 如果选股器要按 MACD 金叉、RSI 超卖等技术条件筛选 5000+ 只股票，实时计算不可接受 |

### 1.2 目标

1. 接入 Tushare `stk_factor` 接口，每日盘后同步全市场技术指标预计算数据入库
2. 提供轻量级技术指标查询 API（基于预计算数据，无需实时计算）
3. 为选股器扩展技术条件筛选提供数据基础（SQL WHERE 即可完成技术面筛选）
4. 保留现有自计算指标体系作为多周期 / 长序列 / 高级指标的补充

### 3.3 同步策略建议

| 策略 | 说明 |
|------|------|
| 请求方式 | 按 `trade_date` 查询（单日全市场），每个交易日一次调用返回全市场数据 |
| 增量同步 | 检测本地最新交易日，仅同步缺失日期 |
| 全量回补 | 支持从 `TUSHARE_DEFAULT_SYNC_START_DATE`（2010-01-01）开始回补，按交易日逐日拉取 |
| 频控处理 | 复用 `SyncHelperService.syncByTradeDate` 的节流机制；如接口有每分钟限制，在循环中加入适当间隔 |

## 四、Prisma Schema 设计

新增文件：`prisma/tushare_stk_factor.prisma`

```prisma
/// Tushare stk_factor — 股票技术因子（每日技术指标预计算数据）
model StkFactor {
  tsCode   String   @map("ts_code")    @db.VarChar(16)
  tradeDate DateTime @map("trade_date") @db.Date

  // ── 行情基础 ──
  close    Float?
  open     Float?
  high     Float?
  low      Float?
  preClose Float?   @map("pre_close")
  change   Float?
  pctChg   Float?   @map("pct_chg")
  vol      Float?
  amount   Float?

  // ── MACD ──
  macdDif  Float?   @map("macd_dif")
  macdDea  Float?   @map("macd_dea")
  macd     Float?

  // ── KDJ ──
  kdjK     Float?   @map("kdj_k")
  kdjD     Float?   @map("kdj_d")
  kdjJ     Float?   @map("kdj_j")

  // ── RSI ──
  rsi6     Float?   @map("rsi_6")
  rsi12    Float?   @map("rsi_12")
  rsi24    Float?   @map("rsi_24")

  // ── 布林带 ──
  bollUpper Float?  @map("boll_upper")
  bollMid   Float?  @map("boll_mid")
  bollLower Float?  @map("boll_lower")

  // ── CCI ──
  cci14    Float?   @map("cci_14")
  cci20    Float?   @map("cci_20")

  // ── ATR / 波动率 ──
  tr       Float?
  atr14    Float?
  atr20    Float?

  // ── VR ──
  vr26     Float?   @map("vr_26")

  @@id([tsCode, tradeDate])
  @@index([tradeDate])
  @@index([tsCode, tradeDate(sort: Desc)])
  @@map("stock_technical_factors")
}
```

### 索引设计说明

| 索引 | 用途 |
|------|------|
| `@@id([tsCode, tradeDate])` | 主键，保证唯一性，支持按股票+日期精确查询 |
| `@@index([tradeDate])` | 按交易日查询全市场数据（同步判重、全市场筛选） |
| `@@index([tsCode, tradeDate(sort: Desc)])` | 按股票查询最近 N 日指标（倒序，详情页场景） |

### 8.2 同步执行方法

在 `FactorDataSyncService` 中新增 `syncStkFactor` 方法。复用已有的 `syncByTradeDateString` 私有方法模式：

```typescript
async syncStkFactor(
  targetTradeDate: string,
  mode: TushareSyncMode = 'incremental',
  onProgress?: (completed: number, total: number, currentKey?: string) => void,
): Promise<void> {
  await this.syncByTradeDateString({
    task: TushareSyncTaskName.STK_FACTOR,
    label: '技术因子',
    modelName: 'stkFactor',
    targetTradeDate,
    fullSync: mode === 'full',
    fetchAndMap: async (td) => {
      const rows = await this.api.getStkFactorByTradeDate(td)
      return rows
        .map(mapStkFactorRecord)
        .filter((r): r is NonNullable<typeof r> => Boolean(r))
    },
    resolveDates: (start) =>
      this.helper.getOpenTradeDatesBetween(start, targetTradeDate),
    onProgress,
  })
}
```

> **注意**：`syncByTradeDateString` 签名需扩展以支持 `onProgress` 回调。参考 `SyncHelperService.syncByTradeDate` 中已有的进度回调机制，在每完成一个交易日后调用 `onProgress(i + 1, tradeDates.length, td)`。

### 8.3 数据量估算

| 维度 | 估算值 |
|------|--------|
| 上市股票数 | ~5300 只 |
| 每日记录数 | ~5300 条 |
| 每年交易日 | ~244 天 |
| 全量回补（2010 至今） | ~244 × 16 × 5300 ≈ **2070 万条** |
| 单条记录字段数 | 29 个 |
| 增量同步（每日） | ~5300 条，1 次 API 调用 |

### 8.4 全量回补频控策略

全量回补需逐日调用，约 3900+ 个交易日。频控建议：

1. 复用 `TushareClient` 现有的按 API 独立节流通道
2. 如果 `stk_factor` 频率限制为每分钟 N 次，在 `syncByTradeDateString` 循环中确保节流
3. 利用已有的断点续传机制（`SyncHelperService.getResumeKey/updateProgress`）支持中断恢复
4. 全量回补优先在非交易时段执行，避免影响增量同步

```typescript
async getTechnicalFactors(dto: StockTechnicalFactorsQueryDto): Promise<StockTechnicalFactorsDataDto> {
  const { tsCode, days = 120, indicators } = dto

  // 构建 select：如果传了 indicators，只查指定字段 + tradeDate + close
  const selectFields = this.buildSelectFields(indicators)

  const rows = await this.prisma.stkFactor.findMany({
    where: { tsCode },
    select: selectFields,
    orderBy: { tradeDate: 'desc' },
    take: days,

    ...row,
  }))

  return { tsCode, count: items.length, items }
}
```

```typescript
  rsiSignal: 'overbought' | 'oversold' | 'neutral' | null

  // 布林位置
  @ApiProperty({ required: false, nullable: true })
  bollPosition: 'above_upper' | 'near_upper' | 'middle' | 'near_lower' | 'below_lower' | null

  // 原始值
  @ApiProperty({ type: StkFactorDataPointDto, required: false, nullable: true })

}
```

#### Service 实现要点

查询最近 2 条记录（当日 + 前一日），用于判断金叉/死叉等需要前后对比的信号：

```typescript
async getLatestFactors(dto: StockLatestFactorsQueryDto): Promise<StockLatestFactorsDataDto> {
  const rows = await this.prisma.stkFactor.findMany({
    where: { tsCode: dto.tsCode },
    orderBy: { tradeDate: 'desc' },
    take: 2,
  })

  if (rows.length === 0) {
    return { tsCode: dto.tsCode, tradeDate: null, close: null, macdSignal: null, kdjSignal: null, rsiSignal: null, bollPosition: null, raw: null }
  }

  const latest = rows[0]
  const prev = rows[1] ?? null
  return {
    tsCode: dto.tsCode,
    tradeDate: dayjs(latest.tradeDate).format('YYYYMMDD'),
    close: latest.close,
    macdSignal: this.detectMacdSignal(latest, prev),
    kdjSignal: this.detectKdjSignal(latest, prev),
    rsiSignal: this.detectRsiSignal(latest),
    bollPosition: this.detectBollPosition(latest),
    raw: this.mapToDataPoint(latest),
  }
}
```

信号判断逻辑（与现有 `stock-analysis.service.ts` 中的 `buildSignalSummary` 保持一致）：

```typescript
private detectMacdSignal(latest: StkFactor, prev: StkFactor | null): string | null {
  if (latest.macdDif == null || latest.macdDea == null) return null
  if (prev?.macdDif != null && prev?.macdDea != null) {
    const prevDiff = prev.macdDif - prev.macdDea
    const currDiff = latest.macdDif - latest.macdDea
    if (prevDiff <= 0 && currDiff > 0) return 'golden_cross'
    if (prevDiff >= 0 && currDiff < 0) return 'death_cross'
  }
  return latest.macdDif > 0 ? 'above_zero' : 'below_zero'
}

private detectKdjSignal(latest: StkFactor, prev: StkFactor | null): string | null {
  if (latest.kdjK == null || latest.kdjD == null) return null
  if (latest.kdjJ != null && latest.kdjJ > 100) return 'overbought'
  if (latest.kdjJ != null && latest.kdjJ < 0) return 'oversold'
  if (prev?.kdjK != null && prev?.kdjD != null) {
    if (prev.kdjK <= prev.kdjD && latest.kdjK > latest.kdjD) return 'golden_cross'
    if (prev.kdjK >= prev.kdjD && latest.kdjK < latest.kdjD) return 'death_cross'
  }
  return null
}

private detectRsiSignal(latest: StkFactor): string | null {
  const rsi = latest.rsi6 ?? latest.rsi12
  if (rsi == null) return null
  if (rsi > 80) return 'overbought'
  if (rsi < 20) return 'oversold'
  return 'neutral'
}

private detectBollPosition(latest: StkFactor): string | null {
  if (latest.close == null || latest.bollUpper == null || latest.bollMid == null || latest.bollLower == null) return null
  if (latest.close > latest.bollUpper) return 'above_upper'
  if (latest.close > latest.bollUpper * 0.98) return 'near_upper'
  if (latest.close < latest.bollLower) return 'below_lower'
  if (latest.close < latest.bollLower * 1.02) return 'near_lower'
  return 'middle'
}
```

在 `StockScreenerService.screener()` 的 SQL 构建中，通过 LEFT JOIN `stock_technical_factors` 表的最新记录来应用技术条件过滤：

```sql
-- 在选股器 SQL 中新增 JOIN（仅当有技术条件时）
LEFT JOIN LATERAL (
  SELECT *
  FROM stock_technical_factors stf
  WHERE stf.ts_code = sb.ts_code
  ORDER BY stf.trade_date DESC
  LIMIT 1
) stf ON true
```

然后在 WHERE 子句中添加技术指标过滤条件。MACD 金叉/死叉判断需要最近 2 条记录对比，可通过子查询或 LAG 窗口函数实现：

```sql
-- 例：MACD 金叉（当日 DIF > DEA 且前日 DIF <= DEA）
WHERE stf.macd_dif > stf.macd_dea
  AND (
    SELECT sf2.macd_dif - sf2.macd_dea
    FROM stock_technical_factors sf2
    WHERE sf2.ts_code = sb.ts_code AND sf2.trade_date < stf.trade_date
    ORDER BY sf2.trade_date DESC LIMIT 1
  ) <= 0
```

## 十、与现有自计算指标的关系

### 10.1 双轨策略

| 场景 | 数据来源 | 原因 |
|------|----------|------|
| 股票详情 - 分析 Tab（完整指标序列） | **保留自计算**（`computeAllIndicators`） | 支持 25+ 指标、多周期（日/周/月）、自定义参数 |
| 股票详情 - 快速指标概览 | **预计算数据**（`stock_technical_factors`） | 直接数据库读取，毫秒级响应 |
| 选股器 - 技术面筛选 | **预计算数据**（`stock_technical_factors`） | SQL WHERE 过滤全市场，无法实时计算 |
| 择时信号综合 | **保留自计算** | 需要更长时间序列和更复杂的信号组合 |

### 10.2 不删除现有自计算逻辑

现有的 `stock-analysis.service.ts` + `technical-indicators.ts` 保持不变，原因：

1. 支持周线/月线周期（`stk_factor` 仅日线）
2. 支持前复权/后复权价格的技术指标
3. 计算 25+ 指标，远多于 `stk_factor` 提供的字段
4. 不依赖 Tushare 积分和频率限制
5. 如果 Tushare 数据延迟或异常，自计算可作为降级方案

### 10.3 Controller 路由分工

| 路由 | Handler | 数据来源 |
|------|---------|----------|
| `POST /stock/detail/analysis/technical` | `StockAnalysisService.getTechnicalIndicators` | 自计算（保持不变） |
| `POST /stock/detail/analysis/factors` | 新 Service（预计算查询） | `stock_technical_factors` 表 |
| `POST /stock/detail/analysis/factors/latest` | 新 Service（最新快照） | `stock_technical_factors` 表 |
