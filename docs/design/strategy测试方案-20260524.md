## 3. 业务理解

### 3.1 业务场景

- **策略管理**：用户创建策略模板，配置策略参数，管理策略生命周期。
- **策略执行**：基于策略配置发起回测，参数覆盖优先级：请求 > backtestDefaults > 系统默认。
- **版本管理**：修改 strategyConfig 时版本号自增，支持查看历史版本和对比差异。
- **策略克隆**：复制为新策略，支持克隆公开策略。
- **统计分析**：按类型分组统计策略数量，查询历史回测业绩。

### 3.2 核心业务规则

| 规则编号 | 规则 | 来源 | 测试影响 |
| -------- | ---- | ---- | -------- |
| R-S01 | 策略名称 1~100 字符 | DTO @MinLength(1) @MaxLength(100) | 空/超长应 400 |
| R-S02 | strategyType 必须是 BACKTEST_STRATEGY_TYPES 之一 | DTO @IsIn | 无效类型应 400 |
| R-S03 | strategyConfig 必须是对象 | DTO @IsObject | 非对象应 400 |
| R-S04 | 标签最多 10 个，每个不超过 30 字 | DTO @MaxLength(30) | 超限应 400 |
| R-S05 | run 的 startDate/endDate 格式 YYYYMMDD | DTO @Matches | 格式错误应 400 |
| R-S06 | run 的 initialCapital >= 1000 | DTO @Min(1000) | 少于应 400 |
| R-S07 | 所有端点需 JWT 认证 | @UseGuards(JwtAuthGuard) | 无 Token 应 401 |
| R-S08 | 用户只能操作自己的策略 | @CurrentUser() + service 层校验 | 越权应 403/404 |
| R-S09 | delete 有关联数据时需 force=true | Controller 逻辑 | 无 force 应拒绝 |
| R-S10 | update 修改 strategyConfig 时版本号自增 | Service 逻辑 | 版本号递增 |
| R-S11 | pageSize 1~100 | DTO @Min(1) @Max(100) | 超范围应 400 |
| R-S12 | compare-versions versionA < versionB | Service 逻辑 | versionA >= versionB 应拒绝 |

### 3.3 状态变化与不变量

| 类型 | 说明 |
| ---- | ---- |
| 数据写入 | strategy, strategyVersion |
| 缓存/队列/通知 | 无 |
| 权限边界 | 所有端点需 JWT，用户只能操作自己的策略 |
| 数据不变量 | 策略 ID 唯一、版本号递增 |
