## 3. 业务理解

### 3.1 业务场景

- 用户先获取验证码，再提交账号密码+验证码登录。
- 登录成功返回 access token，并由服务端设置 httpOnly refresh token cookie。
- access token 过期后使用 refresh token 续期。
- 登出时撤销 refresh token，清理 cookie，并使 access token 失效（黑名单）。

### 3.2 核心业务规则

| 规则编号 | 规则 | 来源 | 测试影响 |
| --- | --- | --- | --- |
| R-AUTH-001 | 验证码一次性且短时有效（60s） | `AuthController` 接口注释 | 需覆盖重复使用/过期失败 |
| R-AUTH-002 | 登录必须携带账号、密码、captchaId、captchaCode | `LoginDto` 契约 | 缺失字段/空值处理 |
| R-AUTH-003 | 登录成功后响应体仅返回 accessToken，refresh token 写 cookie | `POST /auth/login` 契约 | 验证响应体与 `set-cookie` 同时正确 |
| R-AUTH-004 | 刷新优先读取 cookie，无 cookie 时可回退 body.refreshToken | `POST /auth/refresh` 契约 | 需覆盖 cookie 优先级和回退路径 |
| R-AUTH-005 | 无 refresh token 刷新必须失败 | `AuthController.refresh` 显式校验 | 覆盖无 token 错误码与信息 |
| R-AUTH-006 | 登出应容忍 token 过期/无效并清 cookie | `AuthController.logout` 注释 | 覆盖幂等登出与弱网重放 |

### 3.3 状态变化与不变量

| 类型 | 说明 |
| --- | --- |
| 数据写入 | 登录/刷新写 refresh token 状态；登出删除/失效相关状态 |
| 缓存/队列/通知 | 验证码缓存短 TTL；token 黑名单缓存 |
| 权限边界 | `captcha/login/refresh/logout` 均可在无 access token 场景调用，但登录/刷新依赖有效凭证 |
| 时间规则 | 验证码 60 秒、token TTL 按系统配置 |
| 数据不变量 | 同一 refresh token 不可无限重放；登出后旧会话不可继续刷新 |

## 4. 接口契约

| 接口 | Body/Param | 响应数据 | 鉴权/角色 | 备注 |
| --- | --- | --- | --- | --- |
| `POST /auth/captcha` | 无 | `{ captchaId, svgImage }` | Public | 限流：10 req/60s |
| `POST /auth/login` | `LoginDto` | `{ accessToken }` + `Set-Cookie` | Public | 限流：5 req/60s |
| `POST /auth/refresh` | 可选 `refreshToken` | `{ accessToken }`，必要时轮换 cookie | Public | 限流：10 req/60s |
| `POST /auth/logout` | Header: `authorization` | `null`，清除 cookie | Public（可过期 token） | 需兼容无效 token |

## 6. 风险分析

| 风险 | 业务影响 | 测试策略 | 优先级 |
| --- | --- | --- | --- |
| 验证码可重放 | 账户暴力登录风险上升 | 验证一次性与 TTL | P0 |
| refresh token 被重放 | 会话劫持与非法续期 | 并发刷新+旧 token 失效校验 | P0 |
| cookie 属性错误 | 浏览器端会话失效或泄露风险 | 验证 httpOnly/sameSite/path | P0 |
| 登出不彻底 | 已登出用户仍可访问 | 登出后 refresh 失败回归 | P0 |
