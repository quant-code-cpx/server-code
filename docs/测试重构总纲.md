# 测试重构总纲

> **目标**：将现有 793 个测试用例从"验证代码能跑通"重构为"基于业务理解、以发现 Bug 为目的"的高质量测试体系。
> **现状诊断**：58 个 spec 文件中，约 60% 仅覆盖 happy path，Controller 测试几乎全部只验证 HTTP 状态码；Service 测试大量使用 `toHaveBeenCalled()` 验证方法调用，而非验证业务逻辑正确性。
> **原则**：每个测试用例必须回答一个业务问题——"如果这里出了 bug，用户会看到什么错误？"

---

## 一、现有测试体系的核心问题

### 1.1 问题总览

| 问题类型 | 影响范围 | 严重度 | 说明 |
|----------|----------|--------|------|
| **Mock 过度** | 全部 Controller spec（15 个） | 🔴 严重 | Service 完全 mock，只验证路由调用，等于测试 NestJS 框架本身而非业务 |
| **只测正确路径** | ~60% 的 spec 文件 | 🔴 严重 | 假设输入总是合法、依赖总是可用，无法发现边界条件 bug |
| **断言太弱** | ~40% 的 spec 文件 | 🟠 中等 | 大量 `toBeDefined()`、`toHaveBeenCalled()` 断言，不验证具体值 |
| **数值计算验证缺失** | 回测引擎、因子分析、组合绩效 | 🔴 严重 | 金融计算核心几乎不验证精度、边界值、数学正确性 |
| **安全逻辑未覆盖** | Auth、JWT Guard | 🔴 严重 | 账户锁定竞态、Token 刷新宽限期、账户枚举攻击 未测试 |
| **并发/竞态场景零覆盖** | 全部模块 | 🟠 中等 | 无任何并发测试 |
| **数据一致性未验证** | Tushare 同步、组合持仓 | 🟠 中等 | 同步数据变换、缓存一致性从未端到端验证 |
| **完全未测试模块** | event-study, pattern | 🔴 严重 | 两个功能模块零测试覆盖 |

## 二、重构设计原则

### 2.1 测试分类标准

每个测试用例必须属于以下类别之一：

| 类别 | 缩写 | 目的 | 示例 |
|------|------|------|------|
| **业务规则验证** | `BIZ` | 验证关键业务逻辑的正确性 | 账户连续失败 5 次后锁定 |
| **边界条件** | `EDGE` | 验证极端/临界输入下的行为 | 空持仓计算收益率、单笔交易的夏普比率 |
| **错误处理** | `ERR` | 验证异常情况下的系统行为 | 数据库连接失败、Redis 超时 |
| **安全防护** | `SEC` | 验证安全机制有效性 | SQL 注入防护、越权访问、Token 伪造 |
| **数据一致性** | `DATA` | 验证数据变换、缓存一致性 | Tushare 字段映射精度、缓存失效后的正确性 |
| **竞态条件** | `RACE` | 验证并发场景下的正确性 | 同时刷新 Token、同时下单 |

### 2.2 测试命名规范

```typescript
// 旧风格（描述实现）：
it('should call prisma.strategy.create', () => { ... })

// 新风格（描述业务场景）：
it('[BIZ] 用户策略数量达到上限 50 时，创建新策略应返回 BadRequest', () => { ... })
it('[EDGE] 所有交易都亏损时，胜率应为 0，盈亏比应为 NaN 或 Infinity', () => { ... })
it('[SEC] 账户被锁定后，使用正确密码仍应返回 Unauthorized', () => { ... })
it('[RACE] 两个并发请求使用同一 captchaId，只有第一个应成功', () => { ... })
```

### 2.3 断言强度要求

```typescript
// ❌ 弱断言（只验证存在性）
expect(result).toBeDefined();
expect(service.method).toHaveBeenCalled();

// ✅ 强断言（验证业务正确性）
expect(result.annualReturn).toBeCloseTo(0.1234, 4);
expect(result.holdings).toHaveLength(3);
expect(result.holdings[0].weight).toBeCloseTo(0.333, 3);
expect(mockPrisma.strategy.create).toHaveBeenCalledWith(
  expect.objectContaining({
    data: expect.objectContaining({ userId: 'user-1', name: '均线策略' }),
  }),
);
```

> 这些模块如果出 bug，直接影响资金安全、数据正确性、系统安全。

### 3.1 Auth 模块测试重构

**文件**：`src/apps/auth/test/auth.service.spec.ts`
**现状**：20 个用例，已覆盖基本登录/注册，但安全边界测试不足。

#### 需新增/重写的测试用例

| # | 类别 | 测试场景 | 预期行为 | 发现的潜在 Bug |
|---|------|---------|---------|---------------|
| 1 | `SEC` | 账户锁定后，即使输入正确密码也应拒绝 | 抛出 UnauthorizedException，包含"账户已锁定"提示 | 可能直接跳过锁定检查 |
| 2 | `SEC` | 不存在的账户登录失败时间与存在的账户一致 | 两种情况的响应时间差异 < 50ms | 账户枚举攻击：真实账户触发密码比较耗时更长 |
| 3 | `SEC` | 登录失败计数在账户锁定窗口过期后应重置 | 窗口过期后首次失败，计数从 1 重新开始 | 计数可能累积不重置 |
| 4 | `RACE` | 两个并发请求使用同一 captchaId | 仅第一个成功，第二个返回验证码错误 | Redis getDel 原子性测试 |
| 5 | `SEC` | Token 刷新宽限期内不检查用户状态 | 用户在宽限期内被禁用→宽限期内刷新仍成功（当前实现）或应失败 | 宽限期内不重查用户状态可能导致被禁用账户继续获取 Token |
| 6 | `BIZ` | 连续失败 N-1 次后成功登录，应重置计数 | 失败计数清零 | 计数未在成功登录时清除 |
| 7 | `SEC` | 使用已注销的 Access Token 尝试请求 | 返回 401 Unauthorized | 黑名单 TTL 设置不当导致 Token 仍可用 |
| 8 | `EDGE` | Captcha 恰好在验证时过期（TTL 边界） | 返回验证码已过期 | 时间窗口竞态 |
| 9 | `BIZ` | 验证码大小写不敏感比较 | `"AbCd"` 与 `"abcd"` 应匹配 | 大小写比较逻辑 |
| 10 | `SEC` | Refresh Token 被复用（replay attack） | 宽限期外重复使用应失败或触发告警 | Token 复用无检测 |

**文件**：`src/apps/auth/test/auth.controller.spec.ts`
**现状**：6 个用例，全部 happy path，实际不验证任何业务逻辑。

| # | 类别 | 测试场景 | 预期行为 | 发现的潜在 Bug |

| 1 | `ERR` | 空的 LoginDto（无用户名和密码） | 返回 400 + 字段校验错误信息 | 当前 `@Allow()` 装饰器无校验，空 body 也通过 |
