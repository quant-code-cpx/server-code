## 一、设计目标

首页仪表盘所需的后端数据**全部由现有 API 端点提供**，无需新增端点。本文档的目标是：

1. 明确首页使用了哪些后端端点，以便后续优化时有据可查
2. 记录每个端点的缓存策略和性能特征
3. 分析首页并发请求对后端的压力，提出优化方向
4. 说明数据新鲜度（Tushare EOD 数据）对首页展示的影响

## 三、端点详细说明

### 3.1 核心指数行情 — `POST /market/index-quote`

| 属性         | 说明                                             |
| ------------ | ------------------------------------------------ |
| **请求体**   | `{ trade_date?: string }` — 不传则取最新交易日   |
| **返回**     | `IndexQuoteItemDto[]` — 所有已同步指数的日线数据 |
| **数据源表** | `index_daily_prices`                             |
| **缓存**     | 无（单日 `findMany` 查询，毫秒级响应）           |
| **前端用途** | A 区指数卡片（最新点位、涨跌幅、涨跌额、成交额） |

**返回字段**：`tsCode`, `tradeDate`, `open`, `high`, `low`, `close`, `preClose`, `change`, `pctChg`, `vol`, `amount`

**注意**：返回所有已同步指数（上证、深证、创业板、沪深300、中证500、中证1000），前端按需筛选展示。

---

### 3.2 核心指数走势 — `POST /market/index-trend`

| 属性         | 说明                                                                  |
| ------------ | --------------------------------------------------------------------- |
| **请求体**   | `{ ts_code?: string, period?: '1m' \| '3m' \| '6m' \| '1y' \| '3y' }` |
| **返回**     | `IndexTrendResponseDto` — 包含 `tsCode`, `name`, `period`, `data[]`   |
| **数据源表** | `index_daily_prices`                                                  |
| **缓存**     | `market:index-trend:{tsCode}:{period}` — TTL 4 小时                   |
| **前端用途** | A 区指数卡片内的 sparkline 迷你走势图                                 |

**首页场景**：前端为 4 张指数卡片各发起一次 `index-trend` 请求（`period: '5d'` 或 `period: '1m'`），共 4 个并发请求，每个请求的缓存键不同。

---

### 3.3 市场情绪 — `POST /market/sentiment`

| 属性         | 说明                                               |
| ------------ | -------------------------------------------------- |
| **请求体**   | `{ trade_date?: string }`                          |
| **返回**     | `MarketSentimentDto` — 涨跌家数统计                |
| **数据源表** | `stock_daily_prices`                               |
| **缓存**     | 无（聚合查询，~100ms 级）                          |
| **前端用途** | B1 区情绪概览（上涨/下跌/平盘家数、涨停/跌停家数） |

**返回字段**：`tradeDate`, `totalCount`, `riseCount`, `fallCount`, `flatCount`, `limitUpCount`, `limitDownCount`, `riseRate`

### 3.4 涨跌幅分布 — `POST /market/change-distribution`

| 属性         | 说明                                                     |
| ------------ | -------------------------------------------------------- |
| **请求体**   | `{ trade_date?: string }`                                |
| **返回**     | `ChangeDistributionResponseDto` — 各涨跌幅区间的股票数量 |
| **数据源表** | `stock_daily_prices`                                     |

## 四、缓存策略

### 4.1 统一缓存架构

首页相关端点已全部纳入共享缓存层 `CacheService`，使用 `CACHE_NAMESPACE.MARKET` 命名空间。

| 配置项       | 值                            | 说明                                         |
| ------------ | ----------------------------- | -------------------------------------------- |
| 命名空间     | `market`                      | 通过 `CACHE_NAMESPACE.MARKET` 引用           |
| 标准 TTL     | 4 小时（14,400 秒）           | 大部分需聚合计算的端点                       |
| 扩展 TTL     | 8 小时（28,800 秒）           | 估值趋势（数据更新频率更低）                 |
| 失效机制     | Tushare 同步完成后自动失效    | 通过 `MONITORED_CACHE_NAMESPACES` 注册       |
| 前缀清理     | `market:` 前缀全量删除        | 通过 `SYNC_INVALIDATION_PREFIXES` 兜底       |
| 命名空间追踪 | 自动记录 key → namespace 映射 | 支持 `/tushare/admin/cache/stats` 查询命中率 |

### 4.2 缓存命中分析

首页典型加载场景（首次访问，缓存为空）：

| 请求                  | 是否走缓存 | 首次延迟  | 后续命中延迟 |
| --------------------- | ---------- | --------- | ------------ |
| `index-quote`         | ❌ 无缓存  | ~10ms     | —            |
| `index-trend` × 4     | ✅         | ~50-100ms | ~1ms         |
| `sentiment`           | ❌ 无缓存  | ~100ms    | —            |
| `change-distribution` | ✅         | ~200ms    | ~1ms         |
| `money-flow`          | ❌ 无缓存  | ~10ms     | —            |
| `hsgt-flow`           | ❌ 无缓存  | ~20ms     | —            |
| `sector-ranking`      | ✅         | ~100ms    | ~1ms         |
| `sector-flow-ranking` | ✅         | ~100ms    | ~1ms         |
| `main-flow-ranking`   | ✅         | ~150ms    | ~1ms         |
| `backtests/runs`      | ❌ 无缓存  | ~20ms     | —            |
| `tushare/admin/plans` | ❌ 无缓存  | ~10ms     | —            |

> 首次加载总延迟受最慢查询控制（前端并行请求）。典型最慢路径为 `change-distribution` 的全 A 聚合查询 ~200ms。第二次访问时有缓存的端点将全部 ~1ms 返回。

### 4.3 同步后缓存刷新

```
Tushare 日终同步完成
  → SyncService.invalidateCache()
    → CacheService.invalidateNamespaces(['market', ...])  // 删除所有 market: 命名空间下的 key
    → CacheService.invalidateByPrefixes(['market:', ...])  // 兜底清理前缀匹配的 key
  → 下一次首页请求自动触发缓存重建
```

## 五、性能分析与优化建议

### 5.1 当前性能基线

首页单次完整加载产生的后端请求：

- **首屏关键请求**：4 个（index-quote + sentiment + change-distribution + money-flow）
- **次屏延迟请求**：5-6 个（index-trend ×4 + hsgt-flow + sector-ranking + main-flow-ranking）
- **条件请求**：1-2 个（backtests/runs + admin/plans）
- **总计**：约 11 个 HTTP 请求

每个请求独立走 NestJS 管道（解析 → 校验 → Service → Prisma → 序列化），对于单用户场景完全没有压力。

### 5.2 未来优化方向（非必须，按需实施）

#### 方案 A：Dashboard 聚合端点

如果未来首页延迟成为瓶颈，可考虑新增一个聚合端点：

```typescript
// POST /api/dashboard/summary
// 一次请求返回首屏所有数据，减少 HTTP 往返
interface DashboardSummaryResponse {
  indexQuote: IndexQuoteItemDto[]
  sentiment: MarketSentimentDto
  changeDistribution: ChangeDistributionResponseDto
  moneyFlow: MarketMoneyFlowItemDto[]
}
```

**优势**：HTTP 往返从 4 → 1，减少 TCP 连接和 TLS 握手开销  
**代价**：引入一个聚合层，需维护字段同步  
**建议**：当前不必实施，等实际测量后再决定

#### 方案 B：ETL 预聚合表

对于 `change-distribution` 和 `sentiment` 这类每日固定的全 A 聚合计算，可以在每日同步后通过 ETL 任务将结果写入一张预聚合表，查询时直接读表，避免实时 `COUNT` / `GROUP BY`。

**建议**：当前缓存已能覆盖，优先级低

## 六、数据新鲜度与时序说明

### 6.1 数据更新时间线

```
15:00  A 股收盘
15:30  Tushare 开始更新当日数据（日线/资金流/板块等）
16:00  Tushare 大部分日线数据可用
17:00  定时同步任务执行（Cron: 0 17 * * 1-5）
17:30  同步完成 → 缓存自动失效
17:31  首页请求触发缓存重建（使用最新数据）
```

### 6.2 盘中访问首页

在 9:15–17:00 期间访问首页时：

- `index-quote` / `sentiment` / `money-flow` / `hsgt-flow` 返回的是**上一个交易日**的收盘数据
- 前端 60 秒自动刷新在此期间不会获得新数据（Tushare 为 EOD 数据源）
- 前端设计文档中的自动刷新机制已做好了此兼容（不会报错，只是返回相同数据）

### 6.3 前端建议（已在前端设计审视中提出）

- 在指数卡片上展示数据日期（如"2026-04-01 收盘"），避免用户误以为是实时数据
- 后续如接入实时行情源，可在不修改 API 结构的前提下增加数据源切换逻辑

## 七、权限控制

| 端点                       | 鉴权要求          | 说明                 |
| -------------------------- | ----------------- | -------------------- |
| 所有 `/market/*` 端点      | 无需鉴权          | 市场数据为公开数据   |
| `GET /backtests/runs`      | JWT 必须          | 自动按 `userId` 筛选 |
| `GET /tushare/admin/plans` | JWT + SUPER_ADMIN | 管理员专属           |

> **注意**：前端设计中 A-D 区对所有已登录用户可见，E 区仅管理员可见。后端权限控制与此一致。但 `/market/*` 端点本身不需要 JWT，前端在已登录状态下调用即可。

## 八、后续扩展方向

### 8.1 实时行情支持

当前所有数据为 EOD（日终），如需盘中实时数据：

| 方案             | 说明                                | 影响                            |
| ---------------- | ----------------------------------- | ------------------------------- |
| Tushare 实时接口 | `realtime_quote` 等接口，需要高积分 | 新增 WebSocket 推送通道         |
| 第三方行情源     | 如新浪/腾讯免费行情                 | 需新增数据适配层                |
| 混合模式         | 盘中用实时源，盘后切换日线数据      | 需在 Service 层加数据源选择逻辑 |

### 8.2 首页组件扩展

前端设计中预留了未来组件位置：

| 组件         | 依赖的后端 API                         | 当前状态                                 |
| ------------ | -------------------------------------- | ---------------------------------------- |
| 自选股动态   | `GET /watchlists/:id/items` + 实时行情 | ✅ API 已实现（自选股 CRUD），缺实时行情 |
| 策略预警     | 预警规则引擎 + WebSocket 推送          | 🔧 待实现                                |
| 因子异动监控 | 因子预计算 + 异常检测                  | 🔧 待实现                                |

### 8.3 监控与告警

- 缓存命中率可通过 `GET /tushare/admin/cache/stats` 监控
- 如首页多数请求 miss 缓存，可能意味着 TTL 设置过短或 sync 异常频繁触发失效
- 建议监控指标：首页渲染耗时（前端）、各端点 P99 延迟（后端）
