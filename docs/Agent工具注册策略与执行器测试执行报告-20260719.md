## 1. 本轮目标

- 验证模型只能调用构建期注册、显式启用且当前工作流授权的只读 Tool。
- 验证 input/output、来源、行数、字节、并发、deadline、重试和取消全部 fail-closed。
- 验证审计失败、并发冲突、迟到结果和 worker 恢复不会生成伪成功事实。

## 2. 业务理解摘要

| 项目     | 结论                                                                                      |
| -------- | ----------------------------------------------------------------------------------------- |
| 业务场景 | 后续 Orchestrator 把模型业务参数交给统一执行器，身份、scope、预算和 deadline 由服务端注入 |
| 核心规则 | 默认拒绝、固定 key/version、READ+idempotent 才可重试、先审计后 adapter、输出可信后才成功  |
| 关键风险 | 参数提权、跨 scope、错误并发合并、timeout 后迟到成功、输出夹带内部字段、配额绕过          |
| 待澄清   | 本批不注册真实金融 Tool；资源 owner 二次校验由 Batch 007–010 各领域 Facade 实现           |

## 11. 最终结论

- 改了什么：新增默认拒绝 Tool Registry、Policy、strict Schema、可信结果门禁、统一执行器、retry/timeout/cancel、限流、审计状态和 observer hook。
- 如何验证：21 个业务矩阵场景、76/76 自动化回归、production build、lint/format、契约检查、PostgreSQL 集成和容器运行全部通过。
- 当前状态：Batch 006 完成，无阻塞缺陷；15 个 canonical key 仍仅声明能力，不注册真实金融 Tool。
- 运行注意事项：空 allowlist 是安全默认；只支持 READ + idempotent；写 Tool、缓存、confirmation challenge 不在本批。
- 下一轮建议：优先 Batch 007 股票/市场/自选股查询 Tool；Batch 008/009 可在其后并行推进。
