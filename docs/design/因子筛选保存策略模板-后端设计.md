## 一、背景与目标

### 1.1 当前断裂点

因子研究员通过 `POST /factor/screening` 组合筛选条件后，只能通过 `POST /factor/backtest/submit` 发起一次性回测。每次回测都需重新配置相同的筛选条件和回测参数，无法将有效的因子组合保存为可复用的策略模板。

```
当前流程：
  因子筛选条件 → POST /factor/backtest/submit → 一次性回测任务
                                                  ❌ 无法保存为 Strategy 模板
                                                  ❌ 每次需重新配置
```

### 1.2 修复后流程

```
修复后：
  因子筛选条件 → POST /factor/backtest/save-as-strategy → Strategy 模板（可复用）
                                                              ↓
                                                  POST /strategy/run → 回测
                                                  POST /strategy/clone → 派生
                                                  POST /strategy/update → 迭代参数
```

### 1.3 核心收益

- 因子研究成果可固化为策略模板，闭合"研究 → 策略"链路
- 策略模板支持版本管理、标签分类、一键回测，与现有 Strategy 体系完全融合
- Phase 1 全部 4 项优化完成后，"因子研究 → 策略 → 回测 → 组合 → 监控"全链路串联

## 二、接口设计

### 2.1 端点

```
POST /factor/backtest/save-as-strategy
```

- **认证**：`JwtAuthGuard` + `@CurrentUser()`
- **Controller**：`FactorController`
- **Service**：`FactorBacktestService.saveAsStrategy()`

### 2.2 请求 DTO — `SaveAsStrategyDto`

复用 `FactorBacktestSubmitDto` 中的因子筛选参数，新增策略元数据字段：

```typescript
export class SaveAsStrategyDto {
  // ── 因子筛选条件（与 submitBacktest 一致）──────────────────

  /** 因子筛选条件列表 */
  conditions: FactorCondition[]

  /** 股票池 indexCode（如 000300.SH） */
  universe?: string

  /** 权重方式 */
  weightMethod?: 'equal_weight' | 'factor_weight' // 默认 'equal_weight'

  /** 排序因子（用于 topN 选取） */
  sortBy?: string

  /** 排序方向 */
  sortOrder?: 'asc' | 'desc' // 默认 'desc'

  /** 取前 N 只 */
  topN?: number // 默认 20

  // ── 回测默认参数（保存到 backtestDefaults）──────────────────

  /** 初始资金 */
  initialCapital?: number // 默认 1_000_000

  /** 调仓周期（交易日） */
  rebalanceDays?: number // 默认 5

  /** 手续费率 */
  commissionRate?: number // 默认 0.0003

  /** 滑点 bps */
  slippageBps?: number // 默认 5

  /** 基准指数 */
  benchmarkCode?: string // 默认 '000300.SH'

  // ── 策略元数据 ───────────────────────────────────────────

  /** 策略名称（必填） */
  name: string

  /** 策略描述 */
  description?: string

  /** 标签列表（最多 10 个） */
  tags?: string[]
}
```

### 2.3 响应 DTO — `SaveAsStrategyResponseDto`

```typescript
export class SaveAsStrategyResponseDto {
  /** 新建策略 ID */
  strategyId: string

  /** 策略名称 */
  name: string

  /** 策略类型（固定为 FACTOR_SCREENING_ROTATION） */
  strategyType: string

  /** 策略参数（因子筛选配置） */
  strategyConfig: Record<string, unknown>

  /** 回测默认参数 */
  backtestDefaults: Record<string, unknown>

  /** 创建时间 */
  createdAt: Date
}
```

## 三、核心逻辑

### 3.1 处理流程（7 步）

```
1. 校验因子名称存在性  ← 复用 submitBacktest 中的 factorDefinition 查询
2. 校验策略名称唯一性  ← 同一用户下 [userId, name] 唯一约束
3. 校验用户策略数量上限 ← 上限 50 条
4. 构建 strategyConfig  ← 映射 conditions → FactorScreeningRotationStrategyConfig
5. 校验 strategyConfig  ← 通过 BacktestStrategyRegistryService.validateStrategyConfig()
6. 构建 backtestDefaults ← 从 DTO 中提取回测参数
7. 写入 Strategy 表      ← prisma.strategy.create()
```

### 3.2 strategyConfig 映射

与 `submitBacktest()` 完全一致的映射逻辑：

```typescript
const strategyConfig: FactorScreeningRotationStrategyConfig = {
  conditions: dto.conditions.map((c) => ({
    factorName: c.factorName,
    operator: c.operator,
    value: c.value,
    min: c.min,
    max: c.max,
    percent: c.percent,
  })),
  sortBy: dto.sortBy ?? dto.conditions[0]?.factorName,
  sortOrder: dto.sortOrder ?? 'desc',
  topN: dto.topN ?? 20,
  weightMethod: dto.weightMethod ?? 'equal_weight',
}
```

### 3.3 backtestDefaults 构建

将回测相关参数打包为 `backtestDefaults`，使用户后续通过 `POST /strategy/run` 发起回测时自动填入预设值：

```typescript
const rebalanceDays = dto.rebalanceDays ?? 5
let rebalanceFrequency: string
if (rebalanceDays <= 1) rebalanceFrequency = 'DAILY'
else if (rebalanceDays <= 7) rebalanceFrequency = 'WEEKLY'
else if (rebalanceDays <= 25) rebalanceFrequency = 'MONTHLY'
else rebalanceFrequency = 'QUARTERLY'

const backtestDefaults = {
  initialCapital: dto.initialCapital ?? 1_000_000,
  rebalanceFrequency,
  commissionRate: dto.commissionRate ?? 0.0003,
  stampDutyRate: 0.0005,
  slippageBps: dto.slippageBps ?? 5,
  benchmarkTsCode: dto.benchmarkCode ?? '000300.SH',
  universe: dto.universe ? 'CUSTOM' : 'ALL_A',
  maxPositions: dto.topN ?? 20,
  maxWeightPerStock: 0.2,
  minDaysListed: 60,
  priceMode: 'NEXT_OPEN',
}
```

### 3.4 Strategy 记录创建

```typescript
const strategy = await this.prisma.strategy.create({
  data: {
    userId,
    name: dto.name,
    description: dto.description ?? null,
    strategyType: 'FACTOR_SCREENING_ROTATION',
    strategyConfig: validatedConfig as Prisma.InputJsonValue,
    backtestDefaults: backtestDefaults as Prisma.InputJsonValue,
    tags: dto.tags ?? [],
    version: 1,
  },
})
```

## 四、依赖分析

### 4.1 模块依赖

```
FactorModule
  ├── imports: [BacktestModule]  ← 已有，无需修改
  └── providers:
       └── FactorBacktestService
            ├── PrismaService               ← 已注入
            ├── FactorComputeService         ← 已注入
            ├── BacktestRunService           ← 已注入（来自 BacktestModule）
            └── BacktestStrategyRegistryService ← 新增注入（BacktestModule 已 export）
```

### 4.2 为什么不引入 StrategyModule

- `StrategyService.create()` 的校验逻辑中，`schemaValidator.validate()` 实际委托给 `BacktestStrategyRegistryService.validateStrategyConfig()`
- `BacktestStrategyRegistryService` 已由 `BacktestModule` 导出，`FactorModule` 已 import `BacktestModule`
- 因此只需新增注入 `BacktestStrategyRegistryService`，无需引入 `StrategyModule`，**避免不必要的模块耦合**
- 用户策略上限（50 条）、名称唯一约束等业务规则在 `saveAsStrategy()` 内直接实现（与 `StrategyService.create()` 逻辑保持一致）

### 4.3 数据依赖

| 表                 | 用途                         | 读/写   |
| ------------------ | ---------------------------- | ------- |
| `FactorDefinition` | 校验因子名称存在 & 启用状态  | 读      |
| `Strategy`         | 查询用户策略数量、写入新策略 | 读 + 写 |

## 五、错误处理

| 场景                    | 异常                                         | HTTP 状态 |
| ----------------------- | -------------------------------------------- | --------- |
| 因子名称不存在          | `NotFoundException('因子 "xxx" 不存在')`     | 404       |
| 因子已禁用              | `NotFoundException('因子 "xxx" 已禁用')`     | 404       |
| 策略名称重复            | `BusinessException(STRATEGY_NAME_EXISTS)`    | 400       |
| 用户策略数量超限        | `BusinessException(STRATEGY_LIMIT_EXCEEDED)` | 400       |
| strategyConfig 校验失败 | `BusinessException` (code 4001)              | 400       |
| 标签数量超限（>10）     | `BusinessException('标签最多 10 个')`        | 400       |

## 六、安全考量

- 端点受 `JwtAuthGuard` 保护，`userId` 从 JWT token 提取，不可伪造
- 策略数量上限 50 条/用户，防止资源滥用
- `strategyConfig` 通过 `BacktestStrategyRegistryService` 校验，不会存入非法结构
- 标签最多 10 个、每个最长 30 字符

## 八、与 submitBacktest 的关系

两个方法共享相同的**因子条件映射逻辑**（conditions → strategyConfig）：

|                     | `submitBacktest()`      | `saveAsStrategy()`                    |
| ------------------- | ----------------------- | ------------------------------------- |
| 输入                | FactorBacktestSubmitDto | SaveAsStrategyDto                     |
| 因子校验            | ✅ 查 FactorDefinition  | ✅ 查 FactorDefinition                |
| strategyConfig 构建 | ✅ 相同映射             | ✅ 相同映射                           |
| 输出                | BacktestRun（入队执行） | Strategy（持久化模板）                |
| 回测参数            | 直接传给 createRun      | 存入 backtestDefaults                 |
| startDate/endDate   | 必填                    | 不需要（按需在 /strategy/run 时指定） |

可考虑提取 `private buildStrategyConfig(conditions, dto)` 内部方法消除重复，但不强制——两处逻辑独立清晰也可接受。
