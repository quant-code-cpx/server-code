# 模块一：Auth（认证）

## A1. 测试范围

| 项目 | 内容 |
| ---- | ---- |
| 模块 | Auth - 认证 |
| 相关页面/业务入口 | 登录页、Token 刷新、退出登录 |
| 接口列表 | `POST /auth/captcha`、`POST /auth/login`、`POST /auth/refresh`、`POST /auth/logout` |
| 用户角色 | 匿名（所有端点均 @Public） |
| 依赖数据 | users 表、Redis（验证码/登录失败计数/Token 黑名单/Refresh Token） |
| 外部依赖 | 无 |
| 不在本轮范围 | JwtAuthGuard 单独测试（归 Lifecycle 模块） |

## A2. 业务理解

### A2.1 业务场景

- 用户访问系统前必须通过 account + password + captcha 登录，获得 accessToken（JWT）。
- accessToken 过期后可通过 refreshToken 无感续期。
- 用户可主动退出登录，使当前 Token 失效。
- 连续 5 次登录失败后账户锁定 10 分钟。

### A2.2 核心业务规则

| 规则编号 | 规则 | 来源 | 测试影响 |
| -------- | ---- | ---- | -------- |
| R-A01 | 验证码 60s 过期，一次性使用 | auth.constant.ts CAPTCHA_TTL=60s | 验证码过期/重复使用应拒绝 |
| R-A02 | 登录失败 5 次锁定 300s | LOGIN_MAX_FAIL=5, LOGIN_FAIL_WINDOW=300s | 第 5 次失败后应锁定 |
| R-A03 | 锁定期间任何登录尝试均拒绝 | LOGIN_LOCK_DURATION=600s | 锁定期内正确密码也应拒绝 |
| R-A04 | refreshToken 有 10s 宽限期 | REFRESH_TOKEN_GRACE=10s | 旧 RT 在 10s 内仍可用 |
| R-A05 | logout 将 accessToken 加入 Redis 黑名单 | TokenService | logout 后 AT 不可再用 |
| R-A06 | 验证码不区分大小写 | 设计文档 | 大小写不同应能通过 |
| R-A07 | 登录成功返回 accessToken | AccessTokenResponseDto | 响应必须含 accessToken 字段 |

### A2.3 状态变化与不变量

| 类型 | 说明 |
| ---- | ---- |
| 数据写入 | users.lastLoginAt 更新 |
| 缓存/队列/通知 | Redis: captcha、login_fail_count、login_lock、refresh_token、token_blacklist |
| 权限边界 | 所有端点 @Public，无需 Token |
| 数据不变量 | 密码 bcrypt 不可逆、account 唯一 |

## A3. 接口契约

| 接口 | Body/Param | 响应数据 | 鉴权/角色 | 备注 |

| POST `/auth/logout` | Authorization header + cookie | null | Public | 黑名单 AT，撤销 RT |

## C2. 业务理解

### C2.1 业务场景

- **事件分析**：选择事件类型，计算事件窗口内的 AR（超额收益）、CAR（累计超额收益）、AAR（平均超额收益），并做 t 检验。
- **信号规则**：用户创建基于事件的信号规则（如"业绩预增>50%时买入"），系统按规则扫描并生成信号。

| 规则编号 | 规则 | 来源 | 测试影响 |

| R-C02 | AR = stockReturn - benchmarkReturn | event-study.service.ts | 验证 AR 计算正确 |
| R-C03 | CAR = sum(AR) for all days in window | 同上 | CAR 可手算验证 |
| R-C04 | 8 种事件类型 | EventType enum | 每种类型都能查询 |
| R-C05 | signal-rules/scan 需 SUPER_ADMIN | @Roles(SUPER_ADMIN) | 普通用户/ADMIN 应 403 |
| R-C06 | signal-rules CRUD 只操作自己的 | @CurrentUser() | 越权应拒绝 |

## 遗留与建议

1. **LoginDto 校验缺失**（P4-B1）：建议给 LoginDto 四个字段加 `@IsString()` + `@IsNotEmpty()`，避免空 body 穿透到 service。
2. **Calendar 模块独立 JwtAuthGuard**：CalendarController 使用类级别 `@UseGuards(JwtAuthGuard)`，测试时需 overrideGuard，建议统一为全局 guard + `@Public()` 模式。
3. **数据层测试**：本轮为 Controller/DTO/Guard 层测试（mock service），数据正确性（AR/CAR 计算精度、分桶逻辑、聚合统计）需在有真实 DB 数据的环境中补充。
4. **E2E 测试**：跨模块链路（如事件日历 → 历史趋势 → 事件研究分析）需启动完整应用测试。
