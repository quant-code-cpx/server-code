# Agent 编排器与版本化工作流测试执行报告（2026-07-20）

## 1. 结论

Batch 011 已完成。单 Agent 显式工作流可按固定版本执行、暂停恢复、取消、预算终止、引用修复并原子提交终态；未引入 LangGraph、多 Agent、Worker、Controller、HTTP/SSE 或新数据库表。

代码提交：`20cc3df feat(agent): add versioned workflow orchestrator`。

## 2. 实现范围

- 固定节点：`load_context → plan → authorize_tools → execute_tools → synthesize → validate_citations → persist → complete`。
- `WorkflowRegistryService` 冻结 TypeScript 定义、节点顺序、Tool allowlist、Workflow/Prompt hash。
- `WorkflowEngineService` 管理 Step、checkpoint、恢复、lease heartbeat、取消、deadline 与预算。
- `AgentOrchestratorService` 负责 claim、固定版本解析及 `COMPLETED/FAILED/CANCELLED` 终态映射。
- Research Plan 只允许 15 个 canonical Tool；校验 capability、版本、依赖、环、次数和并行上限。
- 只读 Tool 可有限并行；执行仍统一经过 `ToolExecutorService`，模型统一经过 `ModelGatewayPort`。
- 模型调用写 `AiModelCall` 审计；已成功 Model/Tool/Step 可在恢复时复用。
- 引用覆盖失败最多 repair 一次；仍失败返回 `AI_CITATION_INVALID`，不生成伪引用。
- 最终消息、引用、complete Step、`agent.completed` 事件和 Run `COMPLETED` 在同一 PostgreSQL 事务提交。
- 发布命令：`pnpm run agent:workflow:publish-v1`。禁止覆盖已发布的不同 hash 内容。

## 7. 发现并修复的问题

- 高：原有消息仓库没有“最终消息 + 引用 + completed event + Run 终态”原子边界。新增专用完成 Repository 和真实回滚测试。
- 高：checkpoint 复用审计 sanitizer 默认深度过浅，复杂计划/事实包可能被截断。执行 payload 改为受字节上限约束的深层 JSON sanitizer。
- 中：fake Model 不支持 nullable union 与 `date/date-time` strict schema；Model Gateway 也未注册日期 format。两处均已修复并补回归。
- 环境：Docker Desktop 4.20.1 再次出现 PostgreSQL `D` 状态与端口转发失效。项目级 restart 无法收到退出事件后，重启 Docker Desktop 引擎恢复；未删除 volume 或业务数据。恢复后所有容器与探针正常。
