## 一、现状分析与目标

### 1.1 现状

| 维度           | 当前状态                                                 | 差距                                                 |
| -------------- | -------------------------------------------------------- | ---------------------------------------------------- |
| 指标暴露       | ❌ 无 Prometheus `/metrics` 端点                         | 无法被 Prometheus 采集，不支持 Grafana 可视化        |
| 仪表盘         | ❌ 无 Grafana / Prometheus 基础设施                      | 只能通过日志文件手动分析                             |
| 数据库查询性能 | ❌ Prisma Client 未启用 query event 监听                 | 慢查询无法及时发现，缺乏聚合统计                     |
| 同步监控       | ⚠️ 有 `TushareSyncLog` 表记录状态，有 WebSocket 实时推送 | 缺少 Prometheus 指标化暴露，无法做趋势分析和告警阈值 |

- **Prisma 慢查询检测**：记录超过阈值的数据库查询到日志，并通过 Prometheus Histogram 暴露查询耗时分布
- **Tushare 同步指标**：暴露每个同步任务的耗时直方图、成功/失败计数器

## 二、任务 1：Prometheus 指标接入

### 2.1 方案选型

使用 [`@willsoto/nestjs-prometheus`](https://github.com/willsoto/nestjs-prometheus)，理由：

- NestJS 生态最广泛使用的 Prometheus 集成库
- 基于 `prom-client`，支持自动采集 Node.js 默认指标（GC、事件循环延迟、内存用量）
- 提供 `@InjectMetric()` 装饰器，与 NestJS DI 完美集成
- 自动注册 `/metrics` 端点

### 2.6 HTTP 指标拦截器

替换现有 `LoggingInterceptor` 的指标采集职责，独立为新的 `HttpMetricsInterceptor`。`LoggingInterceptor` 保持不变，只做日志输出。

```typescript
// src/shared/metrics/http-metrics.interceptor.ts
import { CallHandler, ExecutionContext, Injectable, NestInterceptor } from '@nestjs/common'
import { InjectMetric } from '@willsoto/nestjs-prometheus'
import { Counter, Histogram } from 'prom-client'
import { Observable, tap } from 'rxjs'
import { HTTP_REQUEST_DURATION, HTTP_REQUEST_TOTAL, HTTP_REQUEST_ERRORS } from './metrics.constants'

/** 健康检查等不需要计量的路径 */
const EXCLUDED_PATHS = ['/metrics', '/health', '/ready', '/api/health', '/api/ready']

@Injectable()
export class HttpMetricsInterceptor implements NestInterceptor {
  constructor(
    @InjectMetric(HTTP_REQUEST_DURATION) private readonly durationHistogram: Histogram,
    @InjectMetric(HTTP_REQUEST_TOTAL) private readonly requestCounter: Counter,
    @InjectMetric(HTTP_REQUEST_ERRORS) private readonly errorCounter: Counter,
  ) {}

  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const request = context.switchToHttp().getRequest()
    const { method, url } = request

    if (EXCLUDED_PATHS.some((p) => url.startsWith(p))) {
      return next.handle()
    }

    // 提取规范化路由：用 Controller route pattern 而非原始 URL（避免基数爆炸）
    const route = this.extractRoute(context)
    const endTimer = this.durationHistogram.startTimer({ method, route })

    return next.handle().pipe(
      tap({
        next: () => {
          const statusCode = context.switchToHttp().getResponse().statusCode
          const labels = { method, route, status_code: String(statusCode) }
          endTimer(labels)
          this.requestCounter.inc(labels)
          if (statusCode >= 400) {
            this.errorCounter.inc(labels)
          }
        },
        error: (error) => {
          const statusCode = (error as any)?.status ?? 500
          const labels = { method, route, status_code: String(statusCode) }
          endTimer(labels)
          this.requestCounter.inc(labels)
          this.errorCounter.inc(labels)
        },
      }),
    )
  }

  /**
   * 从 ExecutionContext 中提取 Controller 路由模式（例如 /api/stock/:id）
   * 而非真实路径（/api/stock/000001.SZ），避免 label 基数爆炸。
   */
  private extractRoute(context: ExecutionContext): string {
    const request = context.switchToHttp().getRequest()
    // Express 已解析的路由模板
    return (request.route?.path as string) || request.url
  }
}
```

> **注册方式**：在 `src/main.ts` 中将 `HttpMetricsInterceptor` 追加为全局拦截器（通过 `app.get(HttpMetricsInterceptor)` 获取 DI 实例后注册）。

### 2.7 BullMQ 队列深度采集

```typescript
// src/shared/metrics/queue-metrics.service.ts
import { Injectable, Logger } from '@nestjs/common'
import { InjectMetric } from '@willsoto/nestjs-prometheus'
import { Gauge } from 'prom-client'
import { Cron, CronExpression } from '@nestjs/schedule'
import { Queue } from 'bullmq'
import { InjectQueue } from '@nestjs/bullmq'
import { BACKTESTING_QUEUE } from 'src/constant/queue.constant'
import { BULLMQ_QUEUE_DEPTH, BULLMQ_ACTIVE_JOBS } from './metrics.constants'

@Injectable()
export class QueueMetricsService {
  private readonly logger = new Logger(QueueMetricsService.name)

  constructor(
    @InjectQueue(BACKTESTING_QUEUE) private readonly backtestQueue: Queue,
    @InjectMetric(BULLMQ_QUEUE_DEPTH) private readonly queueDepthGauge: Gauge,
    @InjectMetric(BULLMQ_ACTIVE_JOBS) private readonly activeJobsGauge: Gauge,
  ) {}

  /** 每 15 秒采集一次队列状态，频率足够且不会给 Redis 造成压力 */
  @Cron(CronExpression.EVERY_10_SECONDS)
  async collectQueueMetrics() {
    try {
      const waiting = await this.backtestQueue.getWaitingCount()
      const active = await this.backtestQueue.getActiveCount()

      this.queueDepthGauge.set({ queue: BACKTESTING_QUEUE }, waiting)
      this.activeJobsGauge.set({ queue: BACKTESTING_QUEUE }, active)
    } catch (error) {
      this.logger.warn(`队列指标采集失败: ${(error as Error).message}`)
    }
  }
}
```

> **注意**：`QueueMetricsService` 需要注入 `@InjectQueue(BACKTESTING_QUEUE)`，因此 `MetricsModule` 需要导入 `BullModule.registerQueue({ name: BACKTESTING_QUEUE })`，或者将 `QueueMetricsService` 移至 `QueueModule` 内部并只在该模块注入指标。
>
> **推荐做法**：把 `QueueMetricsService` 放到 `src/queue/queue-metrics.service.ts`，在 `QueueModule` 中注册。`MetricsModule` 只负责声明指标 Provider 和 `PrometheusModule`。

### 2.8 `/metrics` 端点鉴权

`/metrics` 端点需跳过 JWT 鉴权（Prometheus 采集器不携带 Token），但生产环境应限制访问：

- 通过 `@Public()` 装饰器或在 `PrometheusModule.register()` 中标记无需鉴权
- `PrometheusModule` 默认注册的 Controller 没有挂 `@Public()`，需要通过以下方式处理：

```typescript
// 方案：在 JwtAuthGuard 中白名单放行 /metrics
// src/lifecycle/guard/jwt-auth.guard.ts
const PUBLIC_PATHS = ['/metrics']

canActivate(context: ExecutionContext) {
  const request = context.switchToHttp().getRequest()
  if (PUBLIC_PATHS.some((p) => request.url?.startsWith(p))) {
    return true
  }
  // ... 原有逻辑
}
```

生产环境可通过网络策略（如 Docker network 隔离、Nginx 反向代理白名单）限制只有 Prometheus 容器能访问 `/metrics`。

### 3.7 预置仪表盘模板

`monitoring/grafana/dashboards/quant-server.json` — 包含以下面板：

#### Row 1：HTTP 概览

| 面板                  | 类型         | PromQL                                                                       |
| --------------------- | ------------ | ---------------------------------------------------------------------------- |
| QPS（请求速率）       | Time Series  | `rate(http_requests_total{app="quant-server"}[1m])`                          |
| HTTP 延迟 P50/P95/P99 | Time Series  | `histogram_quantile(0.5, rate(http_request_duration_seconds_bucket[5m]))` 等 |

#### Row 2：Node.js 运行时

| 面板                     | 类型        | PromQL                                                         |
| ------------------------ | ----------- | -------------------------------------------------------------- |
| 进程内存（RSS / Heap）   | Time Series | `process_resident_memory_bytes`, `nodejs_heap_size_used_bytes` |
| GC 暂停时间              | Time Series | `rate(nodejs_gc_duration_seconds_sum[5m])`                     |
| 事件循环延迟             | Time Series | `nodejs_eventloop_lag_seconds`                                 |
| 活跃 Handle / Request 数 | Time Series | `nodejs_active_handles_total`, `nodejs_active_requests_total`  |

#### Row 3：数据库

| 面板                     | 类型        | PromQL                                                                     |
| ------------------------ | ----------- | -------------------------------------------------------------------------- |
| Prisma 查询延迟 P50/P95  | Time Series | `histogram_quantile(0.95, rate(prisma_query_duration_seconds_bucket[5m]))` |
| 慢查询率（>500ms）       | Stat        | `rate(prisma_query_duration_seconds_bucket{le="0.5"}[5m])` 反向计算        |
| 查询 QPS（按 operation） | Time Series | `rate(prisma_queries_total[1m])`                                           |

#### Row 4：BullMQ 队列

| 面板          | 类型  | PromQL               |
| ------------- | ----- | -------------------- |
| 等待中 Job 数 | Gauge | `bullmq_queue_depth` |
| 活跃 Job 数   | Gauge | `bullmq_active_jobs` |

#### Row 5：Tushare 同步

| 面板                  | 类型                   | PromQL                                                                                                      |
| --------------------- | ---------------------- | ----------------------------------------------------------------------------------------------------------- |
| 同步任务成功/失败计数 | Time Series（Stacked） | `increase(tushare_sync_tasks_total[1d])` by `status`                                                        |
| 同步任务耗时 Top 10   | Bar Gauge              | `topk(10, histogram_quantile(0.95, rate(tushare_sync_duration_seconds_bucket[1d])))`                        |
| 成功率（%）           | Stat                   | `sum(rate(tushare_sync_tasks_total{status="success"}[1d])) / sum(rate(tushare_sync_tasks_total[1d])) * 100` |

#### Row 6：缓存

| 面板         | 类型        | PromQL                                                                                                                                                               |
| ------------ | ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 缓存操作速率 | Time Series | `rate(cache_operations_total[5m])` by `operation`                                                                                                                    |
| 命中率       | Stat        | `rate(cache_operations_total{operation="hit"}[5m]) / (rate(cache_operations_total{operation="hit"}[5m]) + rate(cache_operations_total{operation="miss"}[5m])) * 100` |

## 四、任务 3：数据库慢查询监控

### 4.1 方案

利用 Prisma Client Extension（`$extends`）拦截所有查询调用，测量耗时并：

1. 当耗时超过阈值（默认 500ms）时输出 **WARN 级别日志**
2. 所有查询耗时写入 Prometheus Histogram（`prisma_query_duration_seconds`）
3. 查询计数写入 Counter（`prisma_queries_total`）

### 4.2 为什么选择 Client Extension 而非 `$on('query')`

- `$on('query')` 是旧 API，在 Prisma v5+ 中已标记为 deprecated，且只能在 `PrismaClient` 实例化时启用 `log: [{ emit: 'event', level: 'query' }]`，会对所有查询产生额外日志开销。
- Client Extension（`$extends`）是推荐方式，可在 model/query 层面拦截，性能开销更小，且支持类型安全。

### 4.3 PrismaService 改造

```typescript
// src/shared/prisma.service.ts — 新增指标注入与扩展

import { Inject, Injectable, OnModuleDestroy, OnModuleInit, Optional } from '@nestjs/common'
import { PrismaClient } from '@prisma/client'
import { Histogram, Counter } from 'prom-client'
import { LoggerService } from './logger/logger.service'

const SLOW_QUERY_THRESHOLD_MS = parseInt(process.env.PRISMA_SLOW_QUERY_THRESHOLD_MS ?? '', 10) || 500

@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  /** 带有查询指标拦截的扩展 Client，业务代码通过此实例访问数据库 */
  private _extended: ReturnType<typeof this.withMetrics> | null = null

  constructor(
    private readonly logger: LoggerService,
    @Optional() @Inject('PRISMA_QUERY_DURATION_HISTOGRAM') private readonly queryDuration?: Histogram,
    @Optional() @Inject('PRISMA_QUERY_TOTAL_COUNTER') private readonly queryTotal?: Counter,
  ) {
    // ... 现有 super() 逻辑保持不变
  }

  async onModuleInit() {
    await this.$connect()
    // 初始化扩展实例
    if (this.queryDuration || this.queryTotal) {
      this._extended = this.withMetrics()
    }
  }

  /**
   * 获取带指标采集能力的 Prisma Client 实例。
   * 如果 Prometheus 指标模块未加载（如测试环境），则返回原始 PrismaClient。
   */
  get client() {
    return this._extended ?? this
  }

  private withMetrics() {
    return this.$extends({
      query: {
        $allModels: {
          async $allOperations({ model, operation, args, query }) {
            const start = performance.now()
            try {
              const result = await query(args)
              const durationMs = performance.now() - start
              this.recordQueryMetrics(model, operation, durationMs)
              return result
            } catch (error) {
              const durationMs = performance.now() - start
              this.recordQueryMetrics(model, operation, durationMs)
              throw error
            }
          },
        },
      },
    })
  }

  private recordQueryMetrics(model: string | undefined, operation: string, durationMs: number) {
    const labels = { model: model ?? 'unknown', operation }
    const durationSec = durationMs / 1000

    this.queryDuration?.observe(labels, durationSec)
    this.queryTotal?.inc(labels)

    if (durationMs > SLOW_QUERY_THRESHOLD_MS) {
      this.logger.warn(
        {
          message: `慢查询检测: ${model}.${operation} 耗时 ${durationMs.toFixed(1)}ms`,
          model,
          operation,
          durationMs: Math.round(durationMs),
          threshold: SLOW_QUERY_THRESHOLD_MS,
        },
        'PrismaSlowQuery',
      )
    }
  }
}
```

> **重要设计决策**：
>
> - 使用 `@Optional()` 注入指标实例，确保在未加载 `MetricsModule` 的测试环境中不会因 DI 缺失而崩溃。
> - 指标注入使用**字符串 token**（`PRISMA_QUERY_DURATION_HISTOGRAM`、`PRISMA_QUERY_TOTAL_COUNTER`），在 `MetricsModule` 中声明 Provider 时通过 `provide` / `useExisting` 映射。
> - `$extends` 返回的类型与原始 `PrismaClient` 不完全一致；业务代码如果需要使用扩展实例上的查询方法，可以通过 `prisma.client.xxx` 访问。但考虑到改动面过大，**采用保守方案**：让 `PrismaService` 自身仍作为 `PrismaClient` 使用（各业务注入后直接 `this.prisma.model.xxx`），指标采集通过 Prisma middleware 方式实现（见下方 4.4 替代方案）。

### 4.4 替代方案：Prisma `$use` Middleware（推荐采纳）

由于项目中所有业务代码直接访问 `this.prisma.<model>`（而非 `this.prisma.client.<model>`），改用 `$extends` 需要大规模改造调用方。因此**推荐使用 Prisma Middleware**：

```typescript
// src/shared/prisma.service.ts — 在 onModuleInit() 中注册 middleware

async onModuleInit() {
  // 注册查询指标 middleware（在 $connect 之前注册）
  this.$use(async (params, next) => {
    const start = performance.now()
    try {
      const result = await next(params)
      const durationMs = performance.now() - start
      this.recordQueryMetrics(params.model, params.action, durationMs)
      return result
    } catch (error) {
      const durationMs = performance.now() - start
      this.recordQueryMetrics(params.model, params.action, durationMs)
      throw error
    }
  })

  await this.$connect()
}
```

> **注**：Prisma `$use` middleware 在 Prisma v5+ 中标记为 soft-deprecated（官方推荐 `$extends`），但短期内仍然可用且稳定。当项目未来统一迁移到 `$extends` 时再做调整。选择 `$use` 是为了**零侵入**——不需要修改任何业务代码的数据访问方式。

### 4.5 指标 Token Provider 桥接

在 `MetricsModule` 中添加 Provider 桥接，使 `PrismaService` 能通过字符串 token 注入指标：

```typescript

  provide: 'PRISMA_QUERY_DURATION_HISTOGRAM',
  useExisting: getToken(PRISMA_QUERY_DURATION),  // @willsoto/nestjs-prometheus 的 getToken 工具
},
{
  provide: 'PRISMA_QUERY_TOTAL_COUNTER',
  useExisting: getToken(PRISMA_QUERY_TOTAL),
},
```

从 `@willsoto/nestjs-prometheus` 导入 `getToken`：

```typescript

```

## 五、任务 4：Tushare 同步耗时与成功率监控

### 5.1 方案

在 `TushareSyncService` 的 `safeRun()` 和 `runPlans()` 方法中注入 Prometheus 指标，记录：

1. 每个同步任务的**耗时分布**（Histogram）
2. 每个同步任务的**执行计数**（Counter），按 `task`、`category`、`trigger`、`status` 维度区分

### 5.2 TushareSyncService 改造

```typescript
// src/tushare/sync/sync.service.ts — 注入指标

import { InjectMetric } from '@willsoto/nestjs-prometheus'
import { Counter, Histogram } from 'prom-client'
import { TUSHARE_SYNC_DURATION, TUSHARE_SYNC_TOTAL } from 'src/shared/metrics/metrics.constants'

@Injectable()
export class TushareSyncService implements OnApplicationBootstrap {
  // ... 现有字段

  constructor(
    // ... 现有注入
    @InjectMetric(TUSHARE_SYNC_DURATION) private readonly syncDurationHistogram: Histogram,
    @InjectMetric(TUSHARE_SYNC_TOTAL) private readonly syncTotalCounter: Counter,
  ) {
    // ... 现有逻辑
  }

  // safeRun 改造 ——
  private async safeRun(
    plan: TushareSyncPlan,
    fn: () => Promise<void>,
    failedTasks: FailedTask[],
    executedTasks: TushareSyncTaskName[],
    trigger: TushareSyncTrigger, // ← 新增参数
  ) {
    const endTimer = this.syncDurationHistogram.startTimer({
      task: plan.task,
      category: plan.category,
      trigger,
    })

    try {
      this.logger.log(`▶ ${plan.label}`)
      await fn()
      executedTasks.push(plan.task)

      endTimer({ status: 'success' })
      this.syncTotalCounter.inc({
        task: plan.task,
        category: plan.category,
        trigger,
        status: 'success',
      })
    } catch (error) {
      this.logger.error(`✗ ${plan.label} 失败: ${(error as Error).message}`)
      failedTasks.push({ task: plan.task, label: plan.label, fn, error: error as Error })

      endTimer({ status: 'failure' })
      this.syncTotalCounter.inc({
        task: plan.task,
        category: plan.category,
        trigger,
        status: 'failure',
      })
    }
  }
}
```

### 5.4 与现有 SyncLog 的关系

现有 `TushareSyncLog` 表已记录每次同步任务的状态和时间戳，本次 Prometheus 指标是**补充维度**：

| 维度     | SyncLog（数据库）        | Prometheus 指标                             |
| -------- | ------------------------ | ------------------------------------------- |
| 持久性   | 永久存储                 | 内存态，重启后清零（Prometheus 自身持久化） |
| 查询方式 | SQL / API                | PromQL                                      |
| 用途     | 审计、断点续传、详情查看 | 趋势分析、告警、仪表盘                      |
| 实时性   | 写入后可查               | 实时更新                                    |

两者互补，不需要替换。
