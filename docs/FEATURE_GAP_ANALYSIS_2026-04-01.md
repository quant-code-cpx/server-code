## 一、项目现状速览

| 维度 | 现状 |
|------|------|
| **技术栈** | NestJS 11 + Prisma + PostgreSQL 17 + Redis 7.4 + BullMQ + Socket.IO |
| **数据源** | Tushare Pro，已接入 22 类数据，涵盖日/周/月线、复权因子、财务三表、资金流向等 |
| **数据规模** | ~1 亿行（Daily 17.6M + AdjFactor 18.4M + DailyBasic 17.5M + 财务/资金流各 30 万+） |
| **API 数量** | 90+ 端点（Auth 4 / User 9 / Stock 28 / Market 16 / Factor 9 / Backtest 7 / Heatmap 1 等） |
| **Schema 模型** | 31 个 Prisma Model，涵盖用户、行情、财务、资金流、回测、因子 |
| **已完成模块** | 用户管理 ✅ / JWT 鉴权 ✅ / 股票列表与详情 ✅ / 市场总览 ✅ / 选股器 ✅ / 热力图 ✅ / 因子库 ✅ / 回测框架 ✅ |
| **部分完成** | 技术指标（占位）/ 因子分析（数据表空）/ 回测执行引擎（骨架） |
| **测试覆盖** | 4 个 spec 文件，约 5% 覆盖率 |

**核心判断**：项目基础架构扎实、数据层丰富，但从"能用于真实量化研究"角度仍有若干关键缺口。

## 二、痛点一：数据层缺口

### 2.1 三张关键空表导致因子与回测不可用

| 表名 | 用途 | 当前状态 | 影响 |
|------|------|----------|------|
| `stk_limit` | 涨跌停价格 | 0 行 | 回测无法判断涨跌停不可交易，因子分析无法剔除极端样本 |
| `suspend_d` | 停牌日期 | 0 行 | 回测执行引擎无法跳过停牌标的 |
| `index_weight` | 指数成分权重 | 0 行 | 无法构建沪深300/中证500等对标基准池 |

> **痛感**：这是整条"数据 → 因子 → 回测"链路的前置瓶颈，不补数据后续功能全部空转。

### 2.2 缺少的高价值数据

| 数据 | Tushare 接口 | 量化研究价值 |
|------|-------------|-------------|
| **龙虎榜** | `top_list` / `top_inst` | 游资追踪、异动分析、主力行为研究 |
| **大宗交易** | `block_trade` | 机构减持信号、折价率分析 |
| **融资融券** | `margin_detail` | 市场杠杆水平、融券做空情绪指标 |
| **限售股解禁** | `share_float` | 解禁压力日历、事件驱动策略 |
| **ETF/LOF 净值** | `fund_nav` / `fund_daily` | 行业轮动、ETF 套利、资产配置 |
| **期权数据** | `opt_basic` / `opt_daily` | 隐含波动率、期权定价、Put-Call Ratio |
| **宏观经济指标** | `cn_cpi` / `cn_ppi` / `cn_gdp` / `shibor` | 宏观择时、利率周期判断 |
| **行业分类映射** | `concept` / `concept_detail` | 概念板块选股、主题投资 |

### 2.3 数据质量问题

- **无数据校验层**：Tushare 返回的脏数据（空值、异常值、重复记录）直接入库，缺少 ETL 清洗环节
- **无数据完整性监控**：无法自动发现某只股票某天数据缺失
- **无历史数据修正机制**：财务数据事后修正（restatement）无法反映

## 三、痛点二：回测引擎尚未落地

### 3.1 现状

- BullMQ 任务队列 ✅、任务提交/状态查询 API ✅、结果 Schema ✅
- **实际执行引擎为空壳**：`BacktestEngineService` 仅返回模拟数据，无真实下单/撮合/净值计算逻辑

### 3.2 缺失的核心能力

| 能力 | 说明 |
|------|------|
| **撮合引擎** | 按日频模拟买卖，考虑涨跌停/停牌/滑点/手续费 |
| **组合管理** | 多标的仓位跟踪、现金管理、权重再平衡 |
| **策略执行框架** | 将 4 种策略模板（均线交叉/选股轮动/因子排名/自定义池）翻译为可执行信号 |
| **基准对比** | 同步计算基准净值曲线，输出超额收益/跟踪误差 |
| **风险指标计算** | Sharpe/Sortino/Calmar/Max Drawdown/年化波动率 应在引擎内实时计算 |
| **参数验证** | 回测区间数据完整性检查、资金量合理性校验 |

### 3.3 进阶需求

- **Walk-forward 验证**：样本内优化 + 样本外验证，防止过拟合
- **多策略对比**：同一时间段多组回测结果并排比较
- **蒙特卡洛模拟**：随机重采样估算策略鲁棒性
- **滚动窗口回测**：定期重新训练/筛选，模拟真实投研流程

## 四、痛点三：因子研究闭环不完整

### 4.1 现状

- 因子库 ✅（12 个分类，FIELD_REF / DERIVED / CUSTOM_SQL 三种来源）
- IC 分析 / 分位回测 / 衰减分析 / 分布 / 相关矩阵 ✅
- **底层数据为空** → 所有分析接口实际返回空结果

### 4.2 缺失环节

| 环节 | 说明 |
|------|------|
| **自定义因子引擎** | 用户通过表达式语言（如 `rank(close/delay(close,20))` ）定义新因子，后端实时计算 |
| **因子值批量预计算** | 定时任务每日收盘后批量计算全市场因子截面值并落库，避免查询时实时计算 |
| **因子正交化** | 多因子组合时需做回归正交（Fama-MacBeth / 截面回归），当前无此能力 |
| **因子回测到组合** | 从"因子 → 选股 → 组合 → 回测"全链路打通，当前因子筛选与回测模块脱节 |
| **Barra 风险模型** | CNE5/CNE6 风格因子分解，量化界标配但实现复杂 |

## 五、痛点四：缺少自选股与研究工作台

### 5.1 问题

目前系统是"工具型"——用户查完即走，无法**沉淀研究过程**。量化研究人员的核心诉求是：

- 建立自选股池（Watchlist），跟踪关注标的
- 对标的添加研究笔记和标签
- 保存筛选条件并设置定时推送（"每天收盘后看看有没有新进入条件的票"）
- 策略草稿箱（未完成的回测参数配置暂存）

### 5.2 建议新增

| 功能 | Schema 变动 | API |
|------|------------|-----|
| **自选股管理** | `Watchlist`（userId, name）+ `WatchlistItem`（watchlistId, tsCode, note, tags） | CRUD + 批量导入 |
| **研究笔记** | `ResearchNote`（userId, tsCode?, title, content, tags, createdAt） | CRUD + 按股票/标签查询 |
| **条件订阅** | `ScreenerSubscription`（userId, strategyId, frequency, lastRunAt, notifyMethod） | 创建/暂停/删除 |
| **策略草稿** | `StrategyDraft`（userId, name, config JSON, updatedAt） | 自动保存 + 加载 |

## 六、痛点五：缺少数据导出与分享能力

### 6.1 问题

量化研究人员通常需要把数据拿到 Python/R/Excel 中做进一步分析，但当前系统：
- 所有接口仅返回 JSON，无 CSV/Excel 导出
- 无批量数据下载接口
- 回测报告无法导出为 PDF
- 无法分享回测结果给其他用户

### 6.2 建议新增

| 功能 | 说明 | 优先级 |
|------|------|--------|
| **CSV/Excel 导出** | 在股票列表、选股结果、因子截面值等接口添加 `format=csv` 参数 | 高 |
| **回测报告导出** | 生成包含净值曲线、月度收益、交易明细的 PDF 报告 | 中 |
| **批量数据 API** | 供 Python SDK 调用的批量下载接口（按日期范围+代码列表） | 高 |
| **分享链接** | 回测结果生成只读分享链接（含过期时间） | 低 |
| **研究快照** | 将当前筛选/分析状态保存为可分享的"快照" | 低 |

## 七、痛点六：实时行情与预警空缺

### 7.1 现状

- WebSocket 基础设施 ✅（Socket.IO，回测进度推送已实现）
- **无任何实时行情数据源接入**
- **无价格/成交量/指标预警机制**

### 7.2 量化研究场景需求

| 功能 | 说明 | 实现路径 |
|------|------|----------|
| **盘中实时行情** | 按股票代码订阅实时报价（5秒级延迟可接受） | 接入 Tushare 实时接口 或第三方 WebSocket 源（如新浪/腾讯行情） |
| **价格预警** | 股价突破/跌破某阈值时 WebSocket+邮件通知 | `PriceAlert` 模型 + 定时轮询（盘中每分钟）或实时行情触发 |
| **异动监控** | 成交量异常放大、连板监测、大单成交预警 | 基于 moneyflow 数据 + 自定义规则引擎 |
| **事件日历** | 财报发布日、解禁日、分红除权日提醒 | 基于 `express` / `dividend` / `share_float` 数据 |
| **市场情绪仪表盘** | 涨跌家数比、涨停板数、北向资金实时变化 | 综合多数据源实时聚合 |

## 八、痛点七：可观测性与运维薄弱

### 8.1 现状

- Winston 文件日志 ✅，日志轮转 ✅
- **无集中式日志收集**（不能远程查看日志）
- **无指标监控**（不知道接口 P99 延迟、错误率）
- **无链路追踪**（无法追踪一个请求经过了哪些服务/查询）
- **无告警**（数据同步失败只有日志记录，无人收到通知）
- **无数据库备份策略**

### 8.2 建议分阶段建设

| 阶段 | 组件 | 说明 |
|------|------|------|
| **P0** | 健康检查端点 | `@nestjs/terminus` 实现 `/health` + `/ready`，检查 DB/Redis/BullMQ |
| **P0** | 结构化日志 | Winston 输出 JSON 格式，包含 traceId/userId/latency |
| **P1** | Prometheus 指标 | 使用 `@willsoto/nestjs-prometheus` 暴露 HTTP 延迟、QPS、错误率、队列深度 |
| **P1** | Grafana 仪表盘 | Docker Compose 添加 Grafana + Prometheus，预置仪表盘模板 |
| **P1** | 数据库自动备份 | pg_dump 定时任务 + 上传至对象存储（MinIO/S3） |
| **P2** | OpenTelemetry | 分布式链路追踪，打通 NestJS → Prisma → PostgreSQL |
| **P2** | AlertManager | Tushare 同步失败、API 错误率飙升、磁盘空间不足等场景告警 |

## 九、推荐功能：高价值新模块

以下功能超出现有 TODO 规划，但对量化研究后端有显著价值：

### 9.1 📊 量化报告生成引擎

**痛点**：研究员做完分析后需要手动整理结论，无法一键出报告。

**建议**：
- 提供"研究报告模板"机制，用户选择标的/策略/因子后自动拉取数据并组装
- 输出包含：基本面概览、技术面分析、因子暴露、回测摘要、风险提示
- 支持 PDF / Markdown / HTML 三种格式
- 定时生成（如每周五收盘后自动出周报）

### 9.2 🔗 Python SDK / REST API Client

**痛点**：量化研究人员主力工具是 Python（Jupyter Notebook），但当前只能通过 HTTP JSON 接口对接，没有封装好的 SDK。

**建议**：
- 提供 `pip install quant-server-sdk` 的 Python 包
- 基于 Swagger/OpenAPI 自动生成（`openapi-generator`）
- 封装登录鉴权、自动刷新 Token、数据自动转 pandas DataFrame
- 示例 Notebook：在 Jupyter 中拉取 K 线 → 计算因子 → 提交回测 → 获取结果

### 9.3 📈 行业轮动分析模块

**痛点**：当前市场模块关注个股和大盘资金流，缺少行业维度的趋势研究。

**建议**：
- 行业指数收益对比（近 5/20/60 日）
- 行业动量排名与反转检测
- 行业间资金流转热力图（从哪个行业流出、流入哪个行业）
- 行业 PE/PB 分位数（历史百分位判断估值水平）

### 9.4 📅 事件驱动研究模块

**痛点**：量化研究中"事件驱动策略"是重要方向，但当前无法研究事件对股价的影响。

**建议**：
- 事件类型：业绩预告/快报、分红公告、股东增减持、定向增发、限售解禁、高管变动
- 事件影响分析：事件前后 N 日的超额收益统计（Event Study）
- 事件日历视图：按日期聚合各类事件
- 事件信号生成器：符合条件的事件自动触发选股信号

### 9.5 🏦 组合管理与风控模块

**痛点**：回测完成后缺少"模拟持仓 → 日常跟踪 → 风控监控"的闭环。

**建议**：
- 虚拟组合管理：创建组合 → 添加/调整持仓 → 自动计算每日盈亏
- 组合分析：行业分布、个股集中度、市值分布、Beta 暴露
- 风控规则：单票仓位上限、回撤止损线、行业集中度限制
- 组合对比：多组合/组合 vs 基准的绩效对比

### 9.6 🤖 智能选股助手（NLP 驱动）

**痛点**：选股器需要用户手动配置筛选条件，门槛较高。

**建议**：
- 接入 LLM（本地部署或 API），支持自然语言选股："帮我找市盈率低于 20、ROE 大于 15%、近 3 个月主力资金净流入的沪深 300 成分股"
- 将自然语言转换为选股器 JSON 配置
- 提供选股条件解释和优化建议

### 9.7 📉 相似 K 线形态匹配

**痛点**：技术分析师常需查找历史上出现过相似走势的标的。

**建议**：
- 用户选择某只股票近 N 日 K 线 → 系统在全市场历史数据中搜索形态相似的区间
- 基于 DTW（Dynamic Time Warping）或归一化欧氏距离
- 返回相似区间及后续走势统计（上涨/下跌/震荡概率）

### 9.8 🔄 数据同步可视化管理台

**痛点**：当前 Tushare 同步只有管理员手动触发和日志查看，缺少直观的管理界面数据支持。

**建议**：
- 各数据表同步状态总览（最后同步时间、数据行数、缺失天数）
- 同步任务甘特图（各任务耗时、依赖关系）
- 数据完整性检查报告（哪些股票哪些日期缺数据）
- 异常数据标记与人工修正入口
