## 二、任务 1：健康检查端点

### 2.1 问题描述

当前无任何 HTTP 健康检查端点。Docker Compose 内部虽有 `pg_isready` / `redis-cli ping`，但不向外暴露，无法满足：

- 容器编排层（K8s liveness/readiness）探活需求
- 负载均衡器后端健康判定
- 运维人员快速判断服务状态

### 2.2 方案选型

使用 NestJS 官方推荐的 `@nestjs/terminus` 库，提供标准化健康检查框架。

**新增依赖**：

```
@nestjs/terminus ^11.0.0
```

### 2.3 端点设计

| 端点 | 用途 | 检查项 | 鉴权 |
|------|------|--------|------|
| `GET /health` | 存活探针（Liveness） | 应用进程是否正常响应 | 无需鉴权（`@Public()`） |
| `GET /ready` | 就绪探针（Readiness） | DB 连接 + Redis 连接 + 可选 BullMQ 状态 | 无需鉴权（`@Public()`） |

> **注**：健康检查端点使用 `GET` 方法，这是行业标准做法，是项目"所有 Controller 端点统一使用 POST"规则的**唯一例外**。K8s / 云厂商 / 负载均衡器默认使用 GET 探活，强行改 POST 会增加运维配置成本。

#### `/health` 响应

```json
{
  "status": "ok",
  "info": {},
  "error": {},
  "details": {}
}
```

#### `/ready` 响应（正常）

```json
{
  "status": "ok",
  "info": {
    "database": { "status": "up" },
    "redis": { "status": "up" }
  },
  "error": {},
  "details": {
    "database": { "status": "up" },
    "redis": { "status": "up" }
  }
}
```

#### `/ready` 响应（异常）

HTTP 503：

```json
{
  "status": "error",
  "info": {
    "redis": { "status": "up" }
  },
  "error": {
    "database": { "status": "down", "message": "Connection refused" }
  },
  "details": {
    "database": { "status": "down", "message": "Connection refused" },
    "redis": { "status": "up" }
  }
}
```

### 2.4 自定义健康指标

#### 2.4.1 PrismaHealthIndicator

基于 `@nestjs/terminus` 的 `HealthIndicator` 基类，执行 `SELECT 1` 探测 DB 连通性。

**文件**：`src/shared/health/prisma.health.ts`

```typescript
import { Injectable } from '@nestjs/common'
import { HealthCheckError, HealthIndicator, HealthIndicatorResult } from '@nestjs/terminus'
import { PrismaService } from '../prisma.service'

@Injectable()
export class PrismaHealthIndicator extends HealthIndicator {
  constructor(private readonly prisma: PrismaService) {
    super()
  }

  async isHealthy(key: string): Promise<HealthIndicatorResult> {
    try {
      await this.prisma.$queryRaw`SELECT 1`
      return this.getStatus(key, true)
    } catch (error) {
      throw new HealthCheckError(
        `${key} health check failed`,
        this.getStatus(key, false, { message: error.message }),
      )
    }
  }
}
```

#### 2.4.2 RedisHealthIndicator

基于已注入的 `REDIS_CLIENT`，执行 `PING` 命令探测。

**文件**：`src/shared/health/redis.health.ts`

```typescript
import { Inject, Injectable } from '@nestjs/common'
import { HealthCheckError, HealthIndicator, HealthIndicatorResult } from '@nestjs/terminus'
import { RedisClientType } from 'redis'
import { REDIS_CLIENT } from '../redis.provider'

@Injectable()
export class RedisHealthIndicator extends HealthIndicator {
  constructor(@Inject(REDIS_CLIENT) private readonly redis: RedisClientType) {
    super()
  }

  async isHealthy(key: string): Promise<HealthIndicatorResult> {
    try {
      const pong = await this.redis.ping()
      const isUp = pong === 'PONG'
      if (!isUp) {
        throw new Error(`Unexpected PING response: ${pong}`)
      }
      return this.getStatus(key, true)
    } catch (error) {
      throw new HealthCheckError(
        `${key} health check failed`,
        this.getStatus(key, false, { message: error.message }),
      )
    }
  }
}
```

### 2.6 HealthController

**文件**：`src/shared/health/health.controller.ts`

```typescript
import { Controller, Get } from '@nestjs/common'
import { ApiTags, ApiOperation } from '@nestjs/swagger'
import { HealthCheck, HealthCheckService } from '@nestjs/terminus'
import { Public } from 'src/common/decorators/public.decorator'
import { PrismaHealthIndicator } from './prisma.health'
import { RedisHealthIndicator } from './redis.health'

@ApiTags('Health')
@Controller()
export class HealthController {
  constructor(
    private readonly health: HealthCheckService,
    private readonly prismaHealth: PrismaHealthIndicator,
    private readonly redisHealth: RedisHealthIndicator,
  ) {}

  @Get('health')
  @Public()
  @ApiOperation({ summary: '存活探针（Liveness）' })
  @HealthCheck()
  liveness() {
    return this.health.check([])
  }

  @Get('ready')
  @Public()
  @ApiOperation({ summary: '就绪探针（Readiness）' })
  @HealthCheck()
  readiness() {
    return this.health.check([
      () => this.prismaHealth.isHealthy('database'),
      () => this.redisHealth.isHealthy('redis'),
    ])
  }
}
```

> **注**：健康检查端点不使用 `globalPrefix`（`api`），直接注册为 `/health` 和 `/ready`。如果需要保持在 `api/` 前缀下，则端点为 `GET /api/health` 和 `GET /api/ready`，两种方式均可，保持与 `globalPrefix` 一致即可。

## 三、任务 2：结构化日志

### 3.1 问题描述

当前生产环境 Winston 已输出 JSON 格式，但日志内容仅包含 `message`、`context`、`timestamp`、`level` 四个字段，缺少：

- `traceId`：关联同一请求的所有日志
- `userId`：标识请求发起者
- `latency`：记录处理耗时
- `method` / `url` / `statusCode`：HTTP 请求元数据

这导致日志聚合系统（ELK / Loki / CloudWatch）无法按请求维度检索和关联。

### 3.2 方案设计

#### 3.2.1 引入 AsyncLocalStorage 请求上下文

> 此为 **跨任务基础设施**，详见第五节。

使用 Node.js 内置的 `AsyncLocalStorage`（无需第三方库）在请求生命周期内传播上下文数据（`traceId`、`userId`、`method`、`url`）。

#### 3.2.2 增强 LoggerService

在 `LoggerService` 的 Winston 写入逻辑中，自动从 `AsyncLocalStorage` 获取请求上下文并注入到日志 metadata：

**改造后的 Winston format**：

```typescript
// src/shared/logger/logger.service.ts

private initWinstonLogger() {
  const baseConfig = {
    datePattern: 'YYYY-MM-DD',
    maxSize: '20m',
    maxFiles: 31,
  }

  this.winstonLogger = createLogger({
    levels: config.npm.levels,
    format: format.combine(
      format.errors({ stack: true }),
      format.timestamp(),
      // 新增：注入请求上下文字段
      format((info) => {
        const ctx = RequestContextService.getCurrentContext()
        if (ctx) {
          info.traceId = ctx.traceId
          info.userId = ctx.userId ?? null
          info.method = ctx.method ?? null
          info.url = ctx.url ?? null
        }
        return info
      })(),
      format.json(),
    ),
    transports: [/* 保持不变 */],
  })
}
```

#### 3.2.3 结构化日志输出示例

改造后，每条生产日志的 JSON 格式如下：

```json
{
  "level": "info",
  "message": "POST /api/stock/list - 45ms",
  "timestamp": "2026-04-03T16:30:00.000Z",
  "context": "LoggingInterceptor",
  "traceId": "abc123def456",
  "userId": 42,
  "method": "POST",
  "url": "/api/stock/list"
}
```

**错误日志**：

```json
{
  "level": "error",
  "message": "Cannot read properties of undefined (reading 'id')",
  "timestamp": "2026-04-03T16:30:01.000Z",
  "context": "GlobalExceptionsFilter",
  "stack": "TypeError: Cannot read properties...\n    at ...",
  "traceId": "abc123def456",
  "userId": 42,
  "method": "POST",
  "url": "/api/backtest/run"

```

### 3.3 开发环境行为

开发环境仍使用 NestJS ConsoleLogger，**不改变现有 DX**。`traceId` 等字段仅在生产 Winston JSON 中注入。

如需在开发环境也显示 `traceId`，可通过环境变量 `LOG_TRACE_IN_DEV=true` 启用（可选，不在本次任务范围内）。

## 四、任务 3：生产环境请求/响应日志

### 4.1 问题描述

当前 `LoggingInterceptor` 仅在开发环境加载（`main.ts:62-64`），且日志信息极为有限（仅 `method url - ms`）。生产环境下完全没有请求级别的可见性，无法排查：

- 慢请求定位
- 异常请求模式识别
- API 使用频率分析
- 用户行为审计

### 4.2 方案设计

#### 4.2.1 改造 LoggingInterceptor

将现有的 `LoggingInterceptor` 从"仅开发环境使用"升级为"全环境可用"，并丰富日志内容：

**文件**：`src/lifecycle/interceptors/logging.interceptor.ts`

```typescript
import { CallHandler, ExecutionContext, Injectable, NestInterceptor } from '@nestjs/common'
import { Observable, tap } from 'rxjs'
import { LoggerService } from 'src/shared/logger/logger.service'
import { RequestContextService } from 'src/shared/context/request-context.service'

@Injectable()
export class LoggingInterceptor implements NestInterceptor {
  constructor(private readonly loggerService: LoggerService) {}

  intercept(context: ExecutionContext, next: CallHandler): Observable<unknown> {
    const request = context.switchToHttp().getRequest()
    const response = context.switchToHttp().getResponse()
    const { method, url, ip } = request
    const userAgent = request.get('user-agent') || ''
    const start = Date.now()

    return next.handle().pipe(
      tap({
        next: () => {
          const latency = Date.now() - start
          const statusCode = response.statusCode
          const ctx = RequestContextService.getCurrentContext()

          this.loggerService.log(
            {
              message: `${method} ${url} ${statusCode} ${latency}ms`,
              latency,
              statusCode,
              ip,
              userAgent,
              traceId: ctx?.traceId,
              userId: ctx?.userId,
            },
            'HTTP',
          )
        },
        error: (error) => {
          const latency = Date.now() - start
          const ctx = RequestContextService.getCurrentContext()

          this.loggerService.warn(
            {
              message: `${method} ${url} ERROR ${latency}ms`,
              latency,
              ip,
              userAgent,
              traceId: ctx?.traceId,
              userId: ctx?.userId,
              error: error?.message,
            },
            'HTTP',
          )
        },
      }),
    )
  }
}
```

#### 4.2.2 生产环境开关

通过环境变量 `LOG_HTTP_REQUESTS` 控制是否在生产环境启用请求日志：

| 环境变量 | 默认值 | 说明 |
|---------|--------|------|
| `LOG_HTTP_REQUESTS` | `true` | 是否记录 HTTP 请求日志（开发/生产均适用） |
| `LOG_HTTP_BODY` | `false` | 是否记录请求 Body（注意数据安全，默认关闭） |

#### 4.2.3 修改 main.ts 注册逻辑

```typescript

```

#### 4.2.4 LoggerService 增强：支持对象形式 message

当前 `log()` 方法的 `message` 参数为 `unknown`，但 Winston 调用仅传 `string`。需要增强以支持对象形式，使结构化字段能正确展开到 JSON：

```typescript
// src/shared/logger/logger.service.ts — 修改 log() 方法

log(message: unknown, context?: string) {
  // Console 输出仍使用 string 格式（保持开发体验）
  const displayMessage = this.formatUnknownMessage(message)
  super.log(displayMessage, context)

  // Winston 写入支持对象展开
  if (this.winstonLogger) {
    if (typeof message === 'object' && message !== null && !(message instanceof Error)) {
      this.winstonLogger.info({ ...message, context })
    } else {
      this.winstonLogger.info(displayMessage, { context })
    }
  }
}
```

对 `warn()` 和 `error()` 方法做类似增强。

#### 4.2.5 敏感信息过滤

请求 Body 日志（`LOG_HTTP_BODY=true` 时）必须过滤敏感字段：

```typescript
const SENSITIVE_FIELDS = ['password', 'newPassword', 'oldPassword', 'token', 'secret', 'captchaCode']

function sanitizeBody(body: Record<string, unknown>): Record<string, unknown> {
  if (!body || typeof body !== 'object') return body
  const sanitized = { ...body }
  for (const field of SENSITIVE_FIELDS) {
    if (field in sanitized) {
      sanitized[field] = '***'
    }
  }
  return sanitized
}
```

**文件**：`src/lifecycle/interceptors/logging.interceptor.ts` 内定义或放入 `src/common/utils/sanitize.util.ts`。

## 五、跨任务基础设施：请求上下文（traceId）

### 5.1 设计目标

为每个 HTTP 请求分配唯一 `traceId`，并在整个请求生命周期（中间件 → Guard → Interceptor → Service → Filter）中透传，使所有日志可按 `traceId` 关联。

### 5.2 技术选型

使用 Node.js 16+ 内置的 `AsyncLocalStorage`（`node:async_hooks`），**零外部依赖**。

选择理由：
- Node.js 原生 API，无维护风险
- NestJS 11 完全兼容
- 性能开销可忽略（V8 引擎已优化）
- 无需 `cls-hooked` / `@nestjs/cls` 等第三方库

### 5.3 RequestContextService

**文件**：`src/shared/context/request-context.service.ts`

```typescript
import { AsyncLocalStorage } from 'node:async_hooks'

export interface RequestContext {
  /** 请求唯一标识（16 位 hex） */
  traceId: string
  /** 当前用户 ID（JWT 解析后填入） */
  userId?: number
  /** HTTP 方法 */
  method?: string
  /** 请求路径 */
  url?: string
  /** 请求开始时间戳 */
  startTime?: number
}

export class RequestContextService {
  private static readonly storage = new AsyncLocalStorage<RequestContext>()

  /**
   * 在 AsyncLocalStorage 上下文中执行回调。
   * 由中间件在请求入口处调用。
   */
  static run(context: RequestContext, callback: () => void): void {
    this.storage.run(context, callback)
  }

  /** 获取当前请求上下文（可能为 undefined，如非 HTTP 请求场景） */
  static getCurrentContext(): RequestContext | undefined {
    return this.storage.getStore()
  }

  /** 获取当前 traceId 的便捷方法 */
  static getTraceId(): string | undefined {
    return this.storage.getStore()?.traceId
  }

  /** 更新上下文中的 userId（Guard 解析 JWT 后调用） */
  static setUserId(userId: number): void {
    const ctx = this.storage.getStore()
    if (ctx) {
      ctx.userId = userId
    }
  }
}
```

### 5.4 RequestContextMiddleware

**文件**：`src/shared/context/request-context.middleware.ts`

```typescript
import { Injectable, NestMiddleware } from '@nestjs/common'
import { Request, Response, NextFunction } from 'express'
import { randomBytes } from 'node:crypto'
import { RequestContextService } from './request-context.service'

@Injectable()
export class RequestContextMiddleware implements NestMiddleware {
  use(req: Request, _res: Response, next: NextFunction): void {
    // 优先从上游代理/网关获取 trace ID，否则自动生成
    const traceId =
      (req.headers['x-trace-id'] as string) ||
      (req.headers['x-request-id'] as string) ||
      randomBytes(8).toString('hex')

    const context = {
      traceId,
      method: req.method,
      url: req.originalUrl || req.url,
      startTime: Date.now(),
    }

    // 将 traceId 写入响应头，便于客户端关联
    _res.setHeader('X-Trace-Id', traceId)

    RequestContextService.run(context, () => next())
  }
}
```

## 设计原则

- **零外部依赖（tracing）**：`AsyncLocalStorage` 为 Node.js 内置，不引入 `cls-hooked` / `@nestjs/cls`
- **最小侵入**：现有 API 行为不变，仅增强日志输出和新增健康检查端点
- **开发体验不变**：开发环境仍使用 NestJS Console 格式输出
- **默认安全**：请求 Body 日志默认关闭，开启后自动脱敏
- **可配置**：所有新增行为通过环境变量控制，无需改代码即可调整
- **为 P1 铺路**：`traceId` 和结构化日志为后续 Prometheus 指标、Grafana 仪表盘、OpenTelemetry 链路追踪奠定基础
