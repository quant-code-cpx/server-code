# 智能体队列工作进程运行手册

## 1. 适用范围

本文说明 Agent BullMQ 队列的启动、恢复、监控、取消和回滚。实现对应 Batch 012，不包含会话 REST API、POST SSE 或定时 Agent 任务。

权威边界：

```text
PostgreSQL AiAgentRun + AiJobOutbox
  → AgentQueueService 幂等投递
  → Redis/BullMQ agent-execution
  → AgentProcessor
  → AgentOrchestratorService.resume()
  → PostgreSQL checkpoint / Step / Event / Run 终态
```

Redis 只保存运输状态。Run 正文、用户输入、工作流版本、预算、checkpoint 和终态都以 PostgreSQL 为准。

## 2. 进程角色

`PROCESS_ROLE` 支持：

| 值             | 用途                                                  |
| -------------- | ----------------------------------------------------- |
| `api`          | HTTP API 与 Agent producer；不注册 Agent processor    |
| `agent-worker` | 无 HTTP 监听的 Agent processor、reconciler 和队列指标 |
| `scheduler`    | 为后续独立 Scheduler 入口预留                         |
| `all`          | 本地开发合并运行 API 与 Agent Worker                  |

开发环境默认 `all`，生产环境默认 `api`。生产应显式拆分 API 与 Worker，避免 API 多副本意外重复加载消费者。

Docker Compose 默认让 `app` 使用 `PROCESS_ROLE=api`。独立 Worker 通过 profile 启动：

```bash
docker compose --profile agent-worker up -d --build agent-worker
docker compose --profile agent-worker ps
docker compose logs --tail=120 agent-worker
```

Worker 不映射公网端口。Compose 会在启动前执行 Prisma generate 与 migrate deploy，并将 `TUSHARE_SYNC_ENABLED=false`，避免 Worker 进程注册或补跑行情同步任务。

## 3. 配置

| 环境变量                      | 默认值        | 说明                                      |
| ----------------------------- | ------------- | ----------------------------------------- |
| `AGENT_QUEUE_REDIS_URL`       | 空            | 独立 Redis URL；为空时复用通用 Redis 配置 |
| `AGENT_QUEUE_PREFIX`          | `quant:agent` | BullMQ key namespace                      |
| `AGENT_WORKER_CONCURRENCY`    | `2`           | 单 Worker 并发 job 数                     |
| `AGENT_JOB_TIMEOUT_MS`        | `180000`      | Worker 对单次 job 的外层超时              |
| `AGENT_JOB_ATTEMPTS`          | `5`           | BullMQ 基础设施重试次数                   |
| `AGENT_JOB_BACKOFF_MS`        | `2000`        | 指数退避基数                              |
| `AGENT_RECONCILE_INTERVAL_MS` | `10000`       | PostgreSQL 恢复扫描周期                   |
| `AGENT_RECONCILE_BATCH_SIZE`  | `100`         | 单轮 outbox/Run 扫描上限                  |
| `AGENT_RUN_LEASE_MS`          | `30000`       | Run Worker lease 时长                     |
| `AGENT_LEASE_HEARTBEAT_MS`    | `10000`       | lease heartbeat 周期，必须小于 lease 时长 |

生产建议 Agent queue 使用独立 Redis 凭据、ACL、logical DB 或实例。Redis 必须启用 AOF/RDB，并使用 `maxmemory-policy noeviction`；不能让缓存淘汰删除 BullMQ key。

## 4. 入队与幂等

- Job data 严格为 `{ schemaVersion: 1, runId }`。
- `jobId=runId`，重复 producer 调用不会创建并行 job。
- Run 创建事务同时写 `AiJobOutbox`，避免“数据库提交成功但进程在 enqueue 前崩溃”丢任务。
- enqueue 成功后 outbox 进入 `PUBLISHED`；Redis 错误进入 `RETRY` 并设置下一次退避时间。
- 若 Redis 中同 job 仍为 waiting/active/delayed，producer 返回 existing。
- 若 Redis 保留 failed/completed job，但 PostgreSQL Run 仍可恢复，producer 先移除旧 job 再重新投递。

Job payload 禁止放 prompt、正文、token、个人数据、Tool 结果或 checkpoint。Worker 必须按 `runId` 回 PostgreSQL 读取固定版本和上下文。

## 5. Worker 执行和重试

每次 BullMQ delivery 都生成新的 worker identity。过期 lease 不允许使用旧 identity 重新接管。

Processor 只负责：

1. 校验 job name、payload、schemaVersion 和 `jobId`。
2. 创建 timeout/shutdown `AbortSignal`。
3. 调用 `AgentOrchestratorService.resume(runId, { workerId, signal })`。
4. 将业务终态正常返回，将基础设施错误抛给 BullMQ retry。

工作流内部的 Model/Tool retry 仍由 Workflow Engine 管理。BullMQ 不重复实现业务 step retry，避免重复计费。

## 6. 取消与关闭

- Waiting Run：应用先把 PostgreSQL Run 置为 `CANCELLED`，再调用 `removeWaitingRun()` 尝试移除 waiting/delayed job。
- Active Run：应用写 `CANCEL_REQUESTED`。Workflow heartbeat 发现后中止节点 `AbortSignal`，Orchestrator 协作写入 `CANCELLED`。
- Worker `SIGTERM/SIGINT`：Nest shutdown hook 中止 active controllers；未终态 Run 保持 `RUNNING`，不做非法 `RUNNING → QUEUED` 回退。
- 新 Worker 只在旧 lease 过期后使用新 identity 接管，并从 checkpoint 恢复。

迟到的 Model/Tool 响应仍受 Run 状态、lease owner、statusVersion 和最终事务门禁，不能把取消后的 Run 写成 `COMPLETED`。

## 7. PostgreSQL 恢复扫描

Reconciler 每轮执行：

1. 发布到期的 `PENDING/RETRY` outbox。
2. 扫描 deadline 未到、attempt 未耗尽的 `QUEUED` Run。
3. 扫描 lease 已过期的 `RUNNING/CANCEL_REQUESTED` Run。
4. 用同一 `jobId=runId` 幂等补投。

Redis 被清空或重启时不需要手工改 Run 状态。确认 PostgreSQL 与 Redis 可用后启动 Worker，Reconciler 会从 authority 表重建任务。

## 8. 故障处理

### Redis 不可用

1. 保持 PostgreSQL Run/outbox 不变。
2. 修复 Redis ACL、网络、持久化或容量问题。
3. 确认 `maxmemory-policy`：

```bash
docker exec quant_redis sh -lc 'redis-cli --no-auth-warning -a "$REDIS_PASSWORD" CONFIG GET maxmemory-policy'
```

4. 重启 Worker。`RETRY` outbox 与 recoverable Run 会自动补投。

### Worker 异常退出或 stalled

不要手工把 `RUNNING` 改回 `QUEUED`。等待数据库 lease 过期，再由新 delivery/new worker identity 接管。重复 delivery 由数据库 lease 阻止双执行。

### Run 长期不恢复

检查：

- Run 是否已终态、deadline 到期或 `attempt >= max_attempts`。
- `lease_expires_at` 是否仍有效。
- 对应 outbox 是否为 `RETRY/DEAD`，`last_error` 是否表明 Redis 或 payload hash 问题。
- BullMQ 是否保留同 runId 的 active/waiting/failed job。
- Worker 日志中是否存在 claim、heartbeat、stalled 或 processor failed 记录。

## 9. 指标与日志

Prometheus 指标包括：

- `bullmq_queue_depth{queue="agent-execution"}`
- `bullmq_active_jobs{queue="agent-execution"}`
- `bullmq_failed_jobs{queue="agent-execution"}`
- `bullmq_delayed_jobs{queue="agent-execution"}`
- `bullmq_stalled_jobs_total{queue="agent-execution"}`
- `bullmq_enqueue_lag_seconds{queue="agent-execution"}`
- `agent_run_recovery_total{result="enqueued|existing|failed"}`

关键结构化日志 operation：`agentQueue.enqueue`、`agentProcessor.process`、`agentProcessor.failed`、`agentProcessor.stalled`、`agentReconciler.requeue`。

## 11. 回滚

1. 停止 `agent-worker` 服务或将 producer 暂停。
2. 保留 `AiAgentRun`、`AiJobOutbox`、checkpoint、Step 和 Event，不删除 Redis/数据库证据。
3. 修复后重新启动 Worker，由 Reconciler 补投。
4. 不回滚已执行 migration，不删除 outbox 表；旧应用版本不会读取该表，可保持兼容。
