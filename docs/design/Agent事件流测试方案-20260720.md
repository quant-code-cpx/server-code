## 2. 业务范围

| 项目     | 内容                                                                    |
| -------- | ----------------------------------------------------------------------- |
| 接口     | `POST /api/agent/runs/events`                                           |
| 调用方   | 已登录 Web 前端，通过 `fetch` 读取 `ReadableStream`                     |
| 权威数据 | PostgreSQL `AiRunEvent`；Redis/进程通知仅可加速                         |
| 鉴权     | Bearer JWT；Run 必须属于当前用户                                        |
| 输入     | Body `{ runId, afterSequence }`；可选 Header `Last-Event-ID`            |
| 输出     | raw `text/event-stream`，业务帧为 `id/event/retry/data`，心跳为 comment |
| 不在范围 | GET `EventSource`、WebSocket token delta、新事件类型、前端 parser       |

## 3. 核心业务规则

- 端点必须是带非空路径的 `POST`，不能使用 Nest `@Sse()`。
- owner 校验成功前不能写 SSE headers；无 JWT 返回 401，跨租户与不存在统一返回 6002/404。
- SSE `id` 使用持久 `eventId`；`Last-Event-ID` 只能在 Body 指定的同一 `runId` 内解析，Header 与 Body 同时存在时 Header 对应 sequence 优先。
- `afterSequence` 语义为只发送 `sequence > cursor`；每个连接只发送单调递增且不重复的持久事件。
- 连接先确定 high-water mark 并完成 replay，再 tail 新事件；replay/live 交界不能漏事件。
- PostgreSQL 是唯一正确性来源；实时通知丢失、多实例发布或 Redis 不可用时，轮询重放仍能读到已提交事件。
- 仅允许公共契约的 14 个事件名和 `schemaVersion=1.0`；未知类型/版本、CR/LF 注入或敏感内部字段不得原样输出。
- heartbeat 为 `: heartbeat` comment，不占 sequence、不落库；终态事件发送完成后关闭连接。
- 客户端断开只取消订阅和定时器，不取消 Run；半开连接、idle timeout 和 JWT 到期均安全释放资源。
- 慢消费者超过连接缓冲上限时主动断开；事件保留在 PostgreSQL，客户端可从最后连续游标恢复。
- `TransformInterceptor`、异常过滤器和 HTTP 观测不能包装或提前结束 raw stream；响应必须禁缓存、禁代理缓冲。
- 每用户连接数受限；连接关闭、初始化失败和应用 shutdown 均必须归还计数。

## 4. 风险分析

| 风险                            | 业务影响                          | 测试策略                        | 优先级 |
| ------------------------------- | --------------------------------- | ------------------------------- | ------ |
| replay 与 live 窗口丢事件       | 回答缺字、Tool 状态永久卡住       | high-water 交界并发追加事件     | P0     |
| 越权或跨 Run 游标               | 泄露其他用户研究数据              | owner 与 Last-Event-ID 绑定测试 | P0     |
| raw stream 被 JSON wrapper 改写 | 前端 parser 完全不可用            | 真实 HTTP 字节断言              | P0     |
| 慢连接无限缓存                  | 单用户拖垮 API 内存               | 可控 Writable 背压与上限测试    | P0     |
| 通知丢失即停流                  | 多实例/Redis 故障时事件不可见     | 禁用通知后仅靠 PostgreSQL tail  | P0     |
| 重复、倒退或非法事件            | 客户端重复渲染或出现 sequence gap | serializer 与 stream 单调性测试 | P1     |
| 断开后资源泄漏                  | 连接数耗尽、timer/DB 查询累积     | Abort、close、shutdown 回归     | P1     |
| 心跳/idle/JWT 生命周期错误      | 假在线、失效 token 长期占连接     | fake timer + 真实 HTTP 关闭测试 | P1     |

## 6. 性能与压力设计

| 项目     | 设计                                                              |
| -------- | ----------------------------------------------------------------- |
| 环境     | 本机 Docker PostgreSQL/Redis、独立 Nest 测试进程                  |
| 数据规模 | 单 Run 1000 事件；单事件含中文 delta 和 Tool summary              |
| 并发模型 | 1/5/10/20 条连接阶梯；慢消费者单独隔离                            |
| 指标     | 首字节、replay 总耗时、p50/p95/p99、bytes、DB 查询数、RSS、错误率 |
| 停止条件 | 进程失去健康、DB pool 耗尽、内存持续增长或错误率不可控            |
| 数据清理 | 使用独立测试用户/Run，执行后删除本轮测试数据                      |

## 7. 执行顺序

1. serializer、cursor、连接生命周期与背压单元测试。
2. Nest 真实 HTTP stream：DTO、Guard、Interceptor、Filter、headers、Abort。
3. 真实 PostgreSQL replay/high-water/tail、跨租户与通知丢失测试。
4. 真实 Redis/多连接、性能和慢消费者测试。
5. Agent 相关回归、legacy ESLint、build、容器重建、日志与 health/ready。

## 8. 完成条件

- P0/P1 用例通过；P2 记录真实环境和观测值。
- 无丢事件、跨租户、半帧、无限缓冲或连接资源泄漏。
- 文档中的 SSE `id` 语义统一为持久 `eventId`。
- 发现缺陷均有回归用例；执行报告记录命令、结果和剩余风险。
