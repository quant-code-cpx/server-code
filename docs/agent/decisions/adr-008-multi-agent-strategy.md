# ADR-008：多 Agent 策略

## 背景

需求覆盖数据、量化、搜索、报告、定时任务，但金融准确性要求可控。多 Agent 会增加模型调用、上下文同步和故障点。

## 问题

采用单 Agent、多专业 Agent，还是高自主 Agent 群？

## 可选方案

1. 单受控 Agent + 多 Tool。
2. 主 Agent + 数据/量化/新闻子 Agent。
3. 确定性工作流 + 可选受控专业节点。
4. 高自主多 Agent。

## 最终决策

MVP 采用 1 与 3：一个研究 Agent 在版本化工作流内选择标准 Tool；确定性节点负责查询、计算、验证和引用。暂不启用模型间自由委派。后期只有可独立评测、并行且上下文需隔离的研究域才封装专业子 Agent。

## 选择理由

内部数据和计算已由 Service/Tool 专业化，无需再用模型扮演“数据 Agent”；单 Agent 更低成本、更容易复现和取消。

## 放弃其他方案原因

主从 Agent 会重复总结和 Tool 调用；高自主方案难以保证来源、时点和成本。

## 正面影响

链路短、审计完整、故障定位简单。

## 负面影响

超复杂研究的并行推理能力有限。

## 风险

单次上下文过大。通过领域 Tool 输出摘要、并行确定性节点和分阶段综合缓解。

## 后续复审条件

存在至少 3 个可独立评测专业流程；并行节省延迟超过 30%；上下文隔离显著提高准确率；预算允许额外模型调用。
