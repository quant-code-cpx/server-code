## 一、策略管理

### STR-BE-01 — strategies/summary 字段名不匹配

**现状**

```
POST /api/strategies/summary {}
→ 200 { total: 1, byType: [{strategyType, count}], strategies: [...] }
```

**前端期望（TypeScript 类型）**

```typescript
type StrategySummary = {
  totalCount: number
  activeSignalCount: number
  recent7dRunCount: number
  recent7dBestReturn: number | null
  recent7dWorstReturn: number | null
}
```

**需求场景**

策略管理页顶部 4 张 KPI 卡片："策略总数 / 已激活信号 / 近 7 日回测 / 近 7 日最佳收益"。当前 4 张卡片均显示 `undefined`，页面顶部统计信息完全不可用。

**前端调用**

```typescript
apiClient.post('/api/strategies/summary', {})
```

**建议**

后端可选以下任一方式：

1. 将响应改为 `{ totalCount, activeSignalCount, recent7dRunCount, recent7dBestReturn, recent7dWorstReturn }` — 语义明确，前端零改动
2. 或告知前端实际字段名（如 `total` 对应 `totalCount`），前端做适配映射

注：`activeSignalCount`（已激活信号数）与 `recent7dRunCount`（近 7 日回测次数）如后端暂无统计，可返回 `0` 占位，前端会如实展示。

## 八、第三轮新发现 Bug

### RW-NEW-01 — DIVIDEND_EX 事件类型 500 BigInt 序列化失败

**需求场景**

前端"事件驱动研究"页面（路由 `/research/event-study`）允许用户选择事件类型后查看相关股票事件列表及事件日历热图。DIVIDEND_EX（除权除息）是分析股价填权效应的核心事件类型，用户需要查询某时间段内发生除权除息的股票，并进一步分析事件前后的超额收益（CAR）。该类型数据量大（通常每年数千条），是事件研究模块最常用的类型之一。

**复现步骤**

1. 打开"事件驱动研究"页面
2. 在事件类型下拉框中选择"除权除息（DIVIDEND_EX）"
3. 点击查询，前端发出请求：

```http
POST /api/event-study/events
Content-Type: application/json

{
  "eventType": "DIVIDEND_EX",
  "startDate": "20260101",
  "endDate": "20260523",
  "page": 1,
  "pageSize": 20
}
```

4. 服务端返回：

```json
{
  "code": 500,
  "message": "Do not know how to serialize a BigInt"
}
```

**对比：其余 7 种事件类型均正常**

| eventType       | 结果                         |
| --------------- | ---------------------------- |
| FORECAST        | ✅ 200，total: 3304          |
| HOLDER_INCREASE | ✅ 200，total: 283           |
| HOLDER_DECREASE | ✅ 200，total: 1394          |
| SHARE_FLOAT     | ✅ 200，total: 163012        |
| REPURCHASE      | ✅ 200，total: 1376          |
| AUDIT_QUALIFIED | ✅ 200，total: 14            |
| DISCLOSURE      | ✅ 200，total: 11025         |
| **DIVIDEND_EX** | ❌ **500 BigInt 序列化失败** |

**根因分析**

DIVIDEND_EX 对应的数据库表（或视图）中存在 BigInt 类型字段（例如除息金额、分红总额等，可能来自 Tushare 原始数据以整数分/厘存储），Prisma 将其映射为 JS `BigInt` 类型，而 Node.js 原生 `JSON.stringify()` 无法序列化 `BigInt`，导致 NestJS 在序列化响应体时崩溃。

**修复建议（三选一，推荐方案 3 全局最省力）**

1. **Prisma Schema 层修改**（推荐，一劳永逸）：找到 DIVIDEND_EX 相关 model，将 BigInt 字段类型改为 `Decimal` 或 `Float`，然后 `prisma generate`

   ```prisma
   // 修改前
   divAmount  BigInt?
   // 修改后
   divAmount  Float?
   ```

2. **Service 层转换**：在 service 返回结果前，将 BigInt 字段显式转为 Number

   ```typescript
   const events = await this.prisma.stockDivEvent.findMany({ ... });
   return events.map(e => ({ ...e, divAmount: e.divAmount ? Number(e.divAmount) : null }));
   ```

3. **全局 NestJS 拦截（最省力）**：在 `main.ts` 中加一行，影响整个应用所有序列化
   ```typescript
   // main.ts bootstrap 函数内
   BigInt.prototype.toJSON = function () {
     return Number(this)
   }
   ```

**优先级**：🔴 高 — DIVIDEND_EX 类型完全不可用，用户选择该类型后页面报错，功能完全缺失

### ANO-NEW-01 — anomalies/detail 500 Prisma id: undefined

**需求场景**

前端"异动监控"页面（路由 `/alert/anomalies`）展示实时涨跌异常、成交量异常等市场异动信号列表。用户点击某条异动记录时，右侧会弹出详情抽屉（Drawer），展示该异动的详细信息（历史对比、同类异动频次、`anomalyType` 枚举说明等）。详情抽屉内容完全依赖 `POST /api/alert/anomalies/detail` 接口。

**复现步骤**

1. 打开"异动监控"页面
2. 页面加载异动列表（`POST /api/alert/anomalies/list`），假设获取到一条记录 id 为 `888`
3. 用户点击该条记录，前端向后端请求详情：

```http
POST /api/alert/anomalies/detail
Content-Type: application/json

{ "anomalyId": 888 }
```

4. 服务端返回：

```json
{
  "code": 500,
  "message": "\nInvalid `this.prisma.marketAnomaly.findUnique()` invocation in\n/app/src/apps/alert/market-anomaly.service.ts:114:49\n\nArgument `where` needs at least one of `id` or `tradeDate_tsCode_anomalyType`\n{\n  where: {\n    id: undefined,\n  }\n}"
}
```

**完整 Prisma 错误堆栈**

```
Invalid `this.prisma.marketAnomaly.findUnique()` invocation in
/app/src/apps/alert/market-anomaly.service.ts:114:49

  112 async getAnomalyDetail(dto: ...) {
  113   return this.prisma.marketAnomaly.findUnique({
→ 114     where: {
            id: undefined    ← 问题在这里，id 永远是 undefined
          }
        });
  }

Argument `where` needs at least one of `id` or `tradeDate_tsCode_anomalyType`
```

**根因分析**

DTO 接收字段为 `anomalyId`，但 service 第 114 行读取的是 `dto.id`（字段名不匹配）：

```typescript
// DTO（推测）
class GetAnomalyDetailDto {
  anomalyId: number // 前端传的字段名
}

// Service 第 114 行（推测）
where: {
  id: dto.id
} // dto.id 不存在，永远 undefined
// 应该是 dto.anomalyId
```

Prisma `findUnique` 收到 `where: { id: undefined }` 时，认为没有提供任何有效的唯一约束，因此报错。

**修复建议**

`market-anomaly.service.ts:114` 附近，修改读取字段名：

```typescript
// 修复前
where: {
  id: dto.id
}

// 修复后
where: {
  id: dto.anomalyId
}
```

或者统一 DTO 字段名为 `id`（需同时更新前端调用，工作量更大，不推荐）。

**影响范围**

- 异动监控页面的详情抽屉 100% 报错，功能完全不可用
- 由于详情端点始终 500，ANO-BE-01（`history.anomalyType` 字段）和 ANO-BE-05（详情数据结构）均无法验证，这两个需求的实现情况未知

**验证方法（修复后）**

```bash
# 先从列表获取一个真实 id
POST /api/alert/anomalies/list { "tradeDate": "20260522", "page": 1, "pageSize": 1 }
# 取到 id 后请求详情
POST /api/alert/anomalies/detail { "anomalyId": <id from above> }
# 期望：200，返回包含 history 数组，每项含 anomalyType 字段
```

**优先级**：🔴 高 — 异动详情抽屉完全不可用，用户无法查看任何异动详情，属核心功能缺失
