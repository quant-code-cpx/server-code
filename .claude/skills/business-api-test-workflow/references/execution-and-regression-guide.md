# 执行与回归指南

## 执行顺序

每个模块按从小到大的范围验证：

1. 目标 spec：`pnpm exec jest <spec路径> --runInBand`
2. 相关 spec：同模块 `test/` 目录下相关测试。
3. 影响链路 E2E：`pnpm test:e2e` 或指定 e2e spec。
4. 构建：`pnpm build`。
5. 涉及 Docker/Redis/PostgreSQL/Prisma/Tushare 同步时，再做容器日志、迁移或真实返回验证。

不要一开始就只跑全量测试掩盖定位；也不要只跑最小测试就宣布完成。

## 测试实现建议

### 单元测试

- 用 Nest TestingModule 或直接实例化服务。
- 外部依赖使用可状态化 mock，避免 mock 只返回固定值。
- 金融计算使用小而可手算的数据集。
- 并发测试用 `Promise.all`、受控 fake timers、可观测的 mock 状态。

### Controller / 集成测试

- 使用 `test/helpers/create-test-app.ts` 或项目现有测试 App 工具。
- 启用真实 `ValidationPipe`、TransformInterceptor、ExceptionFilter、Guard mock。
- 覆盖 DTO 校验、异常透传、权限、响应包装。
- 本项目 Controller 必须使用 POST；查询条件走 Body。

### E2E 测试

- 走完整 Nest 应用与 supertest。
- 外部 Tushare API 使用 fixture/mock。
- 使用隔离测试数据库/Schema 与 Redis DB，测试后清理。
- 覆盖跨模块关键流程：认证、策略→回测、组合→风控、因子→策略→回测、预警→通知。

## 性能、负载与压力测试执行

### 基准性能 `PERF`

目的：知道正常规模下有多快。

记录：

- OS/硬件/Node 版本。
- 数据量、用户数、时间范围、缓存状态。
- 平均、p50、p95、p99、最大值、错误率。
- DB 慢查询、Redis 命中率、应用日志错误。

### 负载测试 `LOAD`

目的：验证目标并发下稳定性。

建议模型：

- 预热 30-60 秒。
- 固定并发 1/5/10/20/50 阶梯。
- 每阶梯记录吞吐、错误率、p95/p99。
- 读接口优先使用稳定本地数据；写接口必须使用独立测试数据并可清理。

### 压力测试 `STRESS`

目的：找系统拐点和降级行为。

规则：

- 只在可控环境执行，不对第三方 API 施压。
- 明确停止条件：错误率过高、资源耗尽、DB 连接池打满、队列积压不可恢复。
- 压测后检查数据一致性和日志。

### 稳定性测试 `SOAK`

目的：发现泄漏和长时间不稳定。

记录：

- 持续时长。
- 内存趋势、连接数、队列长度、错误日志。
- 周期性结果是否漂移。

## 缺陷分级

| 严重度 | 含义 | 示例 |
| ------ | ---- | ---- |
| S0 | 阻塞/数据破坏/安全事故 | 资金计算错误、越权读写、核心页面 500 |
| S1 | 核心流程不可用 | 回测不能启动、详情接口 100% 失败 |
| S2 | 重要功能错误但可绕过 | 筛选条件失效、统计字段缺失 |
| S3 | 低频边界或体验问题 | 提示文案不准确、低概率边界 |

优先级按业务影响和修复紧急程度确定，不一定等于严重度。

## Bug 记录字段

| 字段 | 说明 |
| ---- | ---- |
| ID | 模块缩写 + 日期 + 序号，如 `PF-20260523-001` |
| 标题 | 一句话说明业务问题 |
| 严重度/优先级 | S0-S3 / P0-P3 |
| 业务影响 | 用户会看到什么错误或承担什么风险 |
| 复现步骤 | 可重复执行的步骤/API 请求 |
| 期望结果 | 从业务规则推导的正确行为 |
| 实际结果 | 真实响应、日志、截图或测试失败信息 |
| 证据 | 测试名、命令、日志摘录、数据样本 |
| 根因假设 | 测试阶段只写假设；修复阶段再确认 |
| 状态 | NEW / CONFIRMED / FIXING / FIXED / REGRESSED / CLARIFY |

## 修复模式

进入修复模式后：

1. 阅读相关实现，确认根因。
2. 设计最小修复，不重构无关代码。
3. 若契约需要调整，先更新测试用例文档并说明前后端影响。
4. 修改代码和测试。
5. 跑目标失败用例，确保红变绿。
6. 跑相关回归和 `pnpm build`。
7. 更新缺陷状态和执行报告。

## 回归策略

每个 Bug 至少回归：

- 原失败用例。
- 同类边界用例。
- 受影响接口的正常流程。
- 与修复相关的 E2E 链路。
- 构建或类型检查。

若修复涉及数据模型、迁移、容器或异步任务，再补充：

- Prisma generate/migrate 状态。
- 容器重建与日志。
- 队列消费/重试行为。
- 数据清理与幂等验证。

## 结束条件

可以收尾必须同时满足：

- 测试用例文档已更新。
- 本轮执行报告已记录命令、结果、Bug 状态、待澄清。
- 已修 Bug 的回归用例通过。
- 未修项都有明确原因：待澄清、环境阻塞、外部依赖、低优先级延后。
- 总结清楚说明“改了什么、怎么验证、运行注意事项”。
