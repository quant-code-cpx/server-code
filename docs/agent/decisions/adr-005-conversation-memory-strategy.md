# ADR-005：会话与记忆策略

## 背景

系统需保存原始消息、Run、Tool、引用、用户偏好和研究结论，同时不能无限把历史发给模型。

## 问题

如何区分记录、短期工作记忆、长期记忆与工作流状态？

## 可选方案

1. 全历史拼接。
2. 仅滚动文本摘要。
3. PostgreSQL 原始记录 + 结构化状态 + 版本摘要 + 按需检索。
4. 首期全部向量化。

## 最终决策

采用方案 3。原始消息不可变；Run/Step 保存执行状态；会话摘要按版本滚动；当前股票、比较对象、日期范围等存结构化 JSON；长期记忆必须有来源、置信度、过期时间和用户可删除标记。上下文构建按预算选择最近消息、摘要、相关 Tool 摘要和用户授权记忆。

## 选择理由

关系库已存在，结构化金融实体比纯向量更可靠；摘要可跨模型；原始记录支持审计和重新生成。

## 放弃其他方案原因

全历史成本无限增长；纯摘要丢证据；全向量化首期无语料规模和评测基础。

## 正面影响

成本可控、可恢复、可删除、跨模型兼容。

## 负面影响

需上下文预算器、摘要版本和过期治理。

## 风险

错误摘要被长期放大。摘要必须引用源消息 ID，用户纠错产生新版本，不覆盖原始记录。

## 后续复审条件

关键词/结构化检索无法满足研究报告召回，且有成套召回评测集后再引入向量检索。
